# analyze-todo

入力（記事URL・技術名・機能要望テキスト等）を解析し、プロジェクトとの適合性を評価して、対応すべき TODO を `docs/todo/` にマークダウンとして出力する。

出力は「やるべきこと」を人間に提案する文書であり、タスクリストではない。対応価値の低い項目は理由付きで除外する。出力した TODO は後から `/issue-plan` で Issue 化して対応するための中間成果物。Issue への分割はスキルの責務外で、Issue 化時に人間が判断する。

**重要: ユーザーとの対話および出力ファイルは日本語で書くこと。**

---

## ワークフロー

```
Phase 1: 入力解析     → 入力タイプの判定・内容取得
Phase 2: プロジェクト調査 → 関連するプロジェクトコンテキストの収集
Phase 3: 適合性評価   → 項目ごとの価値判定・低価値項目の除外
Phase 4: TODO 構造化  → マークダウンファイル生成・ユーザー確認
```

---

## Phase 1: 入力解析

入力タイプを判定し、内容を取得する。

| 入力タイプ | 判定基準 | 取得方法 |
|-----------|---------|---------|
| URL | `http://` or `https://` で始まる | WebFetch で取得。取得できない場合はユーザーに要点を聞く |
| 技術名・ライブラリ名 | 短いフレーズで具体的な技術を指す | WebSearch または deepwiki で概要を調査 |
| 機能要望テキスト | 上記以外の自由テキスト | そのまま使用 |
| 引数なし | 空 | 「分析対象を教えてください（URL、技術名、機能要望など）」と表示 |

複数の入力が混在する場合（URL + コメント等）は、全体を1つのコンテキストとして扱う。

### 入力からの抽出

内容を取得したら、以下を抽出する:
- **提案されていること**: 何を導入・実装すべきと言っているか
- **期待される効果**: なぜそれが有用か
- **前提条件・制約**: 必要な環境・依存関係

記事URLの場合、記事全体を要約するのではなく「このプロジェクトに取り入れられる具体的なアクション」を抽出する。

---

## Phase 2: プロジェクト調査

Phase 1 で抽出した内容に**関連する**プロジェクトのコンテキストを収集する。全ファイルを読むのではなく、入力の性質に応じて読む対象を絞る。

### 常に読むもの
- `docs/c4_overview.md` — システム全体像の把握

### 入力に応じて読むもの

| 入力の性質 | 読む対象 |
|-----------|---------|
| AI ハーネス・開発体験 | `.claude/` 配下、`Justfile`、`lefthook.yml` |
| 新技術・ライブラリ導入 | 関連する `docs/adr/`、既存の依存関係（`go.mod`、`package.json`等） |
| 既存機能の拡張 | 該当サービスのコード、関連 proto ファイル |
| インフラ・CI/CD | `docker-compose.yml`、`.github/workflows/` |
| テスト・品質 | 既存テスト、`.claude/rules/`、`docs/guides/` |

### 既存 TODO との重複チェック

`.output/todo/` 配下の既存ファイルを確認し、内容が重複する提案がないか確認する。重複がある場合は既存 TODO への追記・更新を検討する。

---

## Phase 3: 適合性評価

抽出した各項目について、プロジェクトコンテキストを踏まえて評価する。

### 評価観点

1. **プロジェクトとの関連性** — このプロジェクトの目的・技術スタックに合うか
2. **既存の仕組みとの重複** — 既に解決済み or ADR で決定済みの領域ではないか
3. **実装コスト vs 効果** — 学習用プロジェクトとしての投資対効果
4. **依存関係** — 他の TODO や未実装機能に依存するか

### 対応価値の判定

各項目を以下に分類する:

| 分類 | 基準 | 出力 |
|------|------|------|
| **対応する** | プロジェクトに適合し、実装する価値がある | 優先度別セクションに概要・効果・懸念・影響範囲を記述 |
| **対応しない** | 既に解決済み、スタック不一致、コスト過大等 | 末尾の「対応不要」テーブルに理由付きで記載 |

対応しないと判断した項目はユーザーに提示し、異論があれば TODO に昇格する。

---

## Phase 4: TODO 構造化

### 出力先

`.output/todo/{slug}.md` — slug は入力内容から英語のケバブケースで生成する。

### 出力テンプレート

```markdown
# {タイトル}

{参照元がある場合: 参照元: URL}

## 1. {TODO 項目名}

### 概要
{何をするか。影響するサービスやレイヤーを含めて 1〜3 文で伝える}

### 効果
{なぜやるべきか。実施によりプロジェクトが具体的に何を得るかを説明する}

### 懸念
{トレードオフ（受け入れるコスト）、リスク（不確実な問題）、実装コストの見積もりをまとめて自然文で書く。項目によって書く量は柔軟に調整してよい。懸念がなければセクションごと省略}

### 影響範囲
{具体的にどのファイル・ディレクトリ・レイヤーに変更が必要か。「ここにこういう機能を追加する」レベルの抽象度で書く。実装の詳細（具体的な関数名、コード例、パッケージのインポートパス等）には踏み込まない}

### 優先度
{高 / 中 / 低} — {その優先度と判断した理由を 1〜2 文で}

---

## 2. ...

---

## 対応不要

| 項目 | 理由 |
|------|------|
| {項目名} | {なぜ対応しないか} |
```

### 構造化のルール

- **各項目は提案として書く** — 人間が「やるかやらないか」を判断するための材料を提供する文書であり、実行手順書ではない
- **優先度は項目内に記載する** — 高/中/低 の3段階で、判断理由を添える。見出しで優先度別にグルーピングしない。項目の並び順は優先度の高い順にする
- **影響範囲は抽象的に** — 「product-mgmt の proto と DB スキーマに画像フィールドを追加し、管理画面にアップロード UI を実装する」のように、何をするかは伝えるが具体的な実装方法は書かない。設計はIssue化後に行う
- **依存関係がある場合は概要か影響範囲で触れる** — 既存 TODO や他の項目との依存があれば、自然な文脈で言及する

### ユーザー確認

出力内容をユーザーに提示する:
- TODO として出力する項目の一覧（優先度付き）
- 対応不要とした項目と理由

「問題なければOKと入力してください。修正があればお願いします。」と表示して入力を待つ。

フィードバックがあれば修正して再提示する。承認後にファイルを書き出す。

---

## Gotchas

- **記事の要約にならないようにする**: 記事 URL を渡された場合、記事の内容を要約するのではなく「このプロジェクトで何をすべきか」に変換する。記事に10の提案があっても、プロジェクトに関係するのが3つなら3つだけ書く
- **実装に踏み込まない**: 影響範囲は「何をするか」を伝えるが「どうやるか」は書かない。具体的な関数名、パッケージのインポートパス、コード例は書かない。設計判断は Issue 化後に `/issue-plan` で行う
- **既存 ADR との整合性**: 新技術導入の提案が既存 ADR と矛盾する場合、ADR の再評価が必要かもしれない旨を記載する（`/adr` スキルへの誘導）
- **WebFetch 失敗時の対応**: URL の取得に失敗した場合、ユーザーに記事の要点を聞く。推測で内容を補完しない
