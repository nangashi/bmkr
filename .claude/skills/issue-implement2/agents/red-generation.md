# Red 生成 — サブエージェント詳細手順

契約とテスト戦略からテストまたは検証項目を導出する。実装本体はまだ存在しないため、テストが実装に依存することは原理的にできない。

---

## 前提情報の取得

- `gh issue view {issue_number}` および `gh issue view {issue_number} --comments` で Issue 本文・コメント（計画含む）を直接読む
- `.output/issue-implement2/{issue_number}/contract.md` を読む
- `.output/issue-implement2/{issue_number}/test-strategy.md` を読む
- テストはコンパイルが通る状態にすること。実行すると失敗（Red）になるのが正しい状態

---

## 強制学習（テスト作成前の必須調査）

テスト導出に入る前に、以下の調査を必ず実施する。省略してテストを書き始めてはならない。

### A. 既存テストパターンの調査

変更対象サービスの既存テストファイル（`*_test.go`, `*.test.ts`）を検索し、以下を確認する:

1. **テストヘルパー**: `newContext`, `newRequest` 等があれば再利用する。同等の関数を新規に作らない
2. **モック実装**: 既存の `mock*` 構造体があれば命名規則・構造に従う
3. **フィクスチャ**: `sample*`, `fixture*` 等のデータ生成関数があれば再利用する
4. **コンパイル時チェック**: `var _ Interface = (*Mock)(nil)` パターンの有無を確認し、あれば同じパターンを使う

### B. テスト関連の注意事項

sqlc generate で Querier に新メソッドが追加されている場合、既存テストのモック実装にもメソッドスタブを追加する（`panic("not called")` で良い）。

---

## テスト導出の手順

### 1. 契約からテスト対象を列挙する

`contract.md` の Public Interface / Error Contract / Invariants を読み、テスト可能な振る舞いを列挙する。`// wip:` 動作コメントがあればそちらも参照する。

### 2. テスト戦略に従ってテスト手段を選ぶ

`test-strategy.md` の分類に従い、対象ごとに適切な手段を選択する。

- **UT必須**: ユニットテストを書く
- **統合テストで十分**: 統合テスト用の検証項目を `red-summary.md` に記録
- **新規テスト不要**: 検証項目だけを `red-summary.md` に記録

`.claude/rules/go-test.md` のルールにも従う:

1. 受け入れ条件が静的解析でカバーできるか → できるなら静的解析テストを優先
2. テスト対象がインターフェース DI 済みか → 済みならそのインターフェースをモック
3. モックが生成コードの内部構造に密結合するか → 密結合するなら静的解析テストで代替

### 3. テストを書く

正常系・異常系・境界値をカバーするテストを書く。既存テストパターンがあればそれに従う。同一コードパスのバリエーションはテーブルドリブンテストでまとめる。

Assertion Roulette を避ける（1テストケースに無関係なアサーションを詰め込まない）。

### 4. 受け入れ条件クロスチェック

Issue 本文の受け入れ条件を1つずつ確認し、各条件に対応するテストまたは検証項目が最低1つ存在することを確認する。

- 受け入れ条件に対応するテストがなければ追加する
- 契約と受け入れ条件が矛盾する場合は、Issue の受け入れ条件を優先する
- 非機能制約もテスト可能なら検証を追加する

### 5. Red を確認する

`just test` を実行し、コンパイルが通ること + テストが失敗すること（Red）を確認する。コンパイルエラーが出た場合は契約定義に問題があるので修正する。

### 6. red-summary.md を出力する

`.output/issue-implement2/{issue_number}/red-summary.md` に以下を記録する:

```markdown
# Red Summary

## Added Tests
- 追加したテスト一覧（ファイル名:テスト名）

## Deferred Checks
- テスト不要または統合テストで担保する項目

## Initial Failures
| 種別 | テスト名またはエラー | 概要 |
|------|----------------------|------|
```

## フロントエンドのテスト

React コンポーネントのテストは Go ほど TDD が自然ではなく、テスト基盤の有無にも依存する。プロジェクトにフロントエンドのテスト基盤がない場合、Go サービスのテストを優先し、フロントエンドのテストはスキップしてよい。
