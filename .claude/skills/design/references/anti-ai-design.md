# anti-AI デザインルール

AI が生成する UI の「AIっぽさ」を回避するためのルール集。デザイン生成時に必ず参照し、全ルールに準拠していることを確認する。

**参照元:**
- `.output/ai-design-looks-ai-like-research.md`
- `.output/research-ai-design-to-code-20260324.md`

---

## 1. 禁止事項（これを使うと即座に「AIっぽく」なる）

### 1.1 フォント

以下のフォントを単独で使用してはならない。これらは AI 生成 UI の最も顕著なシグナルである。

- **Inter** — ほぼすべての AI デザインツールのデフォルト。「AI の Comic Sans」と揶揄される
- **Roboto** — Google のデフォルトフォントとして LLM のトレーニングデータに過剰に含まれる
- **Arial** — 汎用すぎて個性がゼロ
- **Space Grotesk** — AI テンプレートで多用される

**例外:** プロジェクトのデザイントークン（`@theme` や Tailwind 設定）で既に定義されているフォントはそのまま使用してよい。トークンに定義されたフォントを変更したい場合は、デザイントークンの更新をユーザーに提案する。

代替: デザイントークンにフォントが未定義の場合はユーザーに確認する。

### 1.2 配色

- **紫〜青グラデーション**（`from-purple-* to-blue-*`, `from-indigo-* to-purple-*`）を使用しない。Tailwind v3 のデフォルトボタン色 `indigo-500` に由来し、LLM のトレーニングデータに刷り込まれている
- **ダークモードの cyan/purple アクセント**（`text-cyan-400`, `text-purple-400` on dark background）を避ける
- **デフォルトカラーをそのまま使わない** — プロジェクトのデザイントークンで定義されたカラーパレットを必ず使用する

### 1.3 レイアウト

- **角丸カードの多重入れ子**（Card Nesting Problem）を避ける。カードの中にカードがある構造は AI 生成の典型。最大1階層まで
- **3列均等グリッドの Feature セクション**（`grid-cols-3` + アイコン + 見出し + 説明文の繰り返し）を避ける
- **Hero + CTA の定型パターン**（中央揃え見出し + サブテキスト + 2つのボタン）をそのまま使わない
- **グラスモーフィズムの過剰適用**（`backdrop-blur-*` + `bg-white/10`）を避ける

### 1.4 装飾

- **グラデーションの背景**を安易に使わない（ソリッドカラーまたはテクスチャを検討する）
- **過剰な `rounded-*`** — すべての要素を `rounded-xl` や `rounded-2xl` にしない。角丸の大きさに意図的な差をつける

## 2. 必須事項（これを守ることで「AIっぽさ」から脱する）

### 2.1 デザイントークンの強制参照

デザイン生成前に、プロジェクトのデザイントークン定義を必ず読み込む。

- **EC サイト（React）**: `services/ec-site/frontend/src/index.css` の `@theme` セクション
- **管理画面（templ）**: Tailwind 設定ファイル（存在する場合）

トークンに定義されたカラー・フォント・スペーシングのみを使用する。トークンが未定義の値を使う場合は、先にトークンへの追加を提案する。

### 2.2 具体的なビジュアルスタイルの指定

「clean」「modern」「minimal」のような曖昧なキーワードではなく、確立されたビジュアルスタイルを選択してからデザインに着手する。

使用可能なスタイル例:
- エディトリアル（雑誌的レイアウト・強い見出し・グリッドの意図的な崩し）
- ブルータリスト（太い罫線・モノスペースフォント・生々しいレイアウト）
- スイスデザイン（グリッドの厳格さ・サンセリフ・情報の階層化）
- マキシマリスト（大胆なカラー・大きなタイポグラフィ・レイヤーの重なり）
- スキューモーフィズム（質感・影・立体感）
- レトロ / Y2K（ピクセル感・ビビッドカラー・懐かしさ）

ユーザーが好みを示さない場合は、2〜3のスタイル候補を提示して選択してもらう。

### 2.3 タイポグラフィの一貫性

- **タイプスケール**を定義してから使う。見出し（h1〜h4）と本文のサイズ・ウェイト・行間を事前に決定する
- セクション間でフォントサイズ・ウェイトがばらつかないようにする
- 見出しと本文の間に明確な視覚的階層（ヒエラルキー）を持たせる

### 2.4 余白・スペーシングの一貫性

- **スペーシングスケール**（4px, 8px, 16px, 24px, 32px, 48px, 64px 等）を定義し、一貫して使用する
- セクション間の余白は統一する（近接の原則: 関連する要素は近く、無関連は遠く）
- padding / margin の値を場当たり的に変えない

### 2.5 コンテンツの適量

- 必要以上に要素を詰め込まない（「Frankenstein レイアウト」の回避）
- 各セクションに明確な目的を持たせる。目的のない装飾セクションは削除する
- 一画面に表示する情報量を意図的に制限する

## 3. 段階的詳細化

デザインは以下の順序で段階的に詳細化する。一度に完璧を求めない。

1. **構造**: ページ構成・セクション配置・情報の階層
2. **配色**: デザイントークンに基づくカラーの適用
3. **タイポグラフィ**: フォント・サイズ・ウェイト・行間の設定
4. **スペーシング**: 余白・padding・margin の調整
5. **インタラクション**: ホバー・フォーカス・遷移の設定

各段階でユーザーに確認を取り、承認後に次の段階に進む。

---

## 4. 修正時のルール

- 1回の修正で1つの変更に絞る
- 変更しない部分を明示する（「ナビゲーションは変更せず、ヒーロセクションのみ修正」）
- 変更前後で何が変わったかを説明する
