# Issue 起票・ファイル復元フロー

Phase 3 で実施する Issue 起票とデザイン作業ファイルの復元手順。

---

## 1. 成果物の書き出し

デザイン作業の成果物を `.output/design/{session_id}/` に書き出す。`session_id` はタイムスタンプ（`YYYYMMDD-HHMMSS`）とする。

書き出す内容:
- デザインのスクリーンショット（ユーザーが提供した場合）
- 変更したファイルの一覧と概要
- デザイン判断の記録（選択したビジュアルスタイル、カラー、フォント等）

```bash
# 例
mkdir -p .output/design/20260325-143000/
```

---

## 2. Issue の起票

`.github/ISSUE_TEMPLATE/feature.md` のテンプレートに沿って Issue を作成する。

### Issue 本文の構成

```markdown
## 概要

{対象画面名} のデザイン実装。/design スキルで作成したデザインをプロダクションコードに反映する。

デザイン成果物: `.output/design/{session_id}/`

## 方針

{Phase 1〜2 で確定したデザイン方針の概要}
- ビジュアルスタイル: {選択したスタイル}
- 主要な変更点: {箇条書き}

## 受け入れ条件

- [ ] {デザインで確定した各要素の実装条件}
- [ ] レスポンシブ対応（モバイル・デスクトップ）
- [ ] デザイントークンに準拠したカラー・フォント・スペーシング

## 補足

- デザイン作成時の対話ログ: {worktree ブランチ名 or セッション ID}
- 関連 Issue: {元の Issue 番号があれば}
```

### 起票手順

1. Issue 本文をテキスト出力でユーザーに提示する
2. 「問題なければOKと入力してください。修正があればお願いします。」と表示して入力を待つ
3. 承認後、`gh issue create` で Issue を起票する

```bash
gh issue create --title "{対象画面名} のデザイン実装" --body "$(cat <<'EOF'
{Issue 本文}
EOF
)"
```

---

## 3. ファイルの復元

Issue 起票後、worktree 上のデザイン作業ファイルを復元する。worktree はデザイン専用であり、実装は `/issue-implement` が別の worktree で行うため、作業ブランチの破棄で復元する。

### 手順

1. `.output/design/{session_id}/` に成果物が書き出されていることを確認する
2. worktree のブランチ上のデザイン作業で変更・追加したファイルのみを復元する

```bash
# 変更したファイルを元に戻す（git diff で変更ファイルを特定）
git checkout -- services/{対象サービス}/

# デザイン作業で追加した新規ファイルのみ削除（対象ディレクトリを限定）
git clean -fd services/{対象サービス}/
```

**注意:** `git clean -fd` をリポジトリルートで実行しない。デザイン作業と無関係な untracked ファイルが削除される。対象サービスのディレクトリに限定すること。

3. ユーザーに以下を案内する:

```
Issue #{番号} を作成しました。

デザインの実装は以下のコマンドで開始できます:
/issue-implement #{番号}

デザイン成果物は .output/design/{session_id}/ に保存されています。
```

---

## 4. 注意事項

- `.output/` 配下のファイルは `.gitignore` 対象のため、git 操作で消えない
- Issue 本文にはデザインの概要のみ記載し、詳細は `.output/` の成果物を参照する形とする
- ユーザーの承認なしに `gh issue create` や `git checkout` を実行しない
