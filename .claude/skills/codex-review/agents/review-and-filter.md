# Codex Review サブエージェント: レビュー実行 + 対応合理性フィルタリング

codex exec でコードレビューを実行し、対応合理性のある指摘だけを構造化して返すサブエージェント。

## 入力パラメータ（親エージェントから渡される）

- `review_target`: レビュー対象（ファイルパスのリスト or diff 指定文字列）
- `review_type`: "file" or "diff"
- `perspectives`: レビュー観点（なければ空文字列）
- `working_directory`: 作業ディレクトリのパス
- `output_path`: 出力ファイルパス（例: `.output/codex-review/20260320-backend-cmd-api-main.md`）

## Phase 1: Codex によるレビュー実行

### 1a. プロンプトの構築

review_type に応じてプロンプトを組み立てる。プロンプトの核心は「コンテキストを広く見せて具体的な出力を要求する」こと。対象ファイルだけでなく関連ファイル（import先、DB スキーマ、設定ファイル等）も確認するよう指示することで、表面的な指摘ではなくプロジェクト固有の問題を引き出す。

**ファイル指定の場合:**
```
以下のファイルをコードレビューしてください。
{perspectives が指定されている場合: "特に以下の観点を重視してください: {perspectives}"}

対象ファイル:
{ファイルパスのリスト}

## レビューの進め方
1. まず対象ファイルを読んでください
2. 次に、対象ファイルが import/参照している関連ファイル（型定義、DB スキーマ、設定、他のハンドラ等）も確認してください。これにより対象コードの設計意図や制約を把握できます
3. その上で、以下の観点で指摘を洗い出してください

## 各指摘に含める内容（すべて必須）
- 該当ファイルパスと行番号（できるだけ正確に）
- 問題の具体的な説明
- **なぜ問題なのか**: 対応しない場合に起きる具体的なシナリオ（「〜の場合に〜が起きる」の形で）
- **修正案**: 必ず具体的なコード例を含めてください。「〜を追加する」だけではなく、実際のコードスニペットを示してください
- **修正の影響範囲**: この修正が他のファイルや機能に波及するか

指摘の粒度は細かくて構いません。後工程でフィルタリングを行うため、気になる点はすべて挙げてください。
```

**git diff 指定の場合:**
```
以下の git diff の内容をコードレビューしてください。
{perspectives が指定されている場合: "特に以下の観点を重視してください: {perspectives}"}

差分は以下のコマンドで取得できます:
git diff {diff指定文字列}

## レビューの進め方
1. まず差分全体を確認してください
2. 変更されたファイルの変更前後のコンテキスト（周辺コード、関連ファイル）も確認してください。差分だけでは設計意図や既存の制約が見えないため、ファイル全体や関連する定義も読んでください
3. その上で、以下の観点で指摘を洗い出してください

## 各指摘に含める内容（すべて必須）
- 該当ファイルパスと行番号（できるだけ正確に）
- 問題の具体的な説明
- **なぜ問題なのか**: 対応しない場合に起きる具体的なシナリオ（「〜の場合に〜が起きる」の形で）
- **修正案**: 必ず具体的なコード例または修正手順を含めてください。ドキュメントの場合は修正後の文面を示してください
- **修正の影響範囲**: この修正が他のファイルや機能に波及するか

指摘の粒度は細かくて構いません。後工程でフィルタリングを行うため、気になる点はすべて挙げてください。
```

### 1b. Codex 実行

構築したプロンプトを `codex exec` で実行する:

```bash
cd {working_directory} && codex exec --full-auto -q "{プロンプト}"
```

- `--full-auto`: codex がファイル読み取り・コマンド実行を自律的に行えるようにする
- `-q`: 不要な TUI 出力を抑制する

codex が失敗した場合（exit code != 0、またはエラーメッセージが出力された場合）は、エラー内容を親エージェントに返す。

## Phase 2: 対応合理性フィルタリング

Codex の出力から指摘を抽出し、各指摘の「対応合理性」を判定する。

### 除外基準（以下に該当する指摘は除外する）:

1. **エッジケースで影響なし**: 発生確率が極めて低く、発生しても影響が軽微。通常の利用パターンでは到達しないコードパスに関する指摘。
2. **実装との適合性が低い**: プロジェクトのアーキテクチャ・設計方針・技術スタックと合致しない提案。例えば、フレームワークが推奨するパターンと異なる方法を提案している場合。
3. **コスト対効果が見合わない**: 修正に大きなリファクタリングが必要だが、得られる改善が限定的。
4. **スタイル上の好み**: 機能的な違いがなく、単なるコーディングスタイルや命名の好みに過ぎない指摘。

### 判定プロセス:

各指摘について以下を評価する:

- **対応しない場合のリスク**: 具体的にどのような問題が起きうるか
- **発生確率**: そのリスクが実際に顕在化する可能性はどの程度か
- **影響範囲**: 問題が発生した場合にどこまで波及するか
- **修正コスト**: 対応に必要な労力と変更範囲

上記を総合して「対応合理性あり」「対応合理性なし」を判定する。

**判定の方向性**: 迷った場合は「対応合理性あり」にする。人間がレビューして最終判断できるよう、偽陰性（本当は対応すべき指摘を除外してしまう）は避ける。偽陽性（不要な指摘が残る）は人間が無視できるので許容する。

### 適合性チェック:

「実装との適合性」を判定するために、必要に応じてプロジェクトの関連ファイルを Read で確認する:
- 既存コードのパターンや規約
- フレームワーク・ライブラリの使い方
- プロジェクト構成（go.mod, package.json 等）
- ADR（docs/adr/）に記載された設計方針

## Phase 3: 結果の構造化

`output_path` に指定されたパスへ Markdown ファイルを出力する。出力前に親ディレクトリが存在することを確認し、なければ `mkdir -p` で作成する。

**severity の判定基準:**
- **high**: セキュリティ脆弱性、データ損失リスク、本番障害に直結する問題
- **medium**: バグの温床、パフォーマンス劣化、保守性の低下など中期的に問題になるもの
- **low**: 改善が望ましいが緊急性はないもの

### Markdown 出力フォーマット:

```markdown
# Codex Review Result

## Summary
- **Target**: {target}
- **Perspectives**: {perspectives}
- **Findings**: {after_filter} (before filter: {before_filter}, filtered out: {filtered_out})

## Recommended Action Order

最も影響が大きく修正コストが低い指摘から順に、対応推奨順を示す。

1. **#{id} {issue}** — {1行の理由: なぜ最初に対応すべきか}
2. ...

## Findings

### 1. [{SEVERITY}] {issue}
- **File**: {file}:{lines}
- **Category**: {category}
- **Reason**: {reason}
- **Impact if Ignored**: {対応しない場合の具体的シナリオ}
- **Tradeoff**: {修正コスト vs 得られる改善を具体的に。「低い/高い」だけでなく、何が必要かを記述}
- **Affected Scope**: {この修正が波及するファイル・機能の範囲}
- **Suggestion**:

{修正案の説明}

```{lang}
{具体的なコード例。すべての指摘に必須}
```

---

## Filtered Out

| # | Issue | Reason |
|---|-------|--------|
| 1 | {issue} | {filter_reason} |
```
