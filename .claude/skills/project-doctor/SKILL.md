# Project Doctor

プロジェクトのドキュメントとコードの整合性を横断的に診断するスキル。3つのフェーズで段階的にチェックを実行し、問題の検出と修正アクションの提示を行う。

## 実行モード

`$ARGUMENTS` に応じて実行範囲を決定する:
- 引数なし or `full`: Phase 1 → 2 → 3 を順次実行
- `phase1` / `phase2` / `phase3`: 特定フェーズのみ
- チェック番号（`1-1`, `2-3` 等）: 個別チェックのみ

Phase 2 を単独実行する場合は Phase 1 の結果がないため、全ドキュメントを健全として扱う。

---

## Phase 1: ドキュメント自体の健全性

Phase 2 でコード整合性をチェックする前提として、ドキュメント自体が信頼できる状態かを検証する。3つのチェックを**サブエージェントで並列実行**する。

### Check 1-1: ADR 鮮度チェック

Skill ツールで `/adr check` を実行し、結果をそのまま Phase 1 レポートに取り込む。

`/adr check` の出力には以下が含まれる:
- 各 ADR の `last-validated` 経過日数と閾値超過の有無
- Prerequisites に記載された依存先 ADR の status 整合性
- Accepted Tradeoffs の監視対象の抽出

結果から各 ADR の健全性を判定する:
- ✅ 健全: 鮮度 OK、依存先も正常
- ⚠️ 要レビュー: 鮮度超過 or 依存先に問題あり

### Check 1-2: ガイド陳腐化チェック

`docs/guides/` 配下の全ガイドを読み、以下を検証する:

1. **コード例の参照先が存在するか** — ガイド内のコード例に含まれるパッケージ名・型名・関数名が現在のコードベースに存在するか Grep で確認する。ガイドの `scope` frontmatter に記載されたディレクトリに絞って検索する
2. **scope のディレクトリが存在するか** — frontmatter `scope` の値（backend, admin-ui, frontend 等）に対応するサービスディレクトリが存在するか Glob で確認する
3. **detectable_by_linter フラグの整合性** — `detectable_by_linter: true` のガイドがある場合、対応する linter ルールが設定ファイル（`.golangci.yml`, `oxlint.json` 等）に含まれているか確認する

各ガイドの判定:
- ✅ 健全: 参照先がすべて存在
- ⚠️ 要確認: 一部の参照先が見つからない（具体的な箇所を列挙）
- ❌ 陳腐化: 主要な参照先が存在しない

**「計画的未作成」の判定:** README やドキュメント内で「今後追加」「計画中」「将来」等と明記されているディレクトリやファイルが存在しない場合は、陳腐化ではなく計画的な未作成と判定する。この場合は ⚠️ ではなく ℹ️ Info として報告する（「計画的未作成: react/ — README に『今後追加』と明記」のように理由を付記する）。

### Check 1-3: ドキュメント内部リンク整合性

`docs/` 配下と `CLAUDE.md` の全 Markdown を読み、以下を検証:

1. **Markdown リンクの検証** — `[text](path)` 形式のリンクで、相対パスの参照先ファイルが存在するか
2. **ADR 間の相互参照** — 本文中の `ADR-NNNN` 形式の参照に対応する ADR ファイルが `docs/adr/` に存在するか
3. **frontmatter の相互参照** — `superseded-by` や依存先 ADR 番号が実在するか
4. **CLAUDE.md からの参照** — CLAUDE.md に記載されたパス（`docs/c4_overview.md`, `docs/adr/`, `docs/guides/` 等）が存在するか

## Phase 2: ドキュメントとコードの整合性

Phase 1 の結果を入力として受け取る。**Phase 1 で健全（✅）と判定されたドキュメントのみ**を対象にチェックする。⚠️ / ❌ のドキュメントに関連するチェックはスキップし、レポートに「Phase 1 で問題検出のためスキップ」と記載する。

4つのチェックを**サブエージェントで並列実行**する。各サブエージェントには Phase 1 の結果（健全/非健全のリスト）を渡す。

### Check 2-1: ADR 準拠チェック

Phase 1 で健全と判定された各 ADR の Decision Outcome を読み、以下を検証する:

1. **採用技術の使用確認** — 選択された技術が実際にコードで使われているか
   - `go.mod`, `package.json` の依存関係を確認
   - import 文を Grep でスキャン
   - 設定ファイル（docker-compose.yml, linter 設定等）の存在と内容を確認
2. **排除技術の混入検出** — Excluded Options や不採用の選択肢で言及された技術がコードに混入していないか
   - 例: ADR-0002 で pnpm を採用 → `yarn.lock` や `package-lock.json` が存在しないか
   - 例: ADR-0010 で oxlint を採用 → `.eslintrc` が存在しないか
3. **withdrawn ADR の残骸** — `status: withdrawn` の ADR が採用していた技術の依存・設定・コードが残存していないか
4. **ADR 本文中の具体的パッケージ名の照合** — Decision Outcome、Consequences、More Information セクションに記載された具体的なパッケージ名・ライブラリ名（例: `@connectrpc/connect-fastify`、`gocloak` 等）が、実際にコードで使われているか検証する。ADR が「この技術で実現する」と記述しているパッケージが `go.mod` や `package.json` に存在しない場合、ADR の記述と実装の乖離として報告する

### Check 2-2: ガイド準拠チェック

Phase 1 で健全と判定された各ガイドについて、「アンチパターン」に該当するコードがないかスキャンする。

**手順:**
1. ガイドの「アンチパターン」セクションと「具体例」の Bad コード例を読む
2. アンチパターンの特徴から Grep パターンを導出する（例: `handler-testability` ガイドなら `*db.Queries` をフィールドに持つ構造体を検索）
3. Grep で検出可能なパターンは Grep でスキャンする。scope に対応するディレクトリに絞る
4. Grep で検出困難なパターン（構造的な問題、設計パターンの違反等）は、scope 内の主要ファイルを Read して LLM が判定する
5. 検出した箇所を「正しいパターン」の具体例と照合し、正しいパターンに該当する場合は除外する

**同種コンポーネント間の一貫性チェック:**

ガイドのアンチパターンチェックに加えて、同じ役割を持つコンポーネント間でパターンの適用が一貫しているかを検証する。あるコンポーネントがガイドの「正しいパターン」に従っているのに、同種の他のコンポーネントが従っていない場合、不一致として報告する。

例:
- Connect RPC ハンドラが複数ある場合、一部だけがインターフェース経由の DI を使用し、他が `*db.Queries` 直接参照 → 不一致
- 一部のハンドラのみが `Unimplemented*Handler` を埋め込んでいる → 不一致（新 RPC 追加時に片方だけコンパイルエラーになるリスク）

不一致は Warning として報告し、どのコンポーネントが「正しいパターン」側かを明示する。

**偽陽性の抑制:**
- ガイドの「判断基準」セクションがある場合はそれに従って判定する
- 判定に自信がない場合は `⚠️ 要確認` として報告し、断定的な表現は避ける
- テストコード内のパターンはガイドの scope が明示的にテストを含まない限り除外する

### Check 2-3: C4 概要ドリフトチェック

`docs/c4_overview.md` を読み、以下を実装と照合する:

1. **File Mapping の検証** — 記載されたディレクトリとエントリポイントが実在するか Glob で確認
2. **サービス一覧の検証** — Mermaid 図のサービスに対応するディレクトリが存在するか。逆に `services/` 配下に図に記載されていないサービスがないか
3. **ポート番号の検証** — 図中のポート番号が実際のコード（`main.go` の Listen ポート、`docker-compose.yml` のポートマッピング等）と一致するか
4. **通信経路の検証** — 実線（実装済み）の経路が実際にコードで実装されているか確認する。点線（未実装）の経路が実装済みになっていないかも確認する
5. **Proto 定義テーブルの検証** — 記載された Proto パスが実在するか。`proto/` 配下に記載されていない `.proto` ファイルがないか
6. **Level 間の整合性検証** — C4 の複数 Level（Level 1 System Context、Level 2 Container 等）が存在する場合、同じコンポーネントや通信経路の表現が Level 間で矛盾していないか確認する。特に実線（実装済み）と点線（未実装）の使い分けが Level 間で一致しているかを検証する（例: Level 1 で実線なのに Level 2 で点線、またはその逆）

### Check 2-4: Proto/実装整合性チェック

Proto 定義と Go ハンドラの整合性を検証する:

1. `proto/` 配下の `.proto` ファイルから全 `service` と `rpc` 定義を抽出する
2. 各サービスの Go ハンドラディレクトリで、対応する RPC メソッドが実装されているか確認する（Connect RPC の場合、ハンドラは `func (s *Server) MethodName(ctx, req) (res, error)` の形式）
3. 未実装の RPC を一覧化する。各未実装 RPC について `docs/todo/system_development.md` を参照し、計画済みかどうかを付記する（計画済みなら `ℹ️ Info`、未計画なら `⚠️ Warning`）

## Phase 3: プロジェクト全体の健全性

Phase 1-2 とは独立して実行可能。

### Check 3-1: CLAUDE.md 正確性チェック

1. **参照パスの検証** — CLAUDE.md 内の全パス参照（`docs/c4_overview.md`, `docs/adr/`, `docs/guides/`, `.output/` 等）が存在するか Glob で確認
2. **コマンド参照の検証** — `just --list` を Bash で実行し、CLAUDE.md が参照するコマンドが有効か確認
3. **スキル参照の検証** — CLAUDE.md で言及されているスキルが `.claude/skills/` に存在するか確認

## レポート生成

全チェック完了後、サマリレポートを `.output/project-doctor-report.md` に出力する。

### レポート構成

```markdown
# Project Doctor Report

**実行日**: YYYY-MM-DD
**実行範囲**: full / phaseN / チェック番号

## サマリ

| Phase | チェック | 結果 | 指摘数 |
|-------|---------|------|--------|
| 1 | 1-1 ADR 鮮度 | ✅ / ⚠️ / ❌ | N |
| 1 | 1-2 ガイド陳腐化 | ... | ... |
| 1 | 1-3 内部リンク整合性 | ... | ... |
| 2 | 2-1 ADR 準拠 | ... | ... |
| 2 | 2-2 ガイド準拠 | ... | ... |
| 2 | 2-3 C4 ドリフト | ... | ... |
| 2 | 2-4 Proto/実装整合性 | ... | ... |
| 3 | 3-1 CLAUDE.md 正確性 | ... | ... |

## 指摘一覧（重要度順）

### Critical（即座に修正が必要）

- [チェック番号] 説明 — `ファイルパス:行番号`

### Warning（確認・対応を推奨）

- [チェック番号] 説明 — `ファイルパス:行番号`

### Info（参考情報）

- [チェック番号] 説明

## 修正アクション

| # | 指摘 | アクション | 自動修正 |
|---|------|-----------|---------|
| 1 | ADR-NNNN 鮮度超過 | `/adr re-evaluate NNNN` を実行 | 手動 |
| 2 | C4 ポート不一致 | c4_overview.md のポート番号を更新 | 可能 |
```

### 重要度の判定基準

- **Critical**: コードとドキュメントが矛盾しており、誤った情報に基づいて開発が進むリスクがある（withdrawn ADR の技術が残存、C4 のサービス構成が実態と不一致 等）
- **Warning**: 対応が望ましいが即座のリスクはない（ADR 鮮度超過、ガイド準拠の可能性ある違反、参照切れ 等）
- **Info**: 認識しておくべき情報（計画済みの未実装 RPC 等）

### 自動修正

レポート提示後、自動修正可能な指摘についてユーザーに確認する:

1. 自動修正可能な指摘を一覧で提示する
2. ユーザーが承認した項目のみ修正を実行する
3. 修正は diff 形式で提示してから適用する

自動修正の対象例:
- C4 概要のポート番号・ディレクトリパス更新
- CLAUDE.md の参照パス修正
- ADR の `last-validated` 日付更新（ユーザーが「現行決定は有効」と判断した場合）

手動対応が必要な例:
- ADR の再評価（`/adr re-evaluate` で別途実施）
- ガイドのコード例更新（コンテキスト理解が必要）
- アンチパターン違反のコード修正（設計判断が必要）

## 設計原則

1. **サブエージェントで並列実行** — 各 Phase 内のチェックはサブエージェントに委譲し並列実行する。Phase 間は依存関係に従って順次実行する
2. **Phase 1 → Phase 2 の依存** — 信頼できないドキュメントに対するコード整合性チェックは価値が低い。Phase 1 の結果で Phase 2 のチェック対象をフィルタする
3. **既存スキルへの委譲** — ADR チェックは `/adr check` に委譲し、ロジックの重複を避ける
4. **偽陽性より見逃しを嫌う** — 判断に迷う場合は `⚠️ 要確認` として報告する。ただし明らかに問題ない箇所は報告しない
5. **修正の自動化は慎重に** — 自動修正は事実の更新（パス・番号の修正）に限定し、設計判断を伴う修正は手動に委ねる
