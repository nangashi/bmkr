# Phase 4: 計画レビュー — 詳細手順

計画作成後、事前検証 + Codex レビュー + 条件付きレビューの3段階でレビューを実施する。

レビューの目的は計画の品質向上であり、計画を書いた LLM 自身のバイアスを検出するために、Codex レビューは別 LLM（Codex CLI）で実行する。

---

## レビューの流れ

1. 計画を `.output/issue-plan/{issue_number}/plan.md` に書き出す
2. **事前検証（メインエージェント）** — 機械的な突き合わせで事実誤りを検出
3. **Codex レビュー** + **条件付きレビュー**（該当する場合）を並列実行
4. 全レビュー結果を統合し、計画を修正
5. 修正済み計画を Issue コメントとして投稿

---

## 事前検証（メインエージェントが実施）

Codex レビュー・条件付きレビューの前に、計画と Issue から検証可能な事実を機械的に抽出・突き合わせる。分析的レビューでは大きな問題に注意が集中し、値の矛盾や命名の逸脱といった細部を見落としやすい。事前検証はこれを防ぐ。

### 数値・条件の突き合わせ

1. Issue 本文の受け入れ条件から、具体的な値を含む条件を全て抽出する
   - 例: 「有効期限は15分」「デフォルト値は20」「最大値は100」
2. 計画の全ステップから、同じ項目に言及している箇所を検索する
3. 値が一致しているか突き合わせる（対応表の値と本文の値の両方を確認する）
4. 不一致があれば記録する

### 命名パターンの突き合わせ

1. 計画の外部契約で新規に導入する名前を全て列挙する
   - sqlc クエリ名、RPC 名、メッセージ名、ルートパス等
2. 対応する既存コードの命名パターンを実際に読んで確認する
   - sqlc クエリ: `GetXxx`（:one）、`ListXxx`（:many）、`CreateXxx`（:one）等
   - Proto RPC: 既存 RPC 名の動詞パターン
   - ルートパス: 既存ルートの命名規則
   - ディレクトリ: 既存サービスのディレクトリ構成
3. パターンに沿っていない名前があれば記録する

**注意:** 構造体名・関数名・変数名は計画に含まれない（`/issue-implement` の Phase 1 で決定するため）。計画に含まれる外部契約レベルの名前のみを検証対象とする。

### スコープの突き合わせ

1. Issue 本文の「スコープ外」セクションから全項目を抽出する
2. 計画の全ステップを走査し、スコープ外の項目に該当する作業がないか確認する
   - Proto フィールド、設定値、コメント、テスト項目も含めて確認する
3. 該当があれば記録する

### ファイル存在・ADR 参照の検証

`scripts/verify-plan.sh` を実行する:

```bash
bash .claude/skills/issue-plan/scripts/verify-plan.sh \
  .output/issue-plan/{issue_number}/plan.md
```

スクリプトが存在しない場合は、以下を手動で確認する:
1. 計画の「変更ファイル一覧」で操作が「変更」のファイルが存在するか
2. 計画で参照する ADR 番号が `docs/adr/` に存在するか

### ディレクトリ構成の突き合わせ

1. 計画の「変更ファイル一覧」から新規ファイル・ディレクトリのパスを抽出する
2. 対応する既存サービスのディレクトリ構成を `ls` で確認する
3. 新規ファイルの配置が既存パターンと一致するか突き合わせる
   - 例: 既存サービスが `handler/`, `service/`, `repository/` の構成なら、新規ファイルもこの構成に沿っているか
   - 例: 既存が単一ファイル（`handler.go`）なら、新規も同様か
4. 構成が逸脱している場合は記録する

### 事前検証結果の出力

検出した不一致を `.output/issue-plan/{issue_number}/pre-review.md` に書き出す。この結果は条件付きレビューのサブエージェントに入力として渡す。

## Codex レビュー

計画を書いた LLM（Opus）とは別のモデル（Codex）で、受け入れ条件カバレッジとシンプルさをレビューする。バイアス軽減が主な目的。

以下のプロンプトを構築し、`timeout 300 codex exec --full-auto` に stdin で渡す。codex CLI が利用できない場合は Opus サブエージェントにフォールバックする。

```
以下の実装計画をレビューしてください。

計画:
{plan.md の内容}

Issue 本文:
{Issue 本文}

## レビューの進め方
1. まず以下のレビュー観点ファイルを読んでください:
   - .claude/skills/issue-plan/references/review-acceptance-coverage.md
   - .claude/skills/issue-plan/references/review-simplicity.md
2. 各観点の Step 1（Analysis）に従い、計画の構造を把握してください
3. 各観点の Step 2（Findings）に従い、指摘を導出してください
4. 必要に応じて、docs/adr/、docs/guides/review/、既存コードも参照してください

## 出力
結果を .output/issue-plan/{issue_number}/review-codex.md に書き出してください。

### フォーマット
観点ごとにセクションを分け、各 finding について以下を含めてください:
- finding_id（通し番号）
- perspective（各観点ファイルの「perspective ラベル」を使用）
- severity（high / medium / low）
- location（計画内の該当箇所: ステップ番号、セクション名等）
- issue（問題の要約）
- analysis（Step 1 で把握した事実）
- reason（なぜ問題か、対応しない場合の具体的シナリオ）
- suggestion（修正案）
```

プロンプト中の `{issue_number}` は実際の Issue 番号に置換すること。`{plan.md の内容}` と `{Issue 本文}` はプロンプト構築時に展開して埋め込む。

## 条件付きレビュー（Sonnet サブエージェント）

以下のレビューは、計画の内容に該当する要素がある場合のみ実行する。該当するものがあれば Codex レビューと**並列起動**する。レビュー観点が構造化された確認タスクのため Sonnet モデルのサブエージェント（`model: sonnet`）で実行する。

### C. セキュリティ設計レビュー（サブエージェント）

**発動条件:** 認証・認可、トークン・セッション、ユーザー入力処理、API エンドポイント追加、データアクセス制御のいずれかを含む場合。

**サブエージェントに伝えること:**
- 計画の内容（plan.md）と作業ディレクトリパス
- 事前検証結果（pre-review.md）
- 出力先パスと「Write で書き出すこと」

**確認してほしい観点（設計フローレベル）:**
- 認証チェックのタイミングと範囲（認証が必要なルートにミドルウェアを配置する計画があるか）
- トークンの発行・検証・失効の処理フロー
- データアクセス制御の設計（誰がどのデータにアクセスできるか）
- エラーレスポンスでの内部情報露出リスク

**注意:** コード実装レベルのセキュリティチェック（入力バリデーションの実装、SQL インジェクション、secrets 非埋め込み等）は `/issue-implement` Phase 4 の `review-security.md` が担当する。ここでは設計フローの漏れに集中する。

深刻度（高/中/低）と対策案を提示する。指摘がなければ「指摘なし」。

出力先: `{作業ディレクトリ}/review-security-design.md`

### D. データアクセス設計レビュー（サブエージェント）

**発動条件:** DB クエリ追加・変更、大量データ処理、API レスポンスに関わる処理、ループ内の外部呼び出しのいずれかを含む場合。

**サブエージェントに伝えること:**
- 計画の内容（plan.md）と作業ディレクトリパス
- 事前検証結果（pre-review.md）
- 出力先パスと「Write で書き出すこと」

**確認してほしい観点（外部契約・設計レベルの問題）:**
- N+1 問題、不要な全件取得、インデックスの考慮漏れ（SQL クエリ仕様に関わるもの）
- ステップ間で明らかに非効率なデータフロー（例: 同じデータを複数回取得する設計）
- 大量データ処理の方針が計画の制約に含まれているか

**注意:** 実装レベルの最適化（並列化、キャッシュ、上限ガード等）は計画に書かせない。これらは `/issue-implement` の Phase 4 で実装時に判断する。

影響度（高/中/低）と改善案を提示する。指摘がなければ「指摘なし」。

出力先: `{作業ディレクトリ}/review-data-access-design.md`

---

## レビュー結果の統合

全レビュー完了後、メインエージェントが結果を統合する。

1. `.output/issue-plan/{issue_number}/` 配下の事前検証結果 + レビュー結果ファイルを読み込む
2. 各指摘について計画を修正するか判断する:
   - **修正する**: 計画を更新
   - **受容する**: リスクとして計画に記載
   - **却下する**: 実装詳細（How）を計画に追加する指摘は却下する
3. 修正後の計画を `.output/issue-plan/{issue_number}/plan.md` に上書き
4. レビューサマリを Issue コメントとして投稿する

### レビューサマリのコメントフォーマット

```markdown
<!-- issue-plan:review:done -->
## 計画レビュー

### 実施したレビュー
- [x] 事前検証（数値・命名・スコープ・ファイル存在・ADR参照・ディレクトリ構成）
- [x] Codex レビュー（受け入れ条件カバレッジ・シンプルさ）
- [x] セキュリティ設計レビュー（条件該当のため実施）
- [ ] データアクセス設計レビュー（条件非該当のため省略）

### 指摘と対応

| 観点 | 指摘 | 対応 |
|------|------|------|
| {観点} | {指摘内容} | 修正済み / リスクとして受容 |

### 修正内容
{計画に加えた修正の要約。なければ「修正なし」}
```
