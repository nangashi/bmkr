# Phase 4: 計画レビュー — 詳細手順

計画作成後、事前検証 + 分析的レビューの2段階でレビューを実施する。

レビューの目的は計画の品質向上であり、計画を書いた LLM 自身のバイアスを検出するために、分析的レビューは別のサブエージェントで実行する。

---

## レビューの流れ

1. 計画を `.output/issue-plan/{issue_number}/plan.md` に書き出す
2. **事前検証（メインエージェント）** — 機械的な突き合わせで事実誤りを検出
3. 分析的レビュー（サブエージェント）を並列実行 — 事前検証で拾えない設計判断の妥当性を確認
4. 条件付きレビュー（該当する場合）を実行
5. 事前検証 + 分析的レビューの結果を統合し、計画を修正
6. 修正済み計画を Issue コメントとして投稿

---

## 事前検証（メインエージェントが実施）

分析的レビューの前に、計画と Issue から検証可能な事実を機械的に抽出・突き合わせる。分析的レビューでは大きな問題に注意が集中し、値の矛盾や命名の逸脱といった細部を見落としやすい。事前検証はこれを防ぐ。

### 数値・条件の突き合わせ

1. Issue 本文の受け入れ条件から、具体的な値を含む条件を全て抽出する
   - 例: 「有効期限は15分」「デフォルト値は20」「最大値は100」
2. 計画の全ステップから、同じ項目に言及している箇所を検索する
3. 値が一致しているか突き合わせる（対応表の値と本文の値の両方を確認する）
4. 不一致があれば記録する

### 命名パターンの突き合わせ

1. 計画で新規に導入する名前を全て列挙する
   - RPC 名、メッセージ名、クエリ名、関数名、ファイル名、変数名
2. 対応する既存コードの命名パターンを実際に読んで確認する
   - sqlc クエリ: `GetXxx`（:one）、`ListXxx`（:many）、`CreateXxx`（:one）等
   - Proto RPC: 既存 RPC 名の動詞パターン
   - Go ファイル: 既存ファイル名のパターン
   - ディレクトリ: 既存サービスのディレクトリ構成
3. パターンに沿っていない名前があれば記録する

### スコープの突き合わせ

1. Issue 本文の「スコープ外」セクションから全項目を抽出する
2. 計画の全ステップを走査し、スコープ外の項目に該当する作業がないか確認する
   - Proto フィールド、設定値、コメント、テスト項目も含めて確認する
3. 該当があれば記録する

### ファイル存在・ADR 参照の検証

`scripts/verify-plan.sh` を実行する:

```bash
bash .claude/skills/issue-plan/scripts/verify-plan.sh \
  .output/issue-plan/{issue_number}/plan.md
```

スクリプトが存在しない場合は、以下を手動で確認する:
1. 計画の「変更ファイル一覧」で操作が「変更」のファイルが存在するか
2. 計画で参照する ADR 番号が `docs/adr/` に存在するか

### 事前検証結果の出力

検出した不一致を `.output/issue-plan/{issue_number}/pre-review.md` に書き出す。この結果は分析的レビューのサブエージェントに入力として渡す。

---

## 分析的レビュー（サブエージェント）

事前検証で拾えない「設計判断の妥当性」をレビューする。事前検証で検出済みの問題（命名、スコープ逸脱等）は繰り返さず、アーキテクチャレベルの判断に集中する。

### A. 整合性レビュー（サブエージェント）

既存コードのパターンと計画の一貫性を確認する。

**サブエージェントに伝えること:**
- 計画の内容（plan.md）と作業ディレクトリパス
- 事前検証結果（pre-review.md）
- 出力先パスと「Write で書き出すこと」

**確認してほしい観点（事前検証で拾えないもの）:**
- 同種の処理（エラーハンドリング、DB アクセス、ルーティング等）が既存実装と同じ方法で書かれているか
- モジュール構成・ファイル分割が既存パターンに沿っているか
- `docs/adr/` の決定事項に反していないか

**ポイント:** 計画の各ステップで変更するファイルの「周辺にある既存ファイル」を実際に読んで比較すること。計画だけ読んで判断せず、コードベースの事実を確認する。

不一致があれば既存パターンに合わせた修正案を提示する。指摘がなければ「指摘なし」と記載し、確認した観点を列挙する。

出力先: `{作業ディレクトリ}/review-consistency.md`

### B. シンプルさレビュー（サブエージェント）

計画が必要最小限の複雑さに収まっているか確認する。LLM は「良いコード = 抽象化されたコード」と考えがちで、これを検出するのがこのレビューの存在意義。

**サブエージェントに伝えること:**
- 計画の内容（plan.md）、Issue 本文、作業ディレクトリパス
- 事前検証結果（pre-review.md）
- 出力先パスと「Write で書き出すこと」

**確認してほしい観点（事前検証で拾えないもの）:**
- 1箇所でしか使わないのに導入されたデザインパターンはないか（3回の類似コード > premature abstraction）
- 固定で良い値を環境変数や設定ファイルに切り出していないか
- 標準ライブラリやプロジェクト既存の仕組みで済むのに新しい依存を追加していないか
- 同じ要件を満たすもっとシンプルな方法はないか

検出した過剰設計があれば、シンプルな代替案を提示する。指摘がなければ「指摘なし」と記載し、確認した観点を列挙する。

出力先: `{作業ディレクトリ}/review-simplicity.md`

---

## 条件付きレビュー

以下のレビューは、計画の内容に該当する要素がある場合のみ実行する。

### C. セキュリティレビュー（サブエージェント）

**発動条件:** 認証・認可、トークン・セッション、ユーザー入力処理、API エンドポイント追加、データアクセス制御のいずれかを含む場合。

**サブエージェントに伝えること:**
- 計画の内容（plan.md）と作業ディレクトリパス
- 出力先パスと「Write で書き出すこと」

**確認してほしい観点:**
- 入力検証・サニタイズの漏れ
- 認証チェックのタイミングと範囲
- トークンの発行・検証・失効の処理
- エラーレスポンスでの内部情報露出
- OWASP Top 10 の観点

深刻度（高/中/低）と対策案を提示する。指摘がなければ「指摘なし」。

出力先: `{作業ディレクトリ}/review-security.md`

### D. パフォーマンスレビュー（サブエージェント）

**発動条件:** DB クエリ追加・変更、大量データ処理、API レスポンスに関わる処理、ループ内の外部呼び出しのいずれかを含む場合。

**サブエージェントに伝えること:**
- 計画の内容（plan.md）と作業ディレクトリパス
- 出力先パスと「Write で書き出すこと」

**確認してほしい観点:**
- N+1 問題、不要な全件取得、インデックスの考慮漏れ
- メモリ上での大量データ保持
- 並列化可能な直列呼び出し
- キャッシュすべきデータの毎回取得

影響度（高/中/低）と改善案を提示する。指摘がなければ「指摘なし」。

出力先: `{作業ディレクトリ}/review-performance.md`

---

## レビュー結果の統合

全レビュー完了後、メインエージェントが結果を統合する。

1. `.output/issue-plan/{issue_number}/` 配下の事前検証結果 + レビュー結果ファイルを読み込む
2. 各指摘について計画を修正するか判断する:
   - **修正する**: 計画を更新
   - **受容する**: リスクとして計画に記載
3. 修正後の計画を `.output/issue-plan/{issue_number}/plan.md` に上書き
4. レビューサマリを Issue コメントとして投稿する

### レビューサマリのコメントフォーマット

```markdown
<!-- issue-plan:review:done -->
## 計画レビュー

### 実施したレビュー
- [x] 事前検証（数値・命名・スコープ・ファイル存在・ADR参照）
- [x] 整合性レビュー
- [x] シンプルさレビュー
- [x] セキュリティレビュー（条件該当のため実施）
- [ ] パフォーマンスレビュー（条件非該当のため省略）

### 指摘と対応

| 観点 | 指摘 | 対応 |
|------|------|------|
| {観点} | {指摘内容} | 修正済み / リスクとして受容 |

### 修正内容
{計画に加えた修正の要約。なければ「修正なし」}
```
