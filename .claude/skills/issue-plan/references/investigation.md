# Phase 2: 調査 — サブエージェント詳細手順

2 つのサブエージェントを **並列** で起動する。各サブエージェントはバイアス防止のため独立したコンテキストで動作する。

---

## 共通: サブエージェントへの指示

Agent ツール（subagent_type=general-purpose, `model: sonnet`）で起動する。コードベース探索・リサーチタスクのため Sonnet で十分な精度が得られる（Anthropic公式のマルチエージェントリサーチパターンに準拠）。各サブエージェントへのプロンプトには以下を含める:

- Issue 本文（構造化済み）
- 作業ディレクトリパス
- 「技術的判断は docs/adr/ で規定」という指示
- 出力先パスと「結果を Write で書き出すこと」という明示的な指示

---

## 影響調査サブエージェント

### 目的

変更が波及する全ファイルを特定し、リスクを洗い出す。

### 伝えるべきこと（WHAT）

- 変更が必要なファイルと、変更が波及するファイルを区別して特定すること
- 自動生成ファイル（buf generate, sqlc generate 等）と手動編集ファイルを区別すること
- 変更の依存順序（何を先にやらないとコンパイルが通らないか）を明らかにすること
- `docs/adr/` の既存 ADR に反する変更がないか確認すること
- API エンドポイント、DB スキーマ、Proto 定義への影響を確認すること

### 出力先

`.output/issue-plan/{issue_number}/impact-investigation.md`

### 出力に含めるべき情報

先頭に `<!-- issue-plan:impact-investigation:done -->` マーカーを付与する。

- **変更対象ファイル**: パス、変更内容の概要、影響度（高/中/低）
- **依存関係**: 変更の波及経路（図やリストで表現）
- **API・スキーマへの影響**: なければ「なし」
- **テストへの影響**: 要修正と要追加
- **関連 ADR**: 番号と関連内容
- **リスク・注意点**: 見落としやすいポイント

具体的なフォーマットはサブエージェントの判断に任せる。上記の情報が網羅されていれば良い。

## 技術調査サブエージェント

### 目的

要件を実現する技術的選択肢を洗い出し、各選択肢の長短を整理する。

### 伝えるべきこと（WHAT）

- 要件を実現する技術的アプローチを複数洗い出すこと（1つしかない場合でもその理由を明記）
- 各アプローチについてプロジェクトの既存技術スタックとの互換性を確認すること
- `docs/adr/` で決定済みの技術選定が制約になっていないか確認すること
- 技術的に不確実な部分があれば明記すること（PoC が必要か判断する材料として）
- context7 プラグインで最新のドキュメントを確認し、LLM の学習データに依存しないこと

### 出力先

`.output/issue-plan/{issue_number}/technical-research.md`

### 出力に含めるべき情報

先頭に `<!-- issue-plan:technical-research:done -->` マーカーを付与する。

- **技術スタック（関連部分）**: この要件に関連する既存技術の概要
- **実現アプローチ**: 各アプローチの概要、メリット、デメリット、互換性、複雑さ
- **技術的制約**: パフォーマンス、セキュリティ等の制約
- **関連 ADR**: 番号と制約・影響
- **不確実な点**: 追加調査や PoC が必要な項目。なければ「なし」

具体的なフォーマットはサブエージェントの判断に任せる。上記の情報が網羅されていれば良い。
