# Phase 4: 計画作成 — 詳細手順

Issue 本文（要件）+ 調査結果 + 方針を入力に実装計画を作成する。

計画は `/issue-implement` が読み取って実装を進めるための入力になる。自己完結的で、計画だけ読めば何を達成すべきか分かる内容にする。

**重要: 計画は What（何を達成するか）と Constraints（制約）に集中する。How（どう実装するか）は `/issue-implement` の各フェーズで決定する。**

---

## 手順

### 1. 入力の収集

以下を読み込む:
- Issue 本文（概要・方針・受け入れ条件・スコープ外・補足）
- Phase 2 の調査結果コメント（影響調査・技術調査）
- Phase 3 の方針検討コメント

### 2. 計画の作成

以下のフォーマットで `.output/issue-plan/{issue_number}/plan.md` に書き出す。

### 3. レビュー

`references/review.md` を Read で読み込み、その内容に従ってレビューを実施する。

### 4. レビュー結果の反映

レビュー指摘を確認し、計画を修正する。修正内容はユーザーに提示する。

### 5. Issue コメント投稿

計画を Issue コメントとして投稿する。レビューサマリを別コメントで投稿する。

---

## 計画フォーマット

```markdown
<!-- issue-plan:plan:done -->
## 実装計画

### 概要

{何を実装するかの要約 — 1-2文}

### 前提

- 採用方針: {Phase 3 で決定した方針}
- 関連 ADR: {関連する ADR 番号}

### 実装ステップ

#### Step 1: {タイトル}

- **ゴール**: {完了状態 — 何が動く・何が存在する状態になるか}
- **外部契約**: {DB クエリ仕様、ルート、API 仕様等。なければ省略}
- **制約**: {ADR 準拠、既存パターン、共存条件等。なければ省略}

#### Step 2: {タイトル}

{同上}

### 変更ファイル一覧

| ファイル / ディレクトリ | 操作 | 概要 |
|---------|------|------|
| {パス} | 新規/変更/削除 | {概要} |

### 受け入れ条件との対応

| 受け入れ条件 | 対応する Step |
|-------------|-------------|
| {条件1} | Step {N} |
| {条件2} | Step {N}, {M} |

### リスクと対策

| リスク | 対策 |
|--------|------|
| {リスク} | {対策} |
```

## 粒度のガイドライン

計画の各項目で適切な粒度を保つための基準。

### ステップの粒度

- **マイルストーン単位で分割する**（タスク単位ではない）。関連する作業はまとめる
- テスト専用ステップ・動作確認ステップは不要（`/issue-implement` の Phase 3・Phase 4 の責務）
- 3〜5 ステップ程度が目安。7 を超えたらタスク粒度になっている可能性が高い

### ゴールの書き方

- 「何をするか」ではなく「何が完了した状態か」を書く
- 検証可能な記述にする（例: 「〜が利用可能な状態」「〜で〜が表示される状態」）

### 外部契約に書くもの（計画で決める）

- SQL クエリの仕様（名前、カラム、ソート順）
- ルートパスとレスポンスの種類
- API のリクエスト/レスポンス仕様
- 画面に表示する項目

### 計画に書かないもの（`/issue-implement` で決める）

以下は Phase 1（I/F設計）または Phase 4（実装）で決定する。計画に含めない。

- 構造体・インターフェースの設計（名前、フィールド、メソッド）
- 依存の持ち方（直接保持 vs インジェクション）
- 処理フローの詳細（並列化、ヘッダ判定方法、エラー処理の具体的コード）
- 定数値（ページサイズ、上限値等）
- コード片（`c.Response().Header().Set(...)` 等）

### 変更ファイル一覧の粒度

- **既存ファイル（変更）**: 正確なパスを記載
- **新規ファイル**: ディレクトリ単位で記載（ファイル名は Phase 1 で決まるため）
- **自動生成ファイル**: 記載不要（コード生成の結果であり計画で指定する意味がない）
