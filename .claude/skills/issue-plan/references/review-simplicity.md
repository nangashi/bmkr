# Review: シンプルさ

## Overview
計画が必要最小限の複雑さに収まっているかをレビューする。LLM は「良い設計 = 抽象化された設計」と考えがちで、これを検出するのがこのレビューの存在意義。

## Step 1: Analysis（計画の構造を把握する）

### マッピング対象
- 計画の各ステップのゴール・外部契約・制約
- 計画の「変更ファイル一覧」で新規作成されるファイル/ディレクトリ
- `docs/guides/review/` 配下のアンチパターンガイド
- 変更対象と同一サービス内の既存コード構成

### 各項目について記録すること
1. 新規導入する抽象層（interface、中間サービス、ヘルパー等）とその必要性
2. 新規導入する外部依存（ライブラリ、外部サービス）と、標準ライブラリや既存の仕組みで代替可能か
3. ステップ数と各ステップの複雑さ — 同じ要件をより少ないステップで実現できないか
4. Phase 2 影響調査の「類似実装」セクションで特定された既存コードと比較して、構成が著しく複雑になっていないか

## Step 2: Findings（分析結果から指摘を導出する）

### 指摘とする基準
- `docs/guides/review/` 配下のアンチパターンに該当する設計
- 標準ライブラリやプロジェクト既存の仕組みで済むのに新しい依存を追加している
- 既存の同種機能と比較して明らかに過剰な構成（例: 既存が 1 ファイルで実現しているのに 3 層に分割）
- 受け入れ条件を満たすもっとシンプルな方法が存在する

### 指摘としない基準
- 既存コードベースで確立されたパターンの踏襲（既存が 3 層構成なら新規も 3 層で良い）
- ADR で決定済みの技術・パターンの採用
- 受け入れ条件で明示的に求められている機能・構成
- 実装の詳細レベルの判断（構造体設計、依存注入方法等は issue-implement の責務）

### perspective ラベル
シンプルさ
