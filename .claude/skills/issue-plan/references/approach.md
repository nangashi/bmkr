# Phase 3: 方針検討 — 詳細手順

調査結果を統合し、実現方針を検討する。バイアスを避けるため、LLM は特定の方針を推奨せず、判断材料の提供に徹する。

---

## 手順

### 1. 調査結果の読み込み

Phase 2 の Issue コメント（影響調査 + 技術調査）を読み込む。コメントが見つからない場合は `.output/issue-plan/{issue_number}/` 配下のファイルをフォールバックとして使用する。

### 2. ADR 作成要否の判定

方針検討に入る前に、技術調査の結果から ADR として記録すべき判断があるか確認する。

**ADR を提案する条件（すべて満たす場合）:**
- 技術的に 2 つ以上の選択肢がある
- この Issue 以外のコードにも影響する（プロジェクト全体への永続的な決定）
- 同じトピックの ADR が `docs/adr/` に存在しない

**ADR の対象となる判断の例:**
- ライブラリ・フレームワークの採用（例: ioredis vs iovalkey）
- アーキテクチャパターンの選択（例: イベント駆動 vs 同期呼び出し）
- 状態管理方式の選択（例: JWT ステートレス vs サーバーサイドセッション）
- 通信プロトコルの選択（例: REST vs gRPC）
- データストアの選択（例: PostgreSQL vs MongoDB）

**ADR の対象としない判断:**
- この Issue 内で完結する実装の詳細（関数分割、ファイル構成）
- 設定値の決定（デフォルト値等）
- 既存 ADR で決定済みの技術の利用方法

**既存 ADR の確認:**
提案前に `docs/adr/` を Glob で検索し、同じトピックの ADR がないか確認する。既存 ADR が `withdrawn` で supersede が必要な場合はその旨を伝える。

**ADR が必要な場合:**

Phase 3 を一時停止し、ユーザーに ADR 作成を依頼する:

```
この Issue の方針検討で、ADR として記録すべき技術判断が見つかりました。

### ADR 作成が必要な判断
{判断のトピック}

### /adr に渡す情報
- **スコープ**: {この ADR が対象とする範囲}
- **前提条件**:
  - {既存 ADR-NNNN: ...}
  - {技術スタックの制約}
- **制約条件**: {Issue から導出された制約}
- **検討候補**（技術調査で発見済み）:
  1. {選択肢A} — {概要}
  2. {選択肢B} — {概要}

以下のコマンドで ADR を作成してください:
`/adr {トピック}`

ADR が完了したら教えてください。ADR の決定を踏まえて方針検討を再開します。
```

ユーザーが ADR 完了を報告したら、作成された ADR を Read で読み込み、その決定を方針検討に反映して続行する。

**ADR が不要な場合:**
そのまま方針検討に進む。

### 3. 方針の検討

技術調査で挙がったアプローチと影響調査の結果を突き合わせ、実現方針を検討する。ADR が作成された場合は、その決定を前提として方針を検討する。

**方針が1つに絞られる場合:**
- その方針の概要、根拠、リスクを整理する

**方針が複数ある場合:**
- 比較フォーマット（ステップ 5）に沿って整理する

この時点ではまだユーザーに提示しない。次の方針レビューを経てから提示する。

### 4. 方針レビュー（Codex）

方針をユーザーに提示する前に、Codex で方針レベルの問題を検出する。計画を書く LLM（Opus）とは別のモデルで検証し、上流の誤りを早期に発見する。

以下のプロンプトを構築し、`timeout 600 codex exec --full-auto` に stdin で渡す。codex CLI が利用できない場合は Opus サブエージェントにフォールバックする。

```
以下の方針検討結果をレビューしてください。

方針:
{方針の内容（1つの場合は概要・根拠・リスク、複数の場合は比較表を含む）}

Issue 本文:
{Issue 本文}

技術調査結果:
{technical-research.md の内容}

## レビュー観点

### A. 受け入れ条件カバレッジ（方針レベル）
- この方針で Issue の全受け入れ条件を実現可能か
- 方針の選択によって実現不可能になる受け入れ条件はないか
- 方針が暗黙的に前提としている条件は妥当か

### B. ADR 整合性
- 方針が docs/adr/ の既存 ADR に矛盾しないか
- ADR で決定済みの技術・パターンと整合するか
- 必要に応じて docs/adr/ の内容を読んで確認すること

### C. 代替案の脱落
- 技術調査で挙がった選択肢が方針検討で不当に排除されていないか
- 排除された選択肢の棄却理由は妥当か
- 技術調査で言及されていない有力な代替案がないか

## 出力
結果を .output/issue-plan/{issue_number}/review-approach.md に書き出してください。

### フォーマット
観点ごとにセクションを分け、各 finding について以下を含めてください:
- finding_id（通し番号）
- perspective（A/B/C のいずれか）
- severity（high / medium / low）
- issue（問題の要約）
- reason（なぜ問題か）
- suggestion（修正案）

指摘がない場合は「指摘なし」と記載してください。
```

プロンプト中の `{issue_number}` は実際の Issue 番号に置換すること。他のプレースホルダもプロンプト構築時に展開して埋め込む。

**レビュー結果の反映:**

1. `.output/issue-plan/{issue_number}/review-approach.md` を読み込む
2. high/medium の指摘があれば方針を修正する
3. 修正内容をユーザーへの提示に含める（「Codex レビューで以下の指摘があり反映しました」）

### 5. ユーザーへの提示

方針レビューの結果を反映した上で、ユーザーに提示する。Codex レビューで high/medium の指摘を反映した場合は「Codex レビューで以下の指摘があり反映しました」と明記する。

**方針が1つの場合:**
- 方針の概要、根拠、リスクを提示する
- 「問題なければOKと入力してください。指摘があればお願いします。」と表示

**方針が複数の場合:**
- 以下の比較フォーマットで提示する
- 「どの方針にしますか？他の案や指摘があればお願いします。」と表示

#### 比較フォーマット（複数方針の場合）

```markdown
## 方針比較

### 判断軸

調査結果から導出した、この決定に固有の評価軸を 3-4 個設定する。
汎用的すぎる軸（「品質」「コスト」）は避け、具体的な軸にする。

### 比較表

| 判断軸 | 方針A: {名前} | 方針B: {名前} |
|--------|-------------|-------------|
| {軸1} | {評価} | {評価} |
| {軸2} | {評価} | {評価} |
| {軸3} | {評価} | {評価} |

### 各方針の詳細

#### 方針A: {名前}
- **概要**: {何をどうするか}
- **メリット**: {箇条書き}
- **デメリット**: {箇条書き}
- **影響範囲**: {影響調査との関連}
- **実装工数の目安**: 大/中/小

#### 方針B: {名前}
{同上}

### 判断ガイド

条件付きの帰結を示す。「推奨」はしない。

- {軸1}を優先するなら → 方針A。ただし{リスク}
- {軸2}を優先するなら → 方針B。ただし{リスク}
```

### 6. ユーザーの判断

```
どの方針を採用しますか？
1. 選択した方針
2. 選択の理由（どの判断軸を重視したか）
```

### 7. Issue 本文の「方針」セクション更新

方針決定後、`gh issue edit` で Issue 本文の「## 方針」セクションを更新する。Issue 本文だけ見ればゴールが分かる状態を維持するため、このステップは必須。

### 8. 設計判断の抽出

方針検討の過程で、計画のステップ構成・外部契約・制約に影響を与える判断を行った場合、テーブル形式で整理する。このテーブルは Issue コメントに含め、`/issue-implement` が I/F 設計時に参照する。

**記録する判断（計画の構成に影響を与えたもの）:**
- データの持ち方（例: customer_id ベース vs session_id ベース）
- API の粒度（例: 1 RPC で完結 vs 複数 RPC に分割）
- 処理の分担（例: バリデーションをどのレイヤーで行うか）
- エラー時の振る舞い方針（例: 即座にエラー返却 vs フォールバック）
- スコープ境界の判断（例: この Issue で在庫チェックまでやるか）

**記録しない判断（計画にないコードレベルの詳細。`/issue-implement` Phase 1 の管轄）:**
- 構造体のフィールド名・メソッド名
- エラーメッセージの文言
- 関数の分割粒度
- 依存注入の方式

該当する判断がなければテーブルは省略する。

### 9. Issue コメント投稿

決定内容を Issue コメントとして投稿する（比較の詳細を残すため）。

**方針が1つの場合:**

```markdown
<!-- issue-plan:approach:done -->
## 方針検討

### 採用方針: {名前}

{方針の概要}

### 根拠

{なぜこの方針になるかの説明}

### リスク

{認識しているリスク}

### 設計判断

| # | 判断 | 選択 | 理由 | 棄却した代替案 |
|---|------|------|------|--------------|
| 1 | {判断内容} | {選択した方} | {なぜその選択か} | {あれば記載。なければ空欄} |

（該当する判断がなければこのセクションは省略）
```

**複数方針から選択した場合:**

```markdown
<!-- issue-plan:approach:done -->
## 方針検討

### 検討した方針

{比較表を含む}

### 採用方針: {名前}

### 選択理由

{ユーザーの判断理由}

### 受け入れるトレードオフ

{選択に伴うリスク・犠牲}

### 設計判断

| # | 判断 | 選択 | 理由 | 棄却した代替案 |
|---|------|------|------|--------------|
| 1 | {判断内容} | {選択した方} | {なぜその選択か} | {あれば記載。なければ空欄} |

（該当する判断がなければこのセクションは省略）
```
