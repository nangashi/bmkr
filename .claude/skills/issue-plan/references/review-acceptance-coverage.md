# Review: 受け入れ条件カバレッジ

## Overview
計画ステップが受け入れ条件を全て網羅しているか、変更ファイル一覧に漏れがないかをレビューする。

## Step 1: Analysis（計画の構造を把握する）

### マッピング対象
- Issue 本文の「受け入れ条件」セクションの全項目
- 計画の「実装ステップ」セクションの各ステップのゴール・外部契約
- 計画の「受け入れ条件との対応」表
- 計画の「変更ファイル一覧」

### 各項目について記録すること
1. 受け入れ条件ごとに、対応する計画ステップが存在するか
2. 「受け入れ条件との対応」表に記載された対応関係が妥当か（ステップのゴールが条件を実現するか）
3. 各ステップで変更が必要なファイルが「変更ファイル一覧」に含まれているか
4. 計画の「リスクと対策」が、識別済みのギャップをカバーしているか

## Step 2: Findings（分析結果から指摘を導出する）

### 指摘とする基準
- 受け入れ条件に対応するステップが存在しない（カバレッジの欠落）
- 「受け入れ条件との対応」表の対応関係が不正確（ステップのゴールが条件を実現しない）
- ステップで変更が必要なファイル/ディレクトリが「変更ファイル一覧」にない
- 受け入れ条件の一部が暗黙的にしかカバーされていない（明示的なステップがなく、他ステップの副作用として実現される想定）

### 指摘としない基準
- テスト作成・動作確認に関するステップの不在（issue-implement の Phase 3・Phase 4 の責務）
- 自動生成ファイル（protobuf, sqlc, templ 等）の変更ファイル一覧への不記載（計画フォーマットのガイドラインで「記載不要」と規定）
- 実装の詳細（構造体設計、処理フロー等）の不在（issue-implement の Phase 1・Phase 4 の責務）
