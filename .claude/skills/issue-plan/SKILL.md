# issue-plan

GitHub Issue をハブとして、要件整理から実装計画までを段階的に進める。各フェーズの成果は Issue コメントとして投稿し、フェーズ間のデータ受け渡しと永続化を兼ねる。

**重要: ユーザーとの対話および Issue への書き込みは日本語で行うこと。**

### ユーザー対話のルール

- **AskUserQuestion は「固定選択肢から1つ選ぶだけ」の場面でのみ使用する**（例: A/B/C のどれにしますか？）
- それ以外の対話（自由入力、承認+フィードバック、確認）は **テキスト出力のみ** で行い、ユーザーの入力を待つ
- 「問題なければOKと入力してください。指摘があればお願いします。」のように、承認とフィードバックの両方を受け付ける形が望ましい
- ワークフローの説明や選択肢の羅列は最小限にし、簡潔に聞く

## ワークフロー

```
Phase 0: 初期化     → Issue読み込み・既存コメントスキャン・再開判定
Phase 1: ヒアリング  → 要件の曖昧さ解消・Issue本文の構造化
Phase 2: 調査       → 影響調査 + 技術調査（並列サブエージェント）
Phase 3: 方針検討    → 実現方針の比較・人間判断
Phase 4: 計画作成    → plan作成・事前検証・Codex レビュー・条件付きレビュー
Phase 5: 人間レビュー → 計画の最終確認
```

---

## Phase 0: 初期化

### 0a. 前提チェック

`gh auth status` で GitHub CLI が利用可能か確認する。失敗時は案内して終了。

### 0b. 引数の解析

| パターン | 動作 |
|---------|------|
| `#123` または `123` | Issue を読み込む |
| 自由テキスト | Phase 1 でヒアリング後に Issue 作成 |
| 引数なし | `Issue番号あるいは実装したいことを教えてください。` とだけ表示して入力を待つ |

### 0c. 既存コメントのスキャン（Issue 番号指定時）

Issue のコメントから `<!-- issue-plan:{phase}:{status} -->` マーカーを検索し、完了済み・中断中フェーズを検出する。完了済みフェーズがある場合、進捗を提示して再開ポイントを確認する。

### 0d. 作業ディレクトリ

`.output/issue-plan/{issue_number}/` をサブエージェントの出力先として使用する。

---

## Phase 1: ヒアリング

Issue 本文（または自由テキスト）から要件を整理し、Issue 本文を構造化する。

### 手順

1. 入力を読み、Issue 本文テンプレート（`.github/ISSUE_TEMPLATE/feature.md`）に沿って整理する
2. 不明点質問 + 推測箇所の明示:
   - Issue 本文と入力を照合し、**判断が分岐するポイント**を抽出する
   - 抽出基準: Issue に根拠がなく、2つ以上の妥当な選択肢があるもの
   - 実装時に容易に変更でき受け入れ条件に影響しない項目は含めない
   - 分岐点が 0 件ならそのままステップ 3 へ
   - 分岐点がある場合、リストとして提示し、分岐点の性質に応じた補足を添える:
     - 画面系の分岐点 → テキストで画面構成を添える（表示項目・操作・件数等）
     - API/ロジック系の分岐点 → 入出力の具体例を添える
   - ユーザーの回答を Issue 本文の該当セクション（受け入れ条件・補足等）に反映する
3. 構造化した内容を提示し、「問題なければOKと入力してください。修正があればお願いします。」と表示して入力を待つ（承認なしに `gh issue create` や `gh issue edit` を実行しない）
4. 承認後、Issue 本文を更新する（既存: `gh issue edit`、新規: `gh issue create`）

### Issue 本文フォーマット

`.github/ISSUE_TEMPLATE/feature.md` のテンプレートに従う:

```markdown
## 概要
<!-- 背景や解決したい課題を伝えつつ、やることを簡潔に記載 -->

## 方針
<!-- どうやるか簡潔に記載。未定の場合は「要検討」と記載 -->

## 受け入れ条件
- [ ] {条件}

## スコープ外
<!-- やらないことを明示。なければセクションごと削除 -->

## 補足
<!-- 制約・前提条件・関連情報など。なければセクションごと削除 -->
```

**各セクションの役割（ゴールのブレ防止）:**
- **概要**: なぜやるか（背景・課題）+ 何をやるか → 的外れな実装を防ぐ
- **方針**: どうやるか → Phase 3 で決定後に更新。初期は「要検討」で可
- **受け入れ条件**: いつ完了か → issue-implement の完了判断基準
- **スコープ外**: 何をやらないか → LLM の過剰な親切を抑制
- **補足**: 制約・前提・関連 ADR 等 → 判断に必要な背景情報

スコープ外と補足は該当がなければ省略する。

### スキップ条件

Issue 本文が既にテンプレートに沿って構造化されており、曖昧さがない場合。

---

## Phase 2: 調査

影響調査と技術調査を並列サブエージェントで実行する。

`references/investigation.md` を Read で読み込み、その内容に従って 2 つの Sonnet モデルのサブエージェント（`model: sonnet`）を**並列起動**する。コードベース探索・リサーチタスクのため Sonnet で十分な精度が得られる。

各サブエージェントに渡す情報:
- Issue 本文（構造化済み）
- プロジェクトの作業ディレクトリパス
- 出力先パス（`.output/issue-plan/{issue_number}/` 配下）
- `CLAUDE.md` の内容（ADR 参照の指示を含む）

サブエージェント完了後、**即座に** Issue コメントとして投稿する。中断に備え、各サブエージェントの結果を受け取り次第、個別にコメント投稿する。

### スキップ条件

新規機能で既存コードへの影響がなく、技術的に自明な場合。スキップ時は理由をユーザーに伝える。

---

## Phase 3: 方針検討

調査結果を踏まえ実現方針を検討する。

`references/approach.md` を Read で読み込み、その内容に従う。

1. **ADR 作成要否の判定**: 技術的に 2 つ以上の選択肢があり、プロジェクト全体に影響する判断がある場合、「ADR を作成すべき判断があります。`/adr` で作成してから再開してください。」と表示して一時停止する
2. **方針が1つ**: 提示して「問題なければOKと入力してください。指摘があればお願いします。」と表示
3. **方針が複数**: 比較表 + 判断ガイドを提示し、「どの方針にしますか？他の案や指摘があればお願いします。」と表示

決定した方針を Issue コメントとして投稿する。

**方針決定後、Issue 本文の「## 方針」セクションも更新する**（`gh issue edit` で本文を書き換え）。Issue 本文だけ見ればゴールが分かる状態を維持する。

### スキップ条件

実現方法が自明で1つしかない場合、Phase 4 に直接組み込める。

---

## Phase 4: 計画作成 + レビュー

実装計画を作成し、事前検証 + Codex レビュー + 条件付きレビューの3段階でレビューする。

### 4a. 計画作成

`references/plan.md` を Read で読み込み、その内容に従う。

1. Issue 本文 + 調査結果 + 方針を入力に、実装計画を作成
2. `.output/issue-plan/{issue_number}/plan.md` に書き出す

### 4b. 計画レビュー

`references/review.md` を Read で読み込み、その内容に従う。

**事前検証（メインエージェント）— Codex レビューの前に実施:**
- 数値・条件の突き合わせ — 計画の値と受け入れ条件の値が一致するか
- 命名パターンの突き合わせ — 新規名が既存コードの命名パターンに沿うか
- スコープの突き合わせ — スコープ外の作業が紛れていないか
- ファイル存在・ADR 参照の検証 — 参照するファイル・ADR が存在するか
- ディレクトリ構成の突き合わせ — 新規ファイルの配置が既存サービスの構成に沿うか

**Codex レビュー（常時）— 計画を書いた LLM とは別のモデルでバイアスを検出:**
- 受け入れ条件カバレッジ — 計画ステップが受け入れ条件を全て網羅しているか
- シンプルさ — 過剰設計・過剰外部化・不要な新規依存の検出

**条件付きレビュー（Sonnet サブエージェント `model: sonnet`、該当時のみ）:**
- セキュリティ設計 — 認証・トークン・入力処理を含む場合
- データアクセス設計 — DB クエリ・データ処理を含む場合

### 4c. 修正・投稿

3. 事前検証 + レビュー指摘があれば計画を修正
4. 計画を Issue コメントとして投稿
5. レビューサマリを別コメントで投稿

---

## Phase 5: 人間レビュー

計画を投稿後、以下のテキストを表示して入力を待つ:

```
計画を Issue #{番号} にコメントしました。
問題なければOKと入力してください。修正があればお願いします。
実装は `/issue-implement #{番号}` で開始できます。
```

フィードバックがあれば計画を修正し、Issue コメントを編集する。

## Issue コメントフォーマット

### マーカー

```html
<!-- issue-plan:{phase}:{status} -->
```

- `{phase}`: `impact-investigation`, `technical-research`, `approach`, `plan`, `review`
- `{status}`: `done`, `in-progress`

### コメント投稿の流れ

1. フェーズ開始時に `in-progress` マーカー付きコメントを投稿
2. フェーズ完了時にコメントを編集し `done` マーカーに更新、本文を差し替え

### コメントの編集

既存コメントを更新する場合は、`gh issue view` の `--json comments` で `databaseId` を取得し、`gh api` の PATCH で編集する。

## Gotchas

このスキルを使う中で判明した注意点。運用しながら追記していく。

- **方針が1つに収束しても比較を省略しない**: 技術調査で複数アプローチが出たのに、方針検討で「自明」として比較をスキップしがち。技術調査に2つ以上のアプローチがある場合は、方針検討でも比較表を出すこと
- **コメント編集は databaseId を使う**: `gh issue view --json comments` で返る `id` は GraphQL の node_id。REST API での編集には `.databaseId` フィールドが必要
- **サブエージェントは必ずファイルに書き出す**: サブエージェントが結果をテキストで返すだけでファイル書き出しを忘れることがある。プロンプトに出力先パスを明記し「Write で書き出すこと」を指示する
- **gh コマンドの本文は必ず `--body-file` を使う**: `gh issue comment --body` や `gh issue edit --body` にインラインで本文を渡すと、zsh が `!`（ヒストリ展開文字）を `\!` にエスケープし、HTMLコメント `<!-- -->` が `<\!-- -->` になって表示されてしまう。`gh issue comment`、`gh issue edit`、`gh issue create` すべてで一時ファイルに書き出して `--body-file` を使うこと
- **`gh issue create` のテンプレート指定**: `--template` フラグでテンプレートを指定しても本文が空になることがある。`--body-file` で直接渡す方が確実
- **レビューは事前検証を先にやる**: Codex レビュー・条件付きレビューだけだと、大きな問題に注意が集中し、命名規則の逸脱や数値の矛盾といった細部を見落としやすい。`references/review.md` の「事前検証」を必ず先に実行し、機械的な突き合わせで細部の問題を拾ってから Codex レビュー・条件付きレビューに進む
- **推測箇所の明示は判断分岐点に限定する**: 「2つ以上の妥当な選択肢があり、Issue に根拠がない」ものだけ提示する。自明な推測や、実装時に容易に変更できる項目まで列挙しない
- **仕様明確化で実現方法（How）に踏み込まない**: 表示項目・操作・ビジネスルール（What）の確認に留め、技術的な実装方式は Phase 3 に委ねる
- **画面構成は表示項目と操作のリストに留める**: レイアウトやスタイルには言及しない。見た目の詳細は実装時に templ で確定する

## フェーズスキップの判断基準

| フェーズ | スキップ可能 | スキップ不可 |
|---------|------------|------------|
| ヒアリング | Issue 本文がテンプレートに沿って構造化済み | 概要・受入条件が不明確 |
| 影響調査 | 新規機能で既存コードに影響なし | 既存機能の変更・リファクタ |
| 技術調査 | 使用技術が自明（ADR 決定済み等） | 新技術導入・複数選択肢あり |
| 方針検討 | 実現方法が1つのみ | 複数アプローチが考えられる |

**スキップ時は必ずユーザーにその旨と理由を伝える。**
