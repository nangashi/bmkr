# 振り返り — サブエージェント詳細手順

プロセスの過剰・不足を検出し、失敗・成功から得た知見を収集・分析して改善提案を作成する。

---

## 前提情報の自己取得

以下のファイルを `.output/issue-implement3/{issue_number}/` から Read する（存在しないものはスキップ）:

| ファイル | 内容 | 作成レベル |
|---------|------|----------|
| `scope.md` | レベル分類・根拠・実行パス | 全レベル |
| `edit-log.md` | 直接編集の変更ファイル一覧 | L1 |
| `interface-design-decisions.md` | I/F設計の計画からの判断差分 | L2 |
| `contract-decisions.md` | 契約設計の計画からの判断差分 | L3 |
| `contract.md` | 公開契約・不変条件 | L3 |
| `test-strategy.md` | テスト戦略の分類結果 | L2(簡易)/L3 |
| `failure-clusters.md` | failure cluster の分類結果 | L3 |
| `escalation.md` | Escalation の記録 | L3 |
| `red-summary.md` | Red 生成の結果サマリー | L3 |
| `review-design.md` / `review-design-filtered.md` | 設計レビュー結果と採用判定 | L2/L3 |
| `review-test.md` / `review-test-filtered.md` | テストレビュー結果と採用判定 | L3 |

| `goal-alignment.md` | ゴール整合性チェック結果 | 全レベル |

## 知見の収集と原因分析

上記の情報源から、失敗・指摘と成功パターンを収集し分析する。

**失敗・指摘の分析:**

| # | 失敗・指摘の内容 | 発生フェーズ | 発生原因 | 解消方法 |
|---|-----------------|-------------|---------|---------|
| 1 | （例）sqlc 生成型の Nullable フィールドに直接値を代入してコンパイルエラー | 実装ループ attempt 1 | pgtype の扱いに関するガイド未参照 | pgtype-nullable.md に従い Valid フィールドを設定 |

- **発生原因**: なぜこの問題が起きたか（例: ガイドの未整備、スキルの指示不足、LLM の知識不足、仕様の曖昧さ）
- **解消方法**: 実際にどう解消したか、または解消できなかったか

**成功パターン（初出のみ）:**

既存のガイド・ルールに記載がなく、今回初めてうまくいったアプローチのみ記録する。既存ガイド通りの成功は記録しない。

| # | 内容 | フェーズ | なぜうまくいったか |
|---|------|---------|-----------------|
| 1 | （例）errgroup で並行クエリの I/F を先に設計し、attempt 1 で Green | I/F設計→実装 | I/F 定義時に並行処理の境界を明示した |

**中断ケースの追加分析（実装フェーズ中断時のみ）:**

実装フェーズが中断に至った場合、上記に加えて以下を分析する:

- 実装ループ上限（L2: 3回、L3: 10回）で解決できなかった根本原因は何か
- 計画の前提・粒度に問題はなかったか（あれば issue-plan へのフィードバックとして改善提案に記録）

**レベル固有の分析:**

scope.md の変更レベルに応じて以下を追加分析する:

- **L1**: 変更が本当に L1 で適切だったか（L2 にすべきだったケースの検出）
- **L2**: 設計レビューの空振り率、L3 昇格が発生したか（発生した場合、分類基準の改善提案）
- **L3**: 契約レビュー・テストレビューの空振り率、Escalation 発火回数と原因、実装ループ上限 10 回の到達有無、failure cluster の分類精度

失敗・指摘も成功パターンも1件もなかった場合は改善提案をスキップし、その旨を返す。

## 改善提案

上記の知見から、改善効果が高いもの（再発可能性が高い・他の Issue でも起きうる）を選定し、以下のいずれかを提案する:

| 改善手段 | 対象 | 条件 |
|---------|------|------|
| `docs/guides/` にガイド追加 | LLM が繰り返し踏む実装上の落とし穴 | Linter で検出できないもの。既存のガイドファイルに追記するか、新規ファイルを `docs/guides/` に作成 |
| `.claude/rules/` にルール追加 | 特定ファイルパターン編集時に常に適用すべき制約 | glob で対象ファイルを指定する |
| スキル改善の提案 | スキルの指示・フロー・スキップ条件に起因する問題 | 実装プロセスの改善に繋がる場合 |
| 計画品質フィードバック | issue-plan の観点（受け入れ条件の明確さ、技術的前提） | ESCALATE または中断が計画起因の場合 |
| レベル分類基準の改善 | Phase 0 の分類ロジックに起因する問題 | L2→L3 昇格が発生した場合、分類基準に追加すべき条件 |

各提案に **期待効果**（何が防げるか）と **コスト**（何を変えるか）を1行で添える。

---

## 出力

分析結果と改善提案をメインエージェントに返す。メインエージェントがユーザーに提示して承認を取る。承認された改善はメインエージェントが実行するか、改善 Issue を作成する。
