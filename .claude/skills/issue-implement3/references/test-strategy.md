# テスト戦略

変更対象ごとに、UT / 統合 / テスト不要を決める。テストの有無は `wip` コメントの有無ではなく、ROI と変更の性質で判定する。

---

## 必須参照

- `docs/guides/testing-strategy.md`
- 変更対象周辺の既存テスト

## 手順

1. 変更対象を関数・モジュール・責務単位で列挙する
2. 各対象について以下を記録する:
   - 複雑さ / ドメイン重要度
   - 外部依存の数
   - 既存テストで回帰検知できるか
3. `UT必須 / 統合テストで十分 / 新規テスト不要` のいずれかに分類する
4. `.output/issue-implement3/{issue_number}/test-strategy.md` に記録する

## 判定基準

### UT必須

- ドメインロジックやアルゴリズムが増える
- バリデーションや境界条件が重要
- バグが既存の compile/lint/統合テストでは見えにくい

### 統合テストで十分

- ハンドラやコントローラが主にオーケストレーションを行う
- 協力者が多く、局所的ユニットテストのモックコストが高い

### 新規テスト不要

- 機械的変更
- トリビアルな委譲
- 既存テストと静的検証で十分

## 出力フォーマット

```markdown
# Test Strategy

| 対象 | 分類 | 理由 | 予定する検証 |
|------|------|------|-------------|
```
