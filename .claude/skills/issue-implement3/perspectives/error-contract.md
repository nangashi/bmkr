# Review: エラー契約

## Overview
動作コメントのエラーパス網羅性、エラー種別の明確さをレビューする。

## Step 1: Analysis（対象コードの構造を把握する）

### マッピング対象
- 新規・変更された関数の動作コメント（`// 動作:` や doc comment で記述されたもの）
- error を返す関数シグネチャ
- エラー型・エラー変数の定義

### 各箇所について記録すること
1. 動作コメントに記述されたエラーパス（どの条件でどのエラーを返すか）
2. 関数シグネチャの戻り値にある error の種類（sentinel error、カスタム型、wrapped error）
3. 正常系と異常系の分岐が動作コメントに明記されているか
4. 呼び出し先が返しうるエラーの伝播方針（wrap して返す、変換する、握りつぶす）

## Step 2: Findings（分析結果から指摘を導出する）

### 指摘とする基準
- 動作コメントに正常系のみ記述があり、エラーパスの記述がない
- error を返す関数だが、どの条件で何のエラーを返すかが不明確
- 複数の異なるエラー原因が同一のエラーメッセージ / 型で返される（呼び出し元が区別できない）
- 動作コメントに記述されたエラー条件と、シグネチャの戻り値型が矛盾している（例: エラーケースが書かれているのに error を返さない）

### 指摘としない基準
- 自明なエラー（io.EOF, context.Canceled 等の標準ライブラリエラー）の個別列挙
- 動作コメントがまだ書かれていない既存コード（今回の変更対象外）
- I/F のみ定義している段階で、実装内部のエラーハンドリング詳細
