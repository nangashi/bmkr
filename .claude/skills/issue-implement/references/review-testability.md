# Review: テスト駆動性

## Overview
動作コメントからテストケース導出が可能か、エッジケース・異常系の具体性をレビューする。

## Step 1: Analysis（対象コードの構造を把握する）

### マッピング対象
- 新規・変更された関数の動作コメント
- 関数シグネチャ（引数・戻り値から入出力の境界を把握）
- interface 定義（テスト時のモック対象を把握）

### 各箇所について記録すること
1. 動作コメントからテストケース名（TestXxx_正常系、TestXxx_空リスト等）を導出できるか
2. 正常系・異常系・エッジケースの入力条件が具体的に記述されているか（「不正な入力」ではなく「空文字列」「負の値」等）
3. 期待する出力・副作用が検証可能な形で記述されているか（「適切に処理する」ではなく「ErrNotFound を返す」等）
4. 外部依存（DB、外部API等）が interface 経由で注入可能か

## Step 2: Findings（分析結果から指摘を導出する）

### 指摘とする基準
- 動作コメントが曖昧で、テストの期待値を導出できない（例: 「適切に処理する」「エラーをハンドルする」）
- エッジケースの入力条件が列挙されていない（例: 空リスト、ゼロ値、nil、境界値）
- 外部依存が具象型に直接依存しており、テスト時にモック差し替えができない
- 動作コメントの記述と関数シグネチャが矛盾している（コメントに書かれた動作が型的に実現不可能）

### 指摘としない基準
- テストコード自体の書き方やテストフレームワークの選択
- 全てのエッジケースを網羅的に列挙することの要求（主要なエッジケースが記述されていれば十分）
- protobuf / sqlc 等の自動生成コードのテスト駆動性

### perspective ラベル
テスト駆動性
