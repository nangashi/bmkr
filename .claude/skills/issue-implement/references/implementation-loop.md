# 実装ループ手順（Phase 4）

テストを全て通す（Green）ことをゴールに、Codex CLI で実装し、オーケストレータ（Opus）が進捗管理とリトライ制御を行う。

---

## 概要

```
best_commit（Phase 3 完了時点）から開始
  → Codex で実装
  → テスト結果でスコア判定
  → 改善ならコミット、停滞/悪化ならロールバック+方針変更して再試行
  → 最大3回で解決しなければ ESCALATE（テスト/I/F修正を試行）
  → 全テスト通過後、最終ゲート（Codex Review + ゴール整合性）
```

---

## Phase 4 開始前のコミット

Phase 3 完了後、Phase 4 開始前に成果物をコミットする:

```bash
git add -A && git commit -m "checkpoint: phase3 complete (issue #{number})"
```

このコミットが best_commit の初期値（safe point）となる。

---

## Codex に渡すプロンプト

以下の情報を含むプロンプトを構築し、`timeout 300 codex exec --full-auto` に stdin で渡す:

- **実装対象の差分**: `git diff main` の出力（I/F定義・動作コメントが含まれる。テストファイルも含めて全体像を把握させる）
- 未実装箇所（`panic("not implemented")` 等）を実装に置き換えるタスク説明
- 計画の方針・スコープ外
- `docs/guides/` 配下のガイドに従うこと
- テスト実行コマンド（`just test`）と期待される状態（全テスト Green）
- 実装後に `just fmt` と `just lint` を実行して問題があれば修正すること
- 公開 API（ハンドラのルーティング、レスポンス構造等）は変更しないこと。非公開 I/F（内部の依存注入方法、責務分割等）の修正が必要と判断した場合は、理由と変更範囲を出力して停止し、指示を仰ぐこと
- **停止条件**: テストに複数回失敗した、またはテスト自体の誤りが疑われる場合は、失敗理由・試したアプローチ・テスト誤りを疑う根拠を自由記述で出力して停止し、指示を仰ぐこと

---

## メインループ（オーケストレータが制御）

```
best_commit = HEAD  # Phase 3 完了コミット
best_score = 0
failure_log = []

for attempt in 1..3:
  # 1. best_commit にリセット
  git checkout -- . && git clean -fd

  # 2. プロンプト組み立て
  prompt = 基本指示 + failure_log（attempt >= 2 の場合）

  # 3. Codex 実行
  timeout 300 codex exec --full-auto < prompt

  # 4. 静的チェック（Codex が見逃した場合のセーフティネット）
  just fmt
  just lint
  → lint 失敗の場合もテスト実行に進む（テスト結果でスコア判定するため）

  # 5. テスト実行・スコア算出
  just test → pass_count, total_count を取得
  score = pass_count

  # 6. 判定
  score == total_count → 全パス → 最終ゲートへ ✅
  score > best_score  → 改善:
    git add -A && git commit -m "wip: phase4-attempt-{attempt}"
    best_commit = HEAD
    best_score = score
  score <= best_score → 停滞/悪化:
    （次ループ冒頭で best_commit にリセットされる）

  # 7. failure_log に記録
  failure_log に追記:
    - 失敗テスト名とエラー内容（テスト出力から）
    - 変更ファイル一覧（git diff --name-only から）
    - Codex の自由記述出力（あれば）
```

---

## failure_log の構成

オーケストレータが各 attempt の結果から構成する。次ラウンドのプロンプトに注入して同じ失敗の繰り返しを防ぐ。

```
## 過去の失敗（同じアプローチを避けること）
- Attempt 1: DB接続の初期化をinit()で行ったがテスト時にモック差し込み不可。
  失敗テスト: TestHandleProductList_DBError
  → init()ではなくコンストラクタインジェクションを使うこと。
- Attempt 2: ...
```

情報源の優先順位:
1. Codex の自由記述出力（試したアプローチと失敗理由が直接得られる）
2. テスト出力（どのテストがどう失敗したか）
3. git diff --name-only（何を変更したか）

---

## ESCALATE（3ラウンド到達時）

3ラウンドで全テスト通過に至らなかった場合、オーケストレータ（Opus）が failure_log を分析し、テスト/I/F 側の問題を検出・修正する。

### 分析

failure_log 全体を見て以下の観点でチェックする:

- **同一テストが全試行で失敗** → テスト自体に問題がある可能性（test_bug）
- **型エラー・インポートエラーが繰り返される** → I/F定義に問題がある可能性（if_gap）
- **実装方針を変えても同一箇所でエラー** → テストの期待値や前提条件が間違っている可能性（test_bug）
- **外部ツール（templ等）の生成物がテストの期待値と一致しない** → 外部ツールの実挙動との差（tool_behavior_gap）
- **実装は動くが責務が密結合・不自然なコードになっている** → I/F の責務分割が粗い可能性（plan_gap）

### 対応

| 判定 | 対応 |
|------|------|
| テスト/I/F に問題あり | 修正 → コミット → 最終1回 codex exec → テスト実行 → 全パスなら最終ゲートへ |
| 実装の複雑さが想定超過 | best_commit の状態で停止、ユーザーに報告 |

### テスト修正の制約

修正が許されるもの:
- フィクスチャの誤り（型の初期化ミス、無効なテストデータ等）
- 動作コメントの解釈ミスに基づくテストの期待値
- テストのセットアップ不備（モック設定漏れ、コンテキスト不足等）
- 外部ツール（templ等）の実挙動に合わせた検証粒度の調整（例: exact string 比較を構造的な検証に緩和）

修正が許されないもの:
- テストの期待値を実装に合わせて変更すること（テストを実装に過適合させない）
- テストカバレッジを下げる変更

### I/F 変更の許可条件

公開 API と非公開 I/F で許可レベルが異なる。

**公開 API**（ハンドラのルーティング、レスポンス構造、gRPC サービス定義等）:
- 原則変更しない
- 以下のいずれかに該当する場合のみ変更を許可する:
  - I/F に不足がある（テストが原理的に通せない、型やメソッドが足りない）
  - 既存 ADR・既存コードパターンに反している
- 変更後は Phase 2 のプロジェクト適合性レビューを再実行する

**非公開 I/F**（内部の依存注入方法、構造体のフィールド構成、責務分割等）:
- 受け入れ条件が不変であれば修正を許可する
- 修正後は Phase 2 のプロジェクト適合性レビュー（軽量: プロジェクト固有レビューのみ）を実施する

### テスト/I/F 修正時の証拠要件

テストまたは I/F を修正する場合、以下を記録してからコミットする:

1. **failing evidence**: 失敗テストの出力、コンパイルエラー、または外部ツールの実出力
2. **invariant statement**: 何が不変で、何だけを変えるかの宣言（例: 「商品一覧が表示されることは不変。変更するのは依存の注入方法だけ」）
3. **minimal patch scope**: 変更対象のファイル・関数の範囲

これにより、修正が「仕様破壊」ではなく「仕様の明確化」であることを検証可能にする。

## 最終ゲート（全テスト通過後）

全テスト通過後、PR 作成前に Codex 観点付きレビューとゴール整合性レビューを実施する。

### レビュー・修正ループ（最大2回）

```
for round in 1..2:
  Codex 観点付きレビュー → 採用判定
  採用指摘なし → ループ終了
  Codex 修正（fix + test + fmt + lint）

ゴール整合性レビュー（修正完了後に1回）
simplify（冗長コード整理）
```

### Codex 観点付きレビュー

以下のプロンプトを構築し、`timeout 300 codex exec --full-auto` に stdin で渡す。codex CLI が利用できない場合はスキップ。

```
以下の git diff の変更内容をコードレビューしてください。

差分:
git diff main

## レビューの進め方
1. まず以下のレビュー観点ファイルを読んでください:
   - .claude/skills/issue-implement/references/review-silent-failure.md
   - .claude/skills/issue-implement/references/review-project-compliance.md
   - .claude/skills/issue-implement/references/review-security.md
2. 各観点の Step 1（Analysis）に従い、対象コードの構造を把握してください
3. 各観点の Step 2（Findings）に従い、指摘を導出してください
4. 必要に応じて、変更対象ファイルの周辺コード、docs/adr/、docs/guides/ も参照してください

## 出力
結果を .output/issue-implement/{issue_number}/review-final.md に書き出してください。

### フォーマット
観点ごとにセクションを分け、各 finding について以下を含めてください:
- finding_id（通し番号）
- perspective（各観点ファイルの「perspective ラベル」を使用）
- severity（high / medium / low）
- file:line
- issue（問題の要約）
- analysis（Step 1 で把握した事実）
- reason（なぜ問題か、対応しない場合の具体的シナリオ）
- suggestion（修正案、コード例付き）

各観点ファイルに「出力時の補足」がある場合はそれにも従ってください。
```

プロンプト中の `{issue_number}` は実際の Issue 番号に置換すること。`git diff main` の出力はプロンプト構築時に展開して埋め込む。

Claude が書いたコードを Claude 自身がレビューすると自分の判断を正当化するバイアスが働くため、別 LLM の目を入れて補完する。

### 採用判定（別サブエージェント）

Codex Review の結果を `agents/review-filter.md` を Read で読み込んだ採用判定サブエージェントに渡す。レビュアとは独立したコンテキストで、各指摘をコードの事実に基づいて判断する。

採用指摘がなければループ終了。

### 修正（Codex）

採用された指摘と `git diff main` を Codex に渡し、修正を実行させる。Codex には以下を指示する:

- 採用された指摘に対応してコードを修正すること
- 修正後に `just test`、`just fmt`、`just lint` を実行して問題がないことを確認すること

### ゴール整合性レビュー（修正完了後に1回）

全ての修正が完了した後に実施する。Issue のコンテキスト（受け入れ条件・スコープ外・計画）を持つサブエージェントで確認する。

**追加で渡す情報:**
- Issue 本文（受け入れ条件・スコープ外）
- 計画の実装ステップ

**確認すべき観点:**

受け入れ条件ごとにトレーサビリティ表を作成し、以下を埋める:

| 受け入れ条件 | 対応コード（ファイル:関数） | 対応テスト（ファイル:テスト名） | 状態 |
|-------------|--------------------------|------------------------------|------|
| 条件1 | handler.go:HandleX | handler_test.go:TestX | OK / 未対応 / 要確認 |

加えて以下も確認する:
- Issue の「スコープ外」に記載された内容に触れていないか
- 計画にない追加機能・リファクタリングが含まれていないか
- 受け入れ条件の独自解釈がないか

「未対応」または「要確認」がある場合はユーザーに報告する。

### simplify（ゴール整合性レビュー後に1回）

レビュー指摘対応で追加・修正されたコードの冗長性を整理する。`/simplify` を Skill ツールで実行する。

動作を変えずに、重複コード・不要な中間変数・過剰な抽象化などを検出・修正する。修正後に `just test`、`just fmt`、`just lint` を実行して問題がないことを確認する。

---

## 注意事項

- **Codex exec のネットワーク制限**: codex exec は sandbox 内で動作するため、`buf generate`（リモートプラグイン）等のネットワーク依存コマンドが失敗する場合がある。コード生成は Phase 1 で完了済みのため、Phase 4 では影響しない

## Codex CLI が利用できない場合

Phase 0 で codex CLI が検出できなかった場合は、Opus サブエージェントで実装を実行する。メインループの構造（リセット→実装→テスト→スコア判定→failure_log）は同様に適用する。

---

## 中断条件

以下のいずれかで中断し、PR は作成しない:
- ESCALATE で「実装の複雑さが想定超過」と判定
- ESCALATE 後の最終1回でもテストが通らない

**中断時の状態報告:**

```
Phase 4 が {理由} で中断しました。
現在の状態: {best_commit のハッシュ}
failure_log サマリ: {全試行の失敗パターン要約}
手動で実装を続ける場合はこのブランチで作業できます。
```

中断後も Phase 6（振り返り & main 復帰）に進むこと。failure_log を振り返りの入力として使用する。
