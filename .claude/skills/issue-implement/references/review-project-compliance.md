# Review: プロジェクト準拠

## Overview
ADR/ガイド準拠、既存パターンとの一貫性、命名規則をレビューする。

## Step 1: Analysis（対象コードの構造を把握する）

### マッピング対象
- 新規・変更されたファイル全て
- `docs/adr/` 配下の ADR（変更内容に関連するもの）
- `docs/guides/` 配下の全ガイド（変更内容との準拠チェック）
- 変更対象と同一パッケージ / ディレクトリの既存コード

### 各箇所について記録すること
1. 変更内容に関連する ADR の決定事項と、コードがそれに従っているか
2. 変更内容に関連するガイドのパターンと、コードがそれに従っているか
3. 同一パッケージ内の既存コードの命名パターン（関数名、型名、変数名、ファイル名）
4. 既存コードのファイル構成パターン（handler, types, service 等の分け方）
5. 既存コードのエラーハンドリングパターン

## Step 2: Findings（分析結果から指摘を導出する）

### 指摘とする基準
- ADR の決定事項に反するコード（例: ADR で選定した技術と異なるライブラリの使用）
- ガイドに記載されたパターンに違反するコード
- 同一パッケージ内の既存コードと命名規則が異なる（例: 既存が camelCase なのに snake_case）
- 同一サービス内の既存ファイル構成と異なる配置（例: 既存が handler.go に集約しているのに別ファイルに分割）
- 既存コードと異なるエラーハンドリングパターン（例: 既存が connect.NewError を使うのに errors.New を使う）

### 指摘としない基準
- ADR やガイドが存在しない領域の設計判断（前例がなければ新しいパターンも許容）
- 既存コードのパターンが明らかに非推奨・問題がある場合の改善
- テストコードの構成（テストは既存パターンとの一貫性より可読性を優先）
- 自動生成コード（protobuf, sqlc, templ 等）のパターン

### perspective ラベル
プロジェクト準拠

### 出力時の補足
- analysis には参照した ADR/ガイド/既存コードのパスを明記すること
- suggestion には準拠すべき ADR/ガイド/既存パターンを引用すること
