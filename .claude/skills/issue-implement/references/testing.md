# Phase 3: テスト作成（Red） — サブエージェント詳細手順

I/F定義と動作コメントをもとにユニットテストを作成する。サブエージェントで実行する。

---

## なぜこのタイミングでテストを書くか

Phase 1 で型・インターフェース・シグネチャが定義済みで、動作コメントに振る舞いが記述されている。実装本体はまだ存在しない。この状態でテストを書くことで:

- **テストはコンパイルが通る**: 型・インターフェースが存在するため、モック実装やテストコードがコンパイルエラーにならない
- **テストは実行すると失敗する（Red）**: 実装が `panic("not implemented")` や空なので、テストは失敗する。これが正しい状態
- **テストが実装に依存しない**: 実装が物理的に存在しないため、内部構造への依存が原理的に起きない
- **テストは動作コメントと受け入れ条件から導出される**: テストケースの源泉が明確

---

## プロンプトに含める情報

- Issue 本文（受け入れ条件が最重要）
- 計画の「変更ファイル一覧」と「受け入れ条件との対応」
- 作業ディレクトリパス
- CLAUDE.md の内容
- 「I/F定義の動作コメントと受け入れ条件からテストを導出すること」
- 「テストはコンパイルが通るが、実行すると失敗（Red）する状態が正しい」

---

## テスト導出の手順

### 1. 動作コメントを読む

Phase 1 で定義された関数・メソッドの動作コメントを読み、テスト可能な振る舞いを列挙する。

例: `HandleProductList` の動作コメントから:
- page パラメータのデフォルト値（1）
- 不正値のフォールバック（0以下、非数値 → 1）
- DB エラー時の 500 レスポンス
- HX-Request ヘッダによるレスポンス分岐
- ページ超過時のクランプと再フェッチ

### 2. 受け入れ条件と突き合わせる

動作コメントから導出したテストケースが、Issue の受け入れ条件を網羅しているか確認する。

### 3. モック実装を作る

Phase 1 で定義されたインターフェースに対するモック実装を作成する。モックは:
- 呼び出しパラメータを記録する
- 設定可能な戻り値・エラーを返す

### 4. テストを書く

正常系・異常系・境界値をカバーするテストを書く。既存テストパターンがあればそれに従う。

### 5. Red を確認する

テストを実行し、コンパイルが通ること + テストが失敗すること（Red）を確認する。コンパイルエラーが出た場合は I/F定義に問題があるので修正する。

---

## フロントエンドのテスト

React コンポーネントのテストは Go ほど TDD が自然ではなく、テスト基盤の有無にも依存する。プロジェクトにフロントエンドのテスト基盤がない場合、Go サービスのテストを優先し、フロントエンドのテストはスキップしてよい。
