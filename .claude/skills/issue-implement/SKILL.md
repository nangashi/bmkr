# issue-implement

GitHub Issue の実装計画をもとに、実装からPR作成まで自動で進める。issue-plan で作成された計画が Issue コメントに存在することが前提。

**重要: ユーザーとの対話および Issue への書き込みは日本語で行うこと。**

## 前提

- `/issue-plan` で計画が作成済みであること
- 計画は Issue コメントに `<!-- issue-plan:plan:done -->` マーカーで記録されている
- 実装不可の問題がなければ、PR作成まで対話なしで自動実行する

## ワークフロー

```
Phase 0: 初期化        → Issue・計画読み込み、ブランチ作成
Phase 1: I/F設計       → 型・インターフェース・シグネチャ定義 + 動作コメント（Opus サブエージェント）
Phase 2: 設計レビュー   → Codex 汎用レビュー + Opus プロジェクト固有レビュー（並列）
Phase 3: テスト作成     → I/F + 動作コメントからユニットテスト導出 — Red（Opus サブエージェント）
Phase 4: 実装          → テストを通すことがゴール — Green（Codex CLI、最大3ラウンド）
Phase 5: 最終レビュー   → Codex Review + Opus 多観点レビュー + 採用判定 + 修正
Phase 6: PR作成        → 受け入れ条件チェック・コミット・PR作成
```

各フェーズでモデルの得意領域を活かすハイブリッド構成:
- **I/F設計・テスト作成（Opus）**: 計画準拠度とマルチサービスの完遂力に優れる
- **設計レビュー（Codex 汎用 + Opus 固有）**: 別 LLM の視点でバイアスを補完しつつ、プロジェクト固有の制約は Opus で確認
- **実装（Codex）**: エッジケース処理・DRY・エラー情報保持など実装品質に優れる
- **最終レビュー（Codex + Opus）**: 既存の多観点レビュー構成を維持

このフローは I/F定義→テスト→実装の順（TDD）で進む。従来の「実装後にテスト」で起きていた問題（テストが実装の内部構造に依存する、設計問題の発見が遅れる）を解消するための設計。I/F定義フェーズで型・インターフェースが先に存在するため、TDD のコンパイルエラー問題も発生しない。

---

## Phase 0: 初期化

### 0a. 前提チェック

- `gh auth status` で GitHub CLI の利用可否を確認
- `which codex` で codex CLI の利用可否を確認（なければ Phase 2 の Codex 汎用レビュー、Phase 4 の Codex 実装、Phase 5 の Codex Review をすべてスキップし、従来の Opus サブエージェントにフォールバック）
- `git status --porcelain` で作業ツリーが clean であることを確認。未コミットの変更がある場合は `作業ツリーに未コミットの変更があります。コミットまたは stash してから再実行してください。` と表示して終了する

### 0b. 引数の解析

| パターン | 動作 |
|---------|------|
| `#123` または `123` | Issue を読み込む |
| 引数なし | `Issue番号を教えてください。` と表示して入力を待つ |

### 0c. 計画の検出

Issue コメントから `<!-- issue-plan:plan:done -->` マーカーを検索する。

**計画が見つからない場合:**

```
この Issue には実装計画がありません。
先に `/issue-plan #{番号}` で計画を作成してください。
```

と表示して終了する。

**計画が見つかった場合、以下を抽出する:**
- 実装ステップ一覧
- 変更ファイル一覧
- 受け入れ条件との対応
- リスクと対策

### 0d. Issue 本文の読み込み

Issue 本文から概要・方針・受け入れ条件（チェックボックス一覧）・スコープ外を抽出する。

### 0e. ブランチ作成・作業ディレクトリ

Issue 番号とタイトルから適切なブランチ名を生成してチェックアウトする。`.output/issue-implement/{issue_number}/` を作業用ディレクトリとして使用する。

---

## Phase 1: I/F設計

計画の変更ファイル一覧をもとに、型・インターフェース・関数シグネチャを定義し、動作コメントを記述する。実装本体は書かない。

`references/implementation.md` を Read で読み込み、その内容に従ってサブエージェントを起動する。

### サブエージェントに渡す情報:
- 計画全文
- Issue 本文（方針・受け入れ条件・スコープ外）
- 作業ディレクトリパス
- `CLAUDE.md` の内容

### このフェーズで作るもの

- 自動生成コードの前提作業（proto 変更 → buf generate、SQL 変更 → sqlc generate）
- 型定義（struct, type, interface）
- 関数・メソッドのシグネチャ（本体は `panic("not implemented")` または空実装）
- 動作コメント（正常系・異常系の振る舞いを自然言語で記述）
- フロントエンドの場合: Props 型、ページコンポーネントのシグネチャ、ルーティング定義

### このフェーズで作らないもの

- 関数・メソッドの実装本体
- テストコード
- ただしコード生成（buf generate, sqlc generate）や依存追加（go.mod, package.json）は実行する

### ブロッカー判定

以下の場合は実装を中断し、ユーザーに報告する:
- 計画の前提が現在のコードと合わない
- 外部サービスの設定変更が必要
- 計画の曖昧さにより複数の解釈が可能

---

## Phase 2: 設計レビュー

I/F定義のみを対象にレビューを実施する。実装前にレビューすることで、設計問題の修正コストを最小化する。Codex（汎用）と Opus（プロジェクト固有）の 2 レビューを **並列実行** する。

`references/review.md` の「設計レビュー」セクションを Read で読み込み、その内容に従う。

### 2a. Codex 汎用レビュー（Codex CLI）

`git diff main` を `codex exec --full-auto` に渡し、観点を縛らず自由にレビューさせる。Opus が書いた I/F を別 LLM の目で検証し、自己正当化バイアスを補完する。

ベンチマークで確認された Codex の得意領域: nullable フィールドの Valid チェック漏れ検出、DRY 違反、フレームワークのベストプラクティス（HTMX の outerHTML + push-url 等）、コンパイル時安全性（var _ 型アサーション）。

出力先: `.output/issue-implement/{issue_number}/review-design-codex.json`

### 2b. Opus プロジェクト固有レビュー（サブエージェント）

プロジェクト固有の知識が必要な観点を Opus サブエージェントで確認する:

- **既存パターン適合**: 周辺の既存コードと命名・構成が一致しているか
- **ADR 準拠**: `docs/adr/` の決定事項に反していないか

出力先: `.output/issue-implement/{issue_number}/review-design-opus.json`

### 指摘の統合と修正（投票ルール）

両レビューの指摘を集約する。独立した2モデルの共通指摘は単独指摘より優先度を高く扱い、以下のルールで採用を判定する:

- **Codex と Opus の両方が同一箇所を指摘** → 自動採用（修正必須）
- **片方のみの指摘** → メインエージェントが妥当性を判断して採用/棄却

採用された指摘で I/F定義を修正してから Phase 3 に進む。

---

## Phase 3: テスト作成 — Red

I/F定義と動作コメントをもとにユニットテストを作成する。テストは I/F（型・シグネチャ・動作コメント）と受け入れ条件だけから導出し、実装が存在しないため実装に依存することが原理的にできない。

`references/testing.md` を Read で読み込み、その内容に従う。

### 3a. テスト導出（Opus サブエージェント）

サブエージェントに渡す情報:
- Issue 本文（受け入れ条件が最重要）
- 計画の「変更ファイル一覧」と「受け入れ条件との対応」
- 作業ディレクトリパス
- `CLAUDE.md` の内容
- 「テストはコンパイルが通る状態にすること。実行すると失敗（Red）になるのが正しい状態」
- テスト導出レポートの出力指示（後述）

### テスト導出レポート

テストコードとともに、`.output/issue-implement/{issue_number}/test-derivation-report.json` にレポートを出力させる。このレポートは Phase 4 の診断・HANDOFF の入力となる。

```json
{
  "test_files": ["作成したテストファイル一覧"],
  "coverage_map": [
    {
      "acceptance_criteria": "受け入れ条件のテキスト（あれば）",
      "tests": ["対応するテスト関数名"],
      "source": "acceptance_criteria | behavior_comment | edge_case"
    }
  ],
  "uncovered_criteria": ["テストでカバーできなかった受け入れ条件（あれば）"],
  "difficulties": ["テスト作成で困難だった点（あれば）"]
}
```

- `source: acceptance_criteria` — Issue の受け入れ条件に由来するテスト
- `source: behavior_comment` — I/F 定義の動作コメントに由来するテスト
- `source: edge_case` — Phase 3b のエッジケース提案に由来するテスト（3c 後にレポートを更新）

**オーケストレータの検証:** レポートの `uncovered_criteria` が空でない場合、Phase 4 に進む前にサブエージェントにテスト追加を指示する。

### 3b. エッジケース提案（Codex CLI）

Opus がテストを書いた後、Codex に I/F 定義（動作コメント含む）を渡してエッジケースを提案させる。ベンチマークで Opus が `pgtype.Timestamptz.Valid` チェックを見落とした事例のように、別モデルの視点で漏れを補完する。

```bash
printf "%s" "$prompt" | timeout 120 codex exec --full-auto -
```

プロンプト:
```
以下は関数のシグネチャと動作コメントです。
この関数で起こりうるエッジケースを5つ挙げ、各ケースのテスト概要を提案してください。
既存テストと重複するものは除外してください。

{I/F定義 + 動作コメント}

既存テスト一覧:
{テストファイルの関数名一覧}
```

### 3c. エッジケース統合

Opus がメインエージェントとして Codex の提案を評価し、有効なものをテストに追加する。追加したテストを `test-derivation-report.json` の `coverage_map` に `source: edge_case` で追記する。

### Red の確認

テスト作成後、テストを実行して失敗することを確認する。コンパイルエラーは不可（I/Fが定義済みなので型エラーにはならないはず）。テストが失敗する（Red）のは正常。

---

## Phase 4: 実装 — Green（Codex CLI）

テストを通すことをゴールに、関数・メソッドの実装本体を書く。Codex CLI（`codex exec --full-auto`）で実行し、ループ制御と Judge 判定はオーケストレータ（Opus）が行う。

ベンチマークで確認された Codex の実装上の強み: nullable フィールドの Valid チェック、fetchProducts のような DRY ヘルパー抽出、エラー情報の保持、事前容量確保。

### Codex に渡すプロンプト（Round 1）

以下の情報を含むプロンプトを構築し、stdin から `timeout 300 codex exec --full-auto` に渡す:

- `panic("not implemented")` を実装に置き換えるタスク説明
- 動作コメント（I/F定義に記載済み）
- 計画の方針・スコープ外
- CLAUDE.md の内容
- テスト実行コマンドと期待される状態（全テスト Green）
- I/F 変更ルール（後述の「I/F 変更の許可条件」を参照）
- 実装レポートの出力指示（後述の「実装レポート」を参照）

### 実装レポート

Codex には実装コードとともに、`.output/issue-implement/{issue_number}/impl-report-round-{round}.json` にレポートを出力させる。このレポートは診断フェーズの入力となる。

```json
{
  "status": "green | partial | blocked",
  "implemented_files": ["変更したファイル一覧"],
  "passed_tests": ["通ったテスト名"],
  "failed_tests": ["失敗したテスト名"],
  "issues": [
    {
      "type": "test_issue | spec_gap | impl_difficulty",
      "test_name": "該当テスト（あれば）",
      "description": "何が問題か",
      "suggestion": "提案（あれば）"
    }
  ]
}
```

- `status: green` — 全テスト通過
- `status: partial` — 一部テスト失敗だが実装アプローチは妥当と判断
- `status: blocked` — テストやI/Fに問題があり、現状では通せないと判断

### Phase 4 開始前のコミット

Phase 3 完了後、Phase 4 開始前に Phase 1-3 の成果物をコミットする:

```bash
git add -A && git commit -m "checkpoint: phase3 complete (issue #{number})"
```

Phase 0 で clean worktree を確認済みのため、この時点の変更は全て当 Issue の作業。このコミットが Phase 4 の safe point となる。

### ループ（オーケストレータが制御）

```
prev_pass_count = 0
for round in 1..3:
  Codex exec → 実装 + レポート出力

  静的チェック:
    go mod tidy（import 修正）
    go vet ./...（静的解析）
    → vet 失敗の場合、テスト実行をスキップし「静的解析エラー」として診断フェーズへ

  テスト実行 → pass_count, fail_count, 失敗詳細をキャプチャ
  全パス → Phase 5 へ

  Judge 判定（pass_count + レポート）:
    pass_count > prev_pass_count → 改善:
      git add -A && git commit -m "wip: phase4-round-{round}"
    pass_count <= prev_pass_count → 停滞 or 悪化:
      git checkout -- . && git clean -fd  # 前コミット状態に復元
      → 診断フェーズへ

  prev_pass_count = pass_count
```

最大3ラウンドだが、診断フェーズで「解決困難」と判定された時点で早期中断する。

**ロールバックの仕組み**: `git add -A && git commit` は Judge が「改善」と判定した場合にのみ実行する。停滞・悪化時は `git checkout -- .`（追跡ファイル復元）+ `git clean -fd`（未追跡ファイル削除）で直前のコミット状態に確実に戻す。

### 診断フェーズ（Opus サブエージェント）

停滞・悪化時にオーケストレータが Opus サブエージェントを起動し、原因を分析して次のアクションを決定する。

**サブエージェントへの入力:**
- Codex の実装レポート（`impl-report-round-{round}.json`）
- テスト導出レポート（`test-derivation-report.json`）— テストの由来（受け入れ条件/動作コメント/エッジケース）の判断に使用
- テスト失敗詳細（テスト出力）
- I/F 定義 + 動作コメント
- 過去ラウンドの試行履歴（レポート + HANDOFF 内容）

**判定と対応:**

| 判定 | 条件 | 対応 |
|------|------|------|
| A. 実装アプローチの問題 | レポートに `impl_difficulty` があり、テスト自体は妥当 | HANDOFF に具体的な方針変更を記述 → 次ラウンドへ |
| B. テストの問題 | レポートに `test_issue` があり、診断で妥当性を確認 | テスト修正 → コミット → 次ラウンドへ |
| C. I/F の問題 | レポートに `spec_gap` があり、I/F 変更許可条件に該当 | I/F 変更プロセス（後述）に従って処理 → 次ラウンドへ |
| D. 解決困難 | 上記いずれでも解決の見込みがない | 中断、ユーザーに状況報告 |

### テスト修正の制約（診断フェーズ B）

診断サブエージェントがテスト修正を行う場合、以下の制約を守る:

**修正が許されるもの:**
- フィクスチャの誤り（型の初期化ミス、無効なテストデータ等）
- 動作コメントの解釈ミスに基づくテストの期待値
- テストのセットアップ不備（モック設定漏れ、コンテキスト不足等）

**修正が許されないもの:**
- テストの期待値を実装に合わせて変更すること（テストを実装に過適合させない）
- 受け入れ条件に由来するテストケースの削除（`test-derivation-report.json` の `source: acceptance_criteria` で判別）
- テストカバレッジを下げる変更

**検証手順:**
1. 修正後のテストが I/F 定義の動作コメントと矛盾しないことを確認
2. 修正後のテストが受け入れ条件をカバーし続けていることを確認
3. テスト修正をコミット: `git add -A && git commit -m "fix: test correction from diagnosis (round {round})"`

### I/F 変更の許可条件

原則として I/F（型・インターフェース・テンプレート）は変更しない。ただし以下のいずれかに該当する場合のみ、診断フェーズで変更を許可する:

- **受け入れ条件と I/F が矛盾する**（テストが原理的に書けない/通せない）
- **ビルド・テスト可能な最小 I/F に不足がある**（型やメソッドが足りない）
- **既存 ADR・既存コードパターンに反している**

実装都合（「この方が書きやすい」等）では変更しない。

**変更時のプロセス:**
1. Codex がレポートで `spec_gap` を報告
2. 診断サブエージェントが上記3条件のいずれかに該当するか判断
3. 該当する場合:
   - I/F を修正し、影響するテストを更新
   - `.output/issue-implement/{issue_number}/if-changes.json` に変更を記録
   - 変更が公開 I/F・複数ファイルに波及・既存 mock/template/handler に影響のいずれかなら Phase 2b（Opus プロジェクト固有レビュー）を再実行
4. 該当しない場合: 既存 I/F 内での実装を継続指示

**if-changes.json のフォーマット:**
```json
{
  "changes": [
    {
      "file": "変更ファイル",
      "type": "add_method | modify_signature | add_type",
      "before": "変更前（省略可）",
      "after": "変更後",
      "condition": "受け入れ条件矛盾 | 最小I/F不足 | ADR/パターン非準拠",
      "detail": "具体的な根拠",
      "requires_review": true
    }
  ]
}
```

Phase 5 のゴール整合性レビューは `if-changes.json` が存在する場合にそれを読み込み、I/F 変更が受け入れ条件との整合性を損なっていないか確認する。

### HANDOFF パターン（Round 2+ のプロンプト構造）

リトライ時は以下の構造でコンテキストを引き継ぐ。診断フェーズの結果を反映し、試行済みアプローチと禁止事項を明示することで同じ失敗の繰り返しを防ぐ:

```
## ゴール
テストを全て通すこと（Green）

## 完了済み
- {実装済みファイルとその状態の一覧}

## 失敗分析（診断フェーズの結果）
- {テスト名}: {失敗内容の要約}
- 根本原因: {診断サブエージェントの分析}
- 推奨アプローチ: {診断サブエージェントの提案}

## 試したアプローチ
- Round N: {前ラウンドで試した方法と結果}

## テスト修正履歴（該当する場合）
- Round N 診断で修正: {修正内容と理由}

## 禁止事項（再試行しないこと）
- テストの期待値を変更してはいけない（診断フェーズ経由の修正を除く）
- I/F（型・インターフェース）を変更してはいけない（診断フェーズ経由の変更を除く）
- {前ラウンドで失敗した具体的アプローチ}
```

### 失敗カテゴリ別リトライ戦略

診断サブエージェントがテスト失敗を分析し、カテゴリに応じた指示を HANDOFF プロンプトに追加する:

| カテゴリ | 判定基準 | 指示 |
|---------|---------|------|
| 静的解析エラー | `go vet` 失敗 | vet 出力を HANDOFF に含めて即リトライ |
| コンパイルエラー | `go build` 失敗 | 型情報を補強して即リトライ |
| ロジックエラー | テスト失敗（期待値と実値の不一致） | 診断結果に基づく具体的アプローチを指示 |
| テスト不備 | テストのセットアップ・フィクスチャに問題 | 診断フェーズでテスト修正後にリトライ |
| インフラエラー | import 未解決、依存不足 | `go mod tidy` で未解決なら依存確認を指示 |
| sandbox 制約 | codex exec タイムアウト or 権限エラー | Opus サブエージェントにフォールバック |

### Codex CLI が利用できない場合

Phase 0 で codex CLI が検出できなかった場合は、従来通り Opus サブエージェントで実装を実行する。実装レポートの出力・診断フェーズの構造は同様に適用する。

### 中断条件

以下のいずれかで中断し、PR は作成しない:
- 診断フェーズで「解決困難」と判定
- 3ラウンド到達後もテストが通らない
- codex exec が連続2回タイムアウト（sandbox 制約と判断し Opus フォールバックへ）

**中断時の状態報告:**
中断した場合、ユーザーに以下を報告する:
```
Phase 4 が {理由} で中断しました。
現在の状態: {最後に成功コミットしたハッシュ}（{phase3 complete or round N}）
診断結果: {最後の診断フェーズの分析サマリ}
手動で実装を続ける場合はこのブランチで作業できます。
```

## Phase 5: 最終レビュー & 修正

実装完了後、多観点でコードレビューを実施する。Phase 2 の設計レビューで I/F レベルの問題は解消済みのため、ここでは実装レベルの問題に集中する。

`references/review.md` の「最終レビュー」セクションを Read で読み込み、その内容に従う。

### 5a. LLM 特有の前処理 + 静的解析

- `go mod tidy` を実行し差分を確認（API 幻覚の検出）
- `go vet ./...` を変更対象の Go サービスで実行（AI 生成コードは並行性エラーが多い傾向がある — CodeRabbit 2025 分析、vendor 調査）
- `git diff main` の変更ファイルに TODO/FIXME/HACK/XXX が残存していないか確認

### 5b. レビュー実行（並列）

**常時実行:**
1. **Codex Review（別LLM）** — `/codex-review` スキルで `--diff main` を対象にレビュー
2. **ゴール整合性レビュー（サブエージェント）** — 受け入れ条件カバレッジ、スコープ逸脱、計画外の追加機能検出
3. **変更容易性レビュー（サブエージェント）** — 実装レベルの確認（LLMアンチパターン、cargo cult 等。I/Fレベルは Phase 2 で済み）

**条件付き実行:**
4. **セキュリティ深掘り（サブエージェント）** — 認証・認可・トークン・入力処理を含む場合
5. **パフォーマンス深掘り（サブエージェント）** — DB クエリ・大量データ処理を含む場合

### 5c. 採用判定（別サブエージェント）

`agents/review-filter.md` を Read で読み込み、採用判定サブエージェントを起動する。

### 5d. 修正 & テスト再実行

採用された指摘に対応してコードを修正し、テストが壊れていないか確認する。

## Gotchas

このスキルを使う中で判明した注意点。運用しながら追記していく。

### プロセス固有

- **サブエージェントがファイル書き出しを忘れる**: プロンプトに出力先パスを明記し「Write で書き出すこと」を指示する。テキスト返却だけでは後続フェーズで参照できない
- **`gh issue edit --body` のシェルエスケープ**: インラインで本文を渡すと特殊文字で問題が起きる。長い本文は一時ファイルに書き出して `--body-file` を使う
- **codex CLI 未インストール時**: Phase 0 で検出し、Phase 2 の Codex 汎用レビュー・Phase 4 の Codex 実装・Phase 5 の Codex Review をスキップし、全て Opus サブエージェントにフォールバック
- **Codex exec のネットワーク制限**: codex exec は sandbox 内で動作するため、`buf generate`（リモートプラグイン）等のネットワーク依存コマンドが失敗する場合がある。Phase 4 で Codex が実行するのは実装のみで、コード生成は Phase 1（Opus）で完了済みのため影響はない
- **Codex exec のセッション独立性**: Phase 4 の各ラウンドで Codex は独立セッションで起動される。前ラウンドの文脈はプロンプトに失敗内容を含めることで引き継ぐ
- **Phase 1 で実装本体を書かない**: I/F設計フェーズでは型・シグネチャ・動作コメントのみ。実装を書いてしまうとテストの独立性が失われる

### 実装パターン

実装時に注意すべきパターンは `docs/guides/` に外出ししている。サブエージェントのプロンプトに該当ガイドのパスを含めること。

- Go 固有: `docs/guides/go/` — sqlc型境界、Handler DI、Querierスコープ、templ doctype、コード生成順序
- アーキテクチャ: `docs/guides/architecture/` — 過剰抽象化、過剰外部化、形式的エラーハンドリング、状態変更後の再フェッチ
