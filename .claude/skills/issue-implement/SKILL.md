# issue-implement

GitHub Issue の実装計画をもとに、実装からPR作成まで自動で進める。issue-plan で作成された計画が Issue コメントに存在することが前提。

**重要: ユーザーとの対話および Issue への書き込みは日本語で行うこと。**

## 前提

- `/issue-plan` で計画が作成済みであること
- 計画は Issue コメントに `<!-- issue-plan:plan:done -->` マーカーで記録されている
- 実装不可の問題がなければ、PR作成まで対話なしで自動実行する

## ワークフロー

```
Phase 0: 初期化        → Issue・計画読み込み、ブランチ作成
Phase 1: I/F設計       → 型・インターフェース・シグネチャ定義 + 動作コメント（サブエージェント）
Phase 2: 設計レビュー   → I/Fのみを対象に変更容易性レビュー（別サブエージェント）
Phase 3: テスト作成     → I/F + 動作コメントからユニットテスト導出 — Red（別サブエージェント）
Phase 4: 実装          → テストを通すことがゴール — Green（サブエージェント、最大3ラウンド）
Phase 5: 最終レビュー   → Codex Review + ゴール整合性 + 残りの観点 + 採用判定 + 修正
Phase 6: PR作成        → 受け入れ条件チェック・コミット・PR作成
```

このフローは I/F定義→テスト→実装の順（TDD）で進む。従来の「実装後にテスト」で起きていた問題（テストが実装の内部構造に依存する、設計問題の発見が遅れる）を解消するための設計。I/F定義フェーズで型・インターフェースが先に存在するため、TDD のコンパイルエラー問題も発生しない。

---

## Phase 0: 初期化

### 0a. 前提チェック

- `gh auth status` で GitHub CLI の利用可否を確認
- `which codex` で codex CLI の利用可否を確認（なければ Phase 5 の Codex Review をスキップ）

### 0b. 引数の解析

| パターン | 動作 |
|---------|------|
| `#123` または `123` | Issue を読み込む |
| 引数なし | `Issue番号を教えてください。` と表示して入力を待つ |

### 0c. 計画の検出

Issue コメントから `<!-- issue-plan:plan:done -->` マーカーを検索する。

**計画が見つからない場合:**

```
この Issue には実装計画がありません。
先に `/issue-plan #{番号}` で計画を作成してください。
```

と表示して終了する。

**計画が見つかった場合、以下を抽出する:**
- 実装ステップ一覧
- 変更ファイル一覧
- 受け入れ条件との対応
- リスクと対策

### 0d. Issue 本文の読み込み

Issue 本文から概要・方針・受け入れ条件（チェックボックス一覧）・スコープ外を抽出する。

### 0e. ブランチ作成・作業ディレクトリ

Issue 番号とタイトルから適切なブランチ名を生成してチェックアウトする。`.output/issue-implement/{issue_number}/` を作業用ディレクトリとして使用する。

---

## Phase 1: I/F設計

計画の変更ファイル一覧をもとに、型・インターフェース・関数シグネチャを定義し、動作コメントを記述する。実装本体は書かない。

`references/implementation.md` を Read で読み込み、その内容に従ってサブエージェントを起動する。

### サブエージェントに渡す情報:
- 計画全文
- Issue 本文（方針・受け入れ条件・スコープ外）
- 作業ディレクトリパス
- `CLAUDE.md` の内容

### このフェーズで作るもの

- 自動生成コードの前提作業（proto 変更 → buf generate、SQL 変更 → sqlc generate）
- 型定義（struct, type, interface）
- 関数・メソッドのシグネチャ（本体は `panic("not implemented")` または空実装）
- 動作コメント（正常系・異常系の振る舞いを自然言語で記述）
- フロントエンドの場合: Props 型、ページコンポーネントのシグネチャ、ルーティング定義

### このフェーズで作らないもの

- 関数・メソッドの実装本体
- テストコード
- ただしコード生成（buf generate, sqlc generate）や依存追加（go.mod, package.json）は実行する

### ブロッカー判定

以下の場合は実装を中断し、ユーザーに報告する:
- 計画の前提が現在のコードと合わない
- 外部サービスの設定変更が必要
- 計画の曖昧さにより複数の解釈が可能

---

## Phase 2: 設計レビュー

I/F定義のみを対象に変更容易性レビューを実施する。実装前にレビューすることで、設計問題の修正コストを最小化する。

`references/review.md` の「設計レビュー」セクションを Read で読み込み、その内容に従ってサブエージェントを起動する。

### レビュー観点（I/Fレベルで判断できるもの）:
- **情報隠蔽**: DB 生成型がプレゼンテーション層に漏れていないか
- **密結合**: 具象型ではなくインターフェースに依存しているか（テスタビリティ）
- **凝集度**: 1つのインターフェース/型が複数の関心を持っていないか
- **名前の正確性**: 型名・メソッド名・フィールド名が実態を表しているか
- **既存パターン適合**: 周辺の既存コードと命名・構成が一致しているか
- **ADR 準拠**: `docs/adr/` の決定事項に反していないか

指摘があれば I/F定義を修正してから Phase 3 に進む。

---

## Phase 3: テスト作成 — Red

I/F定義と動作コメントをもとにユニットテストを作成する。テストは I/F（型・シグネチャ・動作コメント）と受け入れ条件だけから導出し、実装が存在しないため実装に依存することが原理的にできない。

`references/testing.md` を Read で読み込み、その内容に従ってサブエージェントを起動する。

### サブエージェントに渡す情報:
- Issue 本文（受け入れ条件が最重要）
- 計画の「変更ファイル一覧」と「受け入れ条件との対応」
- 作業ディレクトリパス
- `CLAUDE.md` の内容
- 「テストはコンパイルが通る状態にすること。実行すると失敗（Red）になるのが正しい状態」

### Red の確認

テスト作成後、テストを実行して失敗することを確認する。コンパイルエラーは不可（I/Fが定義済みなので型エラーにはならないはず）。テストが失敗する（Red）のは正常。

---

## Phase 4: 実装 — Green

テストを通すことをゴールに、関数・メソッドの実装本体を書く。最大3ラウンド。

### サブエージェントに渡す情報:
- 計画全文
- Issue 本文（方針・スコープ外）
- 作業ディレクトリパス
- `CLAUDE.md` の内容
- 「テストが全て通る（Green）状態にすること」

### ループ

```
for round in 1..3:
  テスト実行
  全パス → Phase 5 へ
  失敗内容を分析 → 実装を修正
```

### 中断条件

3ラウンド後もテストが通らない場合、失敗しているテストと理由を報告して中断する。PR は作成しない。

---

## Phase 5: 最終レビュー & 修正

実装完了後、多観点でコードレビューを実施する。Phase 2 の設計レビューで I/F レベルの問題は解消済みのため、ここでは実装レベルの問題に集中する。

`references/review.md` の「最終レビュー」セクションを Read で読み込み、その内容に従う。

### 5a. LLM 特有の前処理

- `go mod tidy` を実行し差分を確認（API 幻覚の検出）
- `git diff main` の変更ファイルに TODO/FIXME/HACK/XXX が残存していないか確認

### 5b. レビュー実行（並列）

**常時実行:**
1. **Codex Review（別LLM）** — `/codex-review` スキルで `--diff main` を対象にレビュー
2. **ゴール整合性レビュー（サブエージェント）** — 受け入れ条件カバレッジ、スコープ逸脱、計画外の追加機能検出
3. **変更容易性レビュー（サブエージェント）** — 実装レベルの確認（LLMアンチパターン、cargo cult 等。I/Fレベルは Phase 2 で済み）

**条件付き実行:**
4. **セキュリティ深掘り（サブエージェント）** — 認証・認可・トークン・入力処理を含む場合
5. **パフォーマンス深掘り（サブエージェント）** — DB クエリ・大量データ処理を含む場合

### 5c. 採用判定（別サブエージェント）

`agents/review-filter.md` を Read で読み込み、採用判定サブエージェントを起動する。

### 5d. 修正 & テスト再実行

採用された指摘に対応してコードを修正し、テストが壊れていないか確認する。

---

## Phase 6: PR 作成

### 6a. 受け入れ条件の確認 & チェックボックス更新

Issue 本文の受け入れ条件を1つずつ確認する。満たされている条件のチェックボックスを `- [x]` に更新する（`--body-file` で Issue 本文を書き換え）。

満たされていない条件がある場合は、その旨をユーザーに報告し、PR の Review Notes にも記載する。

### 6b. コミット・プッシュ・PR 作成

`.github/PULL_REQUEST_TEMPLATE.md` のフォーマットに従って PR body を作成する。`Closes #{番号}` を含めること。

### 6c. 完了報告

PR URL を表示する。

---

## Gotchas

このスキルを使う中で判明した注意点。運用しながら追記していく。

- **サブエージェントがファイル書き出しを忘れる**: プロンプトに出力先パスを明記し「Write で書き出すこと」を指示する。テキスト返却だけでは後続フェーズで参照できない
- **`gh issue edit --body` のシェルエスケープ**: インラインで本文を渡すと特殊文字で問題が起きる。長い本文は一時ファイルに書き出して `--body-file` を使う
- **自動生成ファイルの実行順序**: proto/SQL の変更を含む場合、`buf generate` / `sqlc generate` を先に実行してから手動実装コードに進む。生成コードに依存する実装が先に来るとコンパイルエラーになる
- **codex CLI 未インストール時**: Phase 0 で検出し、Phase 5 の Codex Review をスキップする。他のレビューは影響なし
- **DB 型をテンプレート/プレゼンテーション層に直接渡さない**: sqlc 生成型（`db.Product` 等）をテンプレートに直接渡すと、DB スキーマ変更がプレゼンテーション層に波及する。テンプレート用の型を定義し、ハンドラで変換すること
- **ハンドラのテスタビリティ**: ハンドラが `*db.Queries` を直接持つとモック不可になり、DB エラーパスがテストできない。DB 依存はインターフェース経由で注入し、テストではモック実装を使う
- **状態変更後の再フェッチ**: ページ番号のクランプなど状態を変更した場合、変更後の値でデータを再取得すること。クランプだけして古いデータを返すバグが起きやすい
- **Phase 1 で実装本体を書かない**: I/F設計フェーズでは型・シグネチャ・動作コメントのみ。実装を書いてしまうとテストの独立性が失われる
- **templ は小文字 `<!doctype html>` を出力する**: テストで `<!DOCTYPE html>`（大文字）を期待するとテストが失敗する。templ 生成コードの実際の出力を確認してからテストの期待値を書くこと
- **sqlc Querier にメソッド追加すると既存テストの mock が壊れる**: sqlc generate で Querier インターフェースにメソッドが追加されると、既存テストの mock 実装にもメソッド追加が必要になる。Phase 1 の自動生成コード前提作業の一部として認識すること
