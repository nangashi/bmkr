# Review: プロジェクト準拠

## Overview
ルール準拠、ADR/ガイド準拠、既存パターンとの一貫性、命名規則をレビューする。

## Step 1: Analysis（対象コードの構造を把握する）

### 1a. ルール照合（最優先）

`.claude/rules/` 配下の全ファイルを読み、各ルールの frontmatter `globs` と変更ファイルのパスを照合する。glob が一致するルールを「適用対象ルール」として列挙する。

各適用対象ルールについて:
1. ルールの「アンチパターン」に該当するコードが変更差分に含まれていないか確認する
2. ルールの「正しいパターン」に従っているか確認する
3. ルールに「例外」がある場合は、変更内容がその例外に該当するか確認する

### 1b. ADR・ガイド・既存パターン照合

以下をマッピング対象とする:
- 新規・変更されたファイル全て
- `docs/adr/` 配下の ADR（変更内容に関連するもの）
- `docs/guides/` 配下の全ガイド（変更内容との準拠チェック）
- 変更対象と同一パッケージ / ディレクトリの既存コード

### 各箇所について記録すること
1. 適用対象ルールとの準拠状況（違反がある場合はルールファイル名とアンチパターン名を記録）
2. 変更内容に関連する ADR の決定事項と、コードがそれに従っているか
3. 変更内容に関連するガイドのパターンと、コードがそれに従っているか
4. 同一パッケージ内の既存コードの命名パターン（関数名、型名、変数名、ファイル名）
5. 既存コードのファイル構成パターン（handler, types, service 等の分け方）
6. 既存コードのエラーハンドリングパターン

## Step 2: Findings（分析結果から指摘を導出する）

### 指摘とする基準
- **ルール違反**（最優先）: `.claude/rules/` のルールに記載されたアンチパターンに該当するコード。ルールは具体的で機械的に判定可能なため、該当すれば必ず指摘する
- ADR の決定事項に反するコード（例: ADR で選定した技術と異なるライブラリの使用）
- ガイドに記載されたパターンに違反するコード
- 同一パッケージ内の既存コードと命名規則が異なる（例: 既存が camelCase なのに snake_case）
- 同一サービス内の既存ファイル構成と異なる配置（例: 既存が handler.go に集約しているのに別ファイルに分割）
- 既存コードと異なるエラーハンドリングパターン（例: 既存が connect.NewError を使うのに errors.New を使う）

### 指摘としない基準
- ルールの「例外」に明示的に該当するケース
- ADR やガイドが存在しない領域の設計判断（前例がなければ新しいパターンも許容）
- 既存コードのパターンが明らかに非推奨・問題がある場合の改善
- テストコードの構成（テストは既存パターンとの一貫性より可読性を優先。ただしテスト用ルールは適用する）
- 自動生成コード（protobuf, sqlc, templ 等）のパターン

### perspective ラベル
プロジェクト準拠

### 出力時の補足
- ルール違反の場合、analysis にルールファイル名（例: `sqlc-exec-rows.md`）と違反したアンチパターンを明記すること
- analysis には参照した ADR/ガイド/既存コードのパスを明記すること
- suggestion には準拠すべきルール/ADR/ガイド/既存パターンを引用すること
