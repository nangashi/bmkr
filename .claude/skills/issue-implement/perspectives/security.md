# Review: セキュリティ

## Overview
エラーレスポンスの情報漏洩、入力バリデーション、secrets 非埋め込みをレビューする。

## Step 1: Analysis（対象コードの構造を把握する）

### マッピング対象
- HTTP / gRPC レスポンスを構築する箇所
- ユーザー入力を受け取る箇所（リクエストパラメータ、フォーム値、パスパラメータ）
- 設定値・シークレットの読み込み箇所
- SQL クエリの構築箇所

### 各箇所について記録すること
1. エラーレスポンスに含まれる情報（スタックトレース、内部エラーメッセージ、DB エラー詳細等）
2. ユーザー入力に対するバリデーションの有無と内容
3. ハードコードされた秘密情報（API キー、パスワード、トークン等）の有無
4. SQL クエリへのパラメータの渡し方（プレースホルダ vs 文字列連結）
5. ファイルパスやURLの構築にユーザー入力が使われているか

## Step 2: Findings（分析結果から指摘を導出する）

### 指摘とする基準
- エラーレスポンスに内部実装の詳細が含まれている（スタックトレース、DB エラーメッセージ、内部パス等）
- ユーザー入力をバリデーションなしで使用している（特に DB クエリ、ファイルパス、外部 API 呼び出しに渡す場合）
- シークレット（API キー、パスワード、トークン等）がソースコードにハードコードされている
- SQL クエリが文字列連結で構築されている（sqlc 使用時は該当しない）
- パストラバーサルの可能性があるファイルパス構築
- ユーザー入力をそのままログに出力している（PII 漏洩のリスク）

### 指摘としない基準
- sqlc が生成したクエリ（パラメータバインディングが保証されている）
- 開発環境専用の設定値（.env.example 等に記載されたデフォルト値）
- テストコード内のハードコード値（テストフィクスチャ）
- connect-go / net/http が標準で返すエラーレスポンス形式
