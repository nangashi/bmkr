# Review: 型設計

## Overview
インターフェース粒度、構造体フィールド可視性、不変条件の型表現が適切かをレビューする。

## Step 1: Analysis（対象コードの構造を把握する）

### マッピング対象
- 新規・変更された struct 定義
- 新規・変更された interface 定義
- 関数シグネチャ（引数・戻り値の型）

### 各箇所について記録すること
1. struct のフィールド可視性（exported / unexported）が意図的か
2. interface が持つメソッド数と粒度（1-3メソッドが理想、肥大化していないか）
3. 不変条件（non-nil、non-empty、正の値等）が型で表現されているか、それとも実行時チェックに依存しているか
4. 型エイリアスや独自型が意味のある区別を提供しているか（string を別名にしただけで混同防止に寄与しない等）
5. 引数・戻り値で具象型と interface のどちらを使っているか、その選択理由が妥当か

## Step 2: Findings（分析結果から指摘を導出する）

### 指摘とする基準
- interface に 4 つ以上のメソッドがあり、利用箇所が単一の実装しか想定していない
- 公開 struct のフィールドが全て exported だが、直接操作されると不変条件が壊れる
- string や int をそのまま使っているが、ドメイン上の制約（例: 空文字禁止、正の整数のみ）がある
- 関数の引数に具象型を要求しているが、テスト時にモック差し替えが必要になる箇所

### 指摘としない基準
- フレームワーク・ライブラリが要求する型（例: http.Handler, connect.Request）への準拠
- 既存コードベースで確立されたパターンの踏襲（一貫性優先）
- protobuf / sqlc 等が自動生成した型

型設計
