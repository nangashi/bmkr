# Review: サイレントフェイル

## Overview
エラー握りつぶし、log して return nil、recover で panic 吸収など、障害を隠蔽するパターンを検出する。

## Step 1: Analysis（対象コードの構造を把握する）

### マッピング対象
- error を返す関数呼び出しの全箇所
- recover() の使用箇所
- log.Error / log.Warn 等のログ出力後の制御フロー
- `_ =` または `_ ,` でエラーを明示的に無視している箇所

### 各箇所について記録すること
1. エラーが返されているか、無視されているか
2. エラーをログに記録した後、nil を返しているか（呼び出し元にエラーが伝播しない）
3. recover() で panic を吸収した後、正常系として処理を続行しているか
4. defer 内でのエラー（例: file.Close()）が無視されているか

## Step 2: Findings（分析結果から指摘を導出する）

### 指摘とする基準
- error を受け取りながら `_ =` で無視し、後続処理がそのエラーに依存している
- error をログ出力した後に `return nil` で呼び出し元に成功を返している
- recover() で panic を吸収し、エラーとして返さずに処理を続行している
- 複数の error を順次無視して最後まで到達するパス（連鎖的な障害隠蔽）

### 指摘としない基準
- defer 内の Close() エラー無視（リソース解放時の慣習的パターン。ただし Write 系の Close は指摘対象）
- テストコード内でのエラー無視（t.Fatal / t.Error で処理されていれば問題なし）
- 意図的なフォールバック（エラー時にデフォルト値を返す設計が動作コメントで明記されている場合）
- fmt.Fprintf 等の書き込み系で、エラーが実質発生しない宛先（bytes.Buffer 等）への書き込み
