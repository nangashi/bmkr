# テスト導出 — サブエージェント詳細手順

I/F定義の動作コメントからユニットテストを導出する。実装本体はまだ存在しないため、テストが実装に依存することは原理的にできない。

---

## 前提情報の取得

- `gh issue view {issue_number}` および `gh issue view {issue_number} --comments` で Issue 本文・コメント（計画含む）を直接読む
- テストはコンパイルが通る状態にすること。実行すると失敗（Red）になるのが正しい状態

---

## 強制学習（テスト作成前の必須調査）

テスト導出に入る前に、以下の調査を必ず実施する。省略してテストを書き始めてはならない。

### A. 既存テストパターンの調査

変更対象サービスの既存テストファイル（`*_test.go`, `*.test.ts`）を検索し、以下を確認する:

1. **テストヘルパー**: `newContext`, `newRequest` 等があれば再利用する。同等の関数を新規に作らない
2. **モック実装**: 既存の `mock*` 構造体があれば命名規則・構造に従う
3. **フィクスチャ**: `sample*`, `fixture*` 等のデータ生成関数があれば再利用する
4. **コンパイル時チェック**: `var _ Interface = (*Mock)(nil)` パターンの有無を確認し、あれば同じパターンを使う

### B. テスト関連の注意事項

sqlc generate で Querier に新メソッドが追加されている場合、既存テストのモック実装にもメソッドスタブを追加する（`panic("not called")` で良い）。

---

## テスト導出の手順

### 1. 動作コメントを読む

I/F設計フェーズで定義された関数・メソッドの動作コメントを読み、テスト可能な振る舞いを列挙する。

例: `HandleProductList` の動作コメントから:
- page パラメータのデフォルト値（1）
- 不正値のフォールバック（0以下、非数値 → 1）
- DB エラー時の 500 レスポンス
- HX-Request ヘッダによるレスポンス分岐
- ページ超過時のクランプと再フェッチ

### 2. テスト手段を選ぶ

`.claude/rules/go-test.md` のルールに従い、テスト対象ごとに適切な手段を選択する。

**判断フロー:**
1. 受け入れ条件が静的解析（`go/parser` で import 検証、grep で命名規約検証等）でカバーできるか → できるなら静的解析テストを優先
2. テスト対象がインターフェース DI 済みか → 済みならそのインターフェースをモック
3. `*db.Queries` 直接依存で DBTX レベルのモックが必要か → 必要なら静的解析テストで代替するか、テスト対象外とする

### 3. テストを書く

正常系・異常系・境界値をカバーするテストを書く。既存テストパターンがあればそれに従う。同一コードパスのバリエーションはテーブルドリブンテストでまとめる。

### 4. 受け入れ条件クロスチェック

Issue 本文の受け入れ条件を1つずつ確認し、各条件に対応するテストが最低1つ存在することを確認する。動作コメントだけでなく、Issue の受け入れ条件を独立した入力として使う。

- 受け入れ条件に対応するテストがなければ追加する
- 動作コメントと受け入れ条件が矛盾する場合は、Issue の受け入れ条件を優先する
- 非機能制約（レスポンス構造不変、パフォーマンス要件等）もテスト可能なら検証を追加する

### 5. Red を確認する

`just test` を実行し、コンパイルが通ること + テストが失敗すること（Red）を確認する。コンパイルエラーが出た場合は I/F定義に問題があるので修正する。

---

## フロントエンドのテスト

React コンポーネントのテストは Go ほど TDD が自然ではなく、テスト基盤の有無にも依存する。プロジェクトにフロントエンドのテスト基盤がない場合、Go サービスのテストを優先し、フロントエンドのテストはスキップしてよい。
