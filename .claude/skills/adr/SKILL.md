# ADR (Architecture Decision Record) Creator

MADR (Markdown Architectural Decision Records) 形式で ADR を作成する。ADR は重要なアーキテクチャ上の意思決定を、その背景と結果とともに記録する。将来のチームメンバーが「何を」決めたかだけでなく「なぜ」そう決めたかを理解できるようにすることが目的。

**重要: ユーザーとの対話および生成する ADR ファイルの内容は、すべて日本語で行うこと。** テンプレート内の固定構文（frontmatter の `status` など）およびセクション見出し（`## Decision Outcome` など）は英語のまま維持する。

## Step 1: トピックの決定

`$ARGUMENTS` を確認する:
- `propose` が指定されている場合（例: `/adr propose`）、**提案モード**に入る → 「提案モード」セクション（本ファイル末尾）に従う
- `check` が指定されている場合（例: `/adr check`）、**チェックモード**に入る → 「チェックモード」セクションに従う
- `re-evaluate` + 番号が指定されている場合（例: `/adr re-evaluate 0012`）、**再評価モード**に入る → 「再評価モード」セクションに従う
- トピックが指定されている場合（例: `/adr ローカルストレージに SQLite を使用`）、それを出発点にする
- 引数がない場合、ユーザーに以下を案内する:

```
ADR スキルでは以下の操作が可能です:

1. **作成** — 特定の意思決定を ADR として記録する（例: `/adr ローカルストレージに SQLite を使用`）
2. **提案** — プロジェクトの要求・設計・既存 ADR を分析し、作成すべき ADR を提案する（`/adr propose`）
3. **チェック** — 全ADRの鮮度を監査する（`/adr check`）
4. **再評価** — 特定ADRを簡易チェックし、必要なら supersede する（`/adr re-evaluate NNNN`）

どの操作を行いますか？トピックを教えていただくか、「提案」と答えていただければ提案モードに入ります。
```

ユーザーが「提案」「propose」等と回答した場合は**提案モード**に入る → 「提案モード」セクション（本ファイル末尾）に従う。
ユーザーが「チェック」「check」等と回答した場合は**チェックモード**に入る → 「チェックモード」セクションに従う。
ユーザーが「再評価」「re-evaluate」「supersede」等と回答した場合は**再評価モード**に入る → 「再評価モード」セクションに従う。

トピックにはユーザーが背景情報、選択肢、要件などを含めている場合がある。これらの情報は Step 3 のフレームワーク提案に活用する。

## Step 2: 次の ADR 番号を決定

Glob で `docs/adr/*.md` にマッチするファイルを検索する。数値プレフィックス（例: `0001`, `0002`）を解析し、次の連番を決定する。ADR ファイルが存在しない場合は `0001` から開始する。

`docs/adr/` ディレクトリが存在しない場合は `mkdir -p docs/adr` で作成する。

## Step 3: フレームワークの提案

LLM がトピック・プロジェクト文脈・既存コードから以下を**提案**し、ユーザーの承認を得る。ユーザーに質問して収集するのではなく、LLM が主体的にドラフトする。

### 3a. 選択肢の調査（候補出し）

選択肢を提案する**前に**、最新の候補を把握するための調査を行う。LLM の学習データは古い可能性があるため、このステップは必須。

**調査方法 — 3段階の検索で網羅性を確保する:**

Agent ツール（subagent_type=general-purpose）を使い、以下の 3 段階で調査する。1 つのクエリで全候補が見つかることはまずないため、観点を変えた複数回の検索が重要。

**第1段階: 主要候補の発見**
- WebSearch で直接的な比較記事・まとめ記事を探す
  - `"{トピック} comparison {現在の年}"`, `"{トピック} vs"`, `"best {トピック} for {技術スタック}"`
- 比較記事が見つかったら WebFetch で記事本文を読み、言及されている選択肢をすべてリストアップする

**第2段階: 見落とし候補の探索**
- 第1段階で見つかった主要候補の名前を使い、さらに代替案を検索する
  - `"{主要候補A} alternatives"`, `"{主要候補A} vs"` — これにより主要候補の「対抗馬」として語られる選択肢を発見できる
- GitHub の awesome リストや技術レーダー（ThoughtWorks Technology Radar 等）も確認する
  - `"awesome-{トピック}" site:github.com"`, `"{トピック} technology radar"`
- ユーザーが既に選択肢を指定している場合でも、調査により見落としている有力な代替案がないか確認する

**第3段階: ニッチ・新興候補の確認**
- 新しいプロジェクトや異なるアプローチの候補を検索する
  - `"{トピック} new {現在の年}"`, `"{トピック} alternative approach"`, `"{トピック} lightweight"`
- 主要候補とは異なる設計思想を持つ選択肢（例: ライブラリ vs フレームワーク、組み込み vs 外部サービス）がないか意識する

**調査結果の活用:**
- 3段階の検索で発見した選択肢を、ユーザー指定の選択肢と合わせて次の提案内容に反映する
- 各選択肢について「なぜ候補に含めたか」の根拠（公式サイト・比較記事・コミュニティでの評価等）を把握しておく
- 候補が多数見つかった場合は、プロジェクトの前提条件に明らかに合わないものを除外し、有力な候補に絞る。除外した候補とその理由も簡潔に記録しておく

### 3b. 提案する内容:

1. **背景と課題（Context and Problem Statement）** — トピックおよびプロジェクト内のドキュメントから読み取れる状況を 1〜2 文でドラフトする。システムが複数のコンポーネントやサービスで構成される場合、この ADR が対象とする範囲と対象外の事項を明示する
2. **前提条件（Prerequisites）** — この決定が依存する既存の意思決定、採用済み技術、制約を推定して列挙する。既存の ADR（`docs/adr/` 内）、プロジェクト内のドキュメント（要件定義・設計文書等）、プロジェクト構成を参照する
3. **検討する選択肢（Considered Options）** — ユーザーが言及した選択肢に加え、**Step 3a の調査で発見した有力な代替案**を含める。各選択肢には調査に基づく簡潔な説明（1行）を付記する
4. **判断軸（Decision Drivers）** — 選択肢を評価するための軸を **4つ** 提案する。判断軸はこの決定に固有の具体的な評価基準とする（例: 「PTY 操作のしやすさ」「バイナリサイズ」など）。汎用的すぎる軸（例: 「品質」「コスト」）は避ける

### 提示形式:

```
以下のフレームワークで比較調査を行います:

**背景・課題**: {ドラフト}

**前提条件**:
* {前提条件1}
* {前提条件2}

**検討する選択肢**（調査に基づく）:
1. {選択肢1} — {簡潔な説明と候補に含めた根拠}
2. {選択肢2} — {簡潔な説明と候補に含めた根拠}
3. {選択肢3} — {簡潔な説明と候補に含めた根拠}

**判断軸**:
1. {軸1} — {その軸で何を評価するかの補足}
2. {軸2} — {補足}
3. {軸3} — {補足}
4. {軸4} — {補足}

修正・追加があれば教えてください。問題なければこの内容で調査を開始します。
```

ユーザーが修正を指示した場合は内容を更新し、再度提示する。承認が得られたら次のステップに進む。

## Step 4: 比較調査

承認されたフレームワーク（選択肢 × 判断軸）に基づき、各選択肢を調査する。

### 調査方法:

**選択肢ごとにサブエージェント（Agent ツール）を起動し、並列で調査する。** これによりメインコンテキストの消費を抑える。

各サブエージェントへの指示には以下を含める:
- 調査対象の選択肢
- 承認済みの判断軸（4つ）
- **承認済みの前提条件（全項目）** — 互換性チェックに必須
- プロジェクトの背景・要件
- 以下の調査タスク

サブエージェントは WebSearch、WebFetch、context7 プラグイン等を活用して事実に基づく情報を収集する。

#### 調査タスク 1: 前提条件との互換性チェック（必須）

判断軸の評価に先立ち、各前提条件との互換性を検証する。これは選択肢の適格性を判断するための最重要チェックであり、省略しない。

前提条件ごとに以下を確認する:
- **互換性**: その選択肢は前提条件の技術・制約と組み合わせて動作するか
- **既知の問題**: `"{選択肢} {前提条件の技術} issue"`, `"{選択肢} {前提条件の技術} compatibility"` 等で検索し、既知の非互換・制限・ワークアラウンドがないか確認する
- **バージョン制約**: 前提条件の技術の特定バージョンでのみ動作する、または特定バージョンで動作しない等の制約がないか確認する

結果を以下の形式で報告する:
```
## 前提条件との互換性
| 前提条件 | 互換性 | 備考 |
|----------|--------|------|
| {前提条件1} | ✅ 問題なし / ⚠️ 条件付き / ❌ 非互換 | {詳細} |
```

**❌ 非互換が発見された場合**、その旨を明確に報告する。この情報は Step 4 の調査結果統合時にユーザーに提示し、当該選択肢を候補から除外するか判断を仰ぐ。

#### 調査タスク 2: 判断軸に基づく評価

判断軸ごとの評価を、Comparison Overview テーブルのセル記述（1〜2文の要約）として返す。軸外の重要な事実も別途返す。

**報告形式:**
```
## Comparison Overview 用データ
| 判断軸 | 評価 | セル記述（1〜2文） |
|--------|------|-------------------|
| {軸1}  | ◎/○/△ | {要約} |

## Notes 候補
- {テーブルに収まらないが判断に重要な事実: 前提矛盾、SemVer 非準拠、ビルド問題等}
```

調査の深さに関する指針:
- 公式ドキュメントの該当セクションを WebFetch で実際に読む。検索結果のスニペットだけで判断しない
- パフォーマンスに関する主張にはベンチマーク記事や根拠を探す。ただし具体的な数値（GitHub Stars、ベンチマーク数値）よりも相対表現（「約N倍高速」「活発にメンテナンスされている」等）を優先する。具体数値は必須ではない
- プロジェクトの健全性（最終リリース日、メンテナンス状況の傾向）も確認する
- ライセンスの種類と商用利用の可否を確認する

### 調査結果の統合:

各サブエージェントの結果を統合し、以下の形式でユーザーに提示する:

#### 前提条件との互換性（最初に提示する）:

```
**前提条件チェック結果:**
| 前提条件 | 選択肢A | 選択肢B | 選択肢C |
|----------|---------|---------|---------|
| {前提条件1} | ✅ | ✅ | ⚠️ {概要} |
| {前提条件2} | ✅ | ❌ {概要} | ✅ |
```

⚠️ や ❌ がある場合は、詳細を比較表の後に補足する。❌（非互換）の選択肢は候補から除外するかユーザーに確認する。

#### 比較表（判断軸 × 選択肢）:

```
| 判断軸 | 選択肢A | 選択肢B | 選択肢C |
|--------|---------|---------|---------|
| 軸1    | ◎ ... | ○ ... | △ ... |
| 軸2    | ○ ... | ◎ ... | ○ ... |
| 軸3    | △ ... | ○ ... | ◎ ... |
| 軸4    | ○ ... | △ ... | ○ ... |
```

#### Notes（テーブルに収まらない重要な事実）:

各サブエージェントが報告した Notes 候補を統合し、判断に影響する重要な事実をまとめて提示する。該当なしの場合は省略する。

```
**Notes:**
- **{選択肢A} は {前提条件X} と矛盾**: {説明}
- **{選択肢B} の SemVer 非準拠**: {説明}
```

#### 判断ガイド:

比較表と Notes の後に、条件付きの判断ガイドを提示する。LLM が特定の選択肢を推奨するのではなく、「何を優先するかによってどれが適するか」を条件付きで示す。

```
**判断ガイド**（条件付きの帰結であり、推奨ではありません）:
- {軸1}を最優先するなら → {選択肢A} が適している。ただし{軸3}にリスクがある
- {軸2}を最優先するなら → {選択肢B} が適している。ただし{軸4}にリスクがある
- バランスを取るなら → {選択肢C}。ただし{留意点}
```

「〜を推奨します」「〜がベストです」等の単一推奨表現は使わない。

提示後、ユーザーに以下を確認する:
- 事実誤認の指摘
- 追加調査の依頼
- 問題なければ次のステップへ

## Step 5: ユーザーによる判断

比較結果を踏まえ、ユーザーに選択を委ねる:

「比較結果は以上です。どの選択肢を採用しますか？以下を教えてください:
1. 選択した選択肢
2. 選択の理由（どの判断軸を重視したか）
3. 受け入れるトレードオフ（この決定により犠牲にすること・受け入れるリスク）」

LLM は判断を**しない**。材料提供に徹し、ユーザーの決定を記録する。

## Step 6: 収集内容の確認

ADR を生成する前に、収集した情報を一覧で提示してユーザーに確認する。

**Consequences、Notes、More Information は LLM が主体的に提案する。** ユーザーに「追加しますか？」と聞くのではなく、比較調査の内容やユーザーの選択理由から推測してドラフトし、確認時に含める。ユーザーが不要と判断すれば削除指示を受ける。

確認は逆ピラミッド順（決定内容を先頭に）で提示する:

```
ADR の内容を確認します:

**タイトル**: {何を決めるかの問い}
**Decision Outcome**: {採用技術・方式} — {理由の要約 1〜2 文}
**Accepted Tradeoffs**: {受け入れるリスク・犠牲}
**Consequences（提案）**: {ポジティブ・ネガティブな結果}
**背景・課題**: {要約 1〜2 文}
**前提条件**: {前提条件のリスト、または「なし」}
**判断軸**: {4つの判断軸（名前のみ）}
**検討した選択肢**: {テーブル形式: 選択肢 | 概要}
**除外した選択肢**: {名前（理由）、または「なし」}
**Comparison Overview**: {判断軸×選択肢テーブル}
**Notes（提案）**: {テーブルに収まらない重要な事実、または「なし」}
**補足情報（提案）**: {関連リンクや参考資料、なければ「なし」}

この内容で ADR を生成してよいですか？
修正・追加・削除があれば教えてください。
```

ユーザーが修正を指示した場合は内容を更新し、再度確認する。承認が得られたら次のステップに進む。

## Step 7: ADR ファイルを生成

ファイル名を構築する: `{NNNN}-{kebab-case-title}.md`
- `NNNN` はゼロ埋め連番
- タイトルは短く説明的な kebab-case（例: `use-sqlite-for-local-storage`）

frontmatter の `date` には当日の日付を使用する。`status` は `"accepted"` に設定する（このプロジェクトではproposed状態を使わない）。`last-validated` には `date` と同じ日付を設定する。

### テンプレート

ADR の本文は日本語で記述する。ただし以下の構文は英語のまま維持する:
- frontmatter のキーと値（`status: "accepted"`, `date:`, `last-validated:`）
- セクション見出し（`## Decision Outcome`, `## Context and Problem Statement` など）

**簡潔さが最重要。** ADR は短く要点を押さえたものにする。逆ピラミッド構成（最も重要な情報を冒頭に）を採用し、Decision Outcome をタイトル直後に配置する。

```markdown
---
status: "accepted"          # accepted | implemented | withdrawn
date: {YYYY-MM-DD}          # 作成日（不変）
last-validated: {YYYY-MM-DD} # 最終検証日（作成時はdateと同じ）
---

# {何を決めるかの問い}

## Decision Outcome

{採用技術・方式と、その選択理由を散文で記述する。どの判断軸を重視し、なぜその優先順位にしたかを含める。}

### Accepted Tradeoffs

{この決定により受け入れるリスク・犠牲を箇条書きで明示する。}

- {受け入れるトレードオフ1}

### Consequences

{この決定がプロジェクトにもたらすポジティブ・ネガティブな結果を箇条書きで記述する。Decision Outcome と重複する内容は書かない。}

- {結果1}
- {結果2}

## Context and Problem Statement

{背景と課題の説明。スコープと対象外を含める。}

## Prerequisites

- {前提条件1、例: "BFF: Fastify (TypeScript)、バックエンド: Go (Echo) (ADR-0001)"}
- {前提条件2}

## Decision Drivers

- {判断軸1}
- {判断軸2}
- {判断軸3}
- {判断軸4}

## Considered Options

| 選択肢 | 概要 |
|--------|------|
| **{選択肢1}（採用）** | {1行の説明} |
| {選択肢2} | {1行の説明} |
| {選択肢3} | {1行の説明} |

除外: {選択肢X}（{理由}）、{選択肢Y}（{理由}）

## Comparison Overview

| 判断軸 | {選択肢1} | {選択肢2} | {選択肢3} |
|--------|:-:|:-:|:-:|
| {軸1}  | ◎ {要約} | ○ {要約} | △ {要約} |
| {軸2}  | ○ {要約} | ◎ {要約} | ○ {要約} |
| {軸3}  | △ {要約} | ○ {要約} | ◎ {要約} |
| {軸4}  | ○ {要約} | △ {要約} | ○ {要約} |

## Notes

{Comparison Overview テーブルに収まらないが判断に重要だった事実を記録する。前提矛盾、SemVer 非準拠、ビルド問題、バージョン制約など。}

- **{選択肢X} は {前提} と矛盾**: {説明}
- **{選択肢Y} の {問題}**: {説明}

## More Information

- [{リンクテキスト}]({URL})
- {補足事実}
```

### セクション包含ルール:
- **常に含める**: Decision Outcome (Accepted Tradeoffs, Consequences を含む), Context and Problem Statement, Prerequisites, Decision Drivers, Considered Options (除外を含む), Comparison Overview
- **LLM が提案し、ユーザーが削除指示しなかった場合に含める**: Notes, More Information
- ユーザーが削除を指示した場合は**セクションごと省略**する — プレースホルダーテキストは残さない
- 前提条件がない場合は Prerequisites セクションに「なし」と記載する — 「前提条件なし」と明記することに情報価値がある
- 除外した選択肢がない場合は「除外:」行に「なし」と記載する
- Decision Drivers には承認済みの判断軸を名前のみ記載する（説明文は付けない）
- Notes は判断に重要な事実がある場合のみ含める。全選択肢に問題がない場合は省略してよい

## Step 8: 書き出し

1. `docs/adr/{NNNN}-{title}.md` にファイルを書き出す
2. **再評価モードからの supersede の場合**: 旧 ADR の frontmatter を更新する
   - `status` を `"withdrawn"` に変更
   - `superseded-by: "NNNN"` を追加（NNNN は新 ADR の番号）
   - 旧 ADR の本文は一切変更しない
3. ファイルパスを伝える。追加の確認は行わず、ここで完了とする

## チェックモード

`/adr check` または「チェック」が選択された場合のフロー。

`bash .claude/skills/adr/scripts/adr-freshness-check.sh` を Bash で実行し、結果をそのまま表示する。

スクリプトは以下を行う:
- `docs/adr/*.md` の frontmatter から `status`, `last-validated`（フォールバック: `date`）, `review-interval` を抽出
- `status: withdrawn` はスキップ
- `last-validated` から今日までの経過日数を計算し、閾値（`review-interval` またはデフォルト 90 日）を超えた ADR を `STALE` として報告
- STALE な ADR がある場合は exit 1

STALE な ADR が見つかった場合、ユーザーに `/adr re-evaluate NNNN` での再評価を案内する。

---

## 再評価モード

`/adr re-evaluate NNNN` または「再評価」「supersede」が選択された場合のフロー。supersede もこのフローに統合されている。

### Re-evaluate Step 1: 対象ADRの読み込み

対象 ADR（番号 NNNN）を `docs/adr/` から Read で読み込む。以下を把握する:
- Decision Outcome で選択した技術
- Decision Drivers（判断軸）
- Accepted Tradeoffs

### Re-evaluate Step 2: 簡易チェック（2エージェント並列）

Agent ツール（subagent_type=general-purpose）を **2つ並列**で起動する。

**エージェント A: 新たな選択肢の検索**
- ADR のトピック領域で、ADR 作成後に登場した有力な代替候補がないか WebSearch で調査する
- 検索例: `"{トピック} alternative {現在の年}"`, `"{採用技術} vs"`, `"{トピック} new {現在の年}"`
- 見つかった候補を簡潔にリストアップして返す（候補なしの場合はその旨を返す）

**エージェント B: 採用技術の変更チェック**
- 採用した技術に、判断を覆しうる変更がないか WebSearch で調査する
- チェック観点: deprecation、メジャーバージョン変更による破壊的変更、重大なセキュリティ問題、メンテナンス停止
- 検索例: `"{採用技術} deprecated"`, `"{採用技術} breaking changes {現在の年}"`, `"{採用技術} security vulnerability"`
- 発見事項を簡潔にリストアップして返す（問題なしの場合はその旨を返す）

### Re-evaluate Step 3: 結果提示と判断

2つのエージェントの結果を統合し、以下の形式で報告する:

```
## ADR-NNNN 再評価レポート

### 新たな選択肢
- {見つかった候補、またはなし}

### 採用技術の変更
- {発見事項、または問題なし}

### 判断
上記を踏まえ、以下のいずれかを選んでください:
1. **現状維持** — last-validated を更新して終了
2. **supersede** — 新しい ADR を作成して置き換える
```

### Re-evaluate Step 4: 分岐

**現状維持の場合:**
- 既存 ADR の `last-validated` を本日の日付に更新する
- Change Log に再評価した旨を記録する
- ここで完了

**supersede の場合:**
1. 旧 ADR の Context and Problem Statement、Prerequisites、Decision Drivers を引き継ぎの土台にする
2. 簡易チェックの結果を Step 3 のフレームワーク提案の入力として活用する（新たな選択肢が見つかっていれば候補に含める）
3. ADR 作成フロー（Step 2〜8）に入る。Step 8 で旧 ADR の frontmatter を更新する（`status: "withdrawn"` + `superseded-by: "NNNN"`）

## 提案モード

`/adr propose` または「提案」が選択された場合のフロー。

### Propose Step 1: サブエージェントによる収集・分析

`.claude/skills/adr/agents/proposal-collector.md` を Read で読み込み、その内容をプロンプトとして Agent ツール（subagent_type=general-purpose）を起動する。サブエージェントがコンテキスト収集・候補列挙・フィルタリング・構造化を行い、結果を返す。

### Propose Step 2: 提案結果の提示

サブエージェントの結果をそのままユーザーに提示し、以下を案内する:

```
作成する ADR を選んでください。選択した候補の内容で ADR 作成フローに入ります。
内容の修正・候補の追加・質問があれば教えてください。
```

### Propose Step 3: ADR 作成フローへの接続

ユーザーが候補を選択したら、通常フロー（Step 2 以降）に接続する。その際、提案フォーマットの情報を以下のようにマッピングして引き継ぐ:

| 提案フォーマット | ADR 作成フローでの活用先 |
|---|---|
| 決めるべきこと | Step 1 のトピック（確定済みとして扱う） |
| 前提 | Step 3b の前提条件（Prerequisites）のドラフト入力 |
| 前提 ADR（依存先） | Step 3b の前提条件に含める |
| スコープ | Step 3b の背景・課題（Context）のスコープ記述に活用 |
| スコープ外 | Step 3b の背景・課題で「この ADR では扱わない」として明記 |
| 制約条件 | Step 3b の前提条件、および Step 3a の選択肢調査時のフィルタ条件 |
| 影響範囲 | Step 3b の背景・課題に含める |

**重要**: 提案フォーマットの情報は Step 3 のドラフトの**入力**として使う。Step 3 では引き続きユーザーの承認を得るプロセスを経る — 提案情報があるからといってユーザー確認をスキップしない。
