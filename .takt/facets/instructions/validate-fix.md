修正が根本原因に対処しているか、バンドエイド（表面的な修正）になっていないかを検証する。
また、修正の振り子パターン（オシレーション）を検出する。

## バリデーション

### 1. 修正内容の確認

`git diff` で直近の修正内容を確認し、トリアージで accept された各指摘に対して:
- 修正が指摘の根本原因に対処しているか
- ワークアラウンド（条件分岐の追加、エラー握りつぶし等）で問題を隠していないか
- 修正が新たな問題を導入していないか

### 2. オシレーション検出

`git log --oneline -10` で直近のコミット履歴を確認する。
`git diff HEAD~2..HEAD~1` と `git diff HEAD~1..HEAD` を比較し、
以前の変更を打ち消す「A → B → A」パターンが検出された場合:

1. 両バージョン（A と B）のコードを比較し、どちらが優れているか判定する
2. 劣った方の変更を revert する directive を作成する
3. `{report:oscillation-directives.md}` に累積的に追記する:

```markdown
## Directive {番号}
- ファイル: {file}
- 固定するバージョン: {A or B}
- 理由: {判定理由}
- 禁止する変更: {具体的に何をしてはいけないか}
```

この directive は後続の fix / implement ムーブメントが参照し、同じ振り子を回避する。

## 判定

- 全修正が根本対策 → 「修正が根本対策、次のレビューサイクルへ」
- バンドエイド修正が含まれる → 「バンドエイド修正、再修正が必要」（具体的にどの修正がバンドエイドか明記）
