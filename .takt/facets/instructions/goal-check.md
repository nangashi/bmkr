受け入れ条件の充足確認、スコープチェック、PR 準備を行う。

## 1. 受け入れ条件トレーサビリティ

`{report:init-context.md}` から Issue 本文を読み、受け入れ条件を抽出する。
各条件について対応コード・テストを特定する:

| 受け入れ条件 | 対応コード（ファイル:関数） | 対応テスト（ファイル:テスト名） | 状態 |
|-------------|--------------------------|------------------------------|------|
| 条件1 | handler.go:HandleX | handler_test.go:TestX | OK / 未対応 / 要確認 |

## 2. スコープチェック

- Issue の「スコープ外」に記載された内容に触れていないか
- 計画にない追加機能・リファクタリングが含まれていないか
- 受け入れ条件の独自解釈がないか

## 3. 受け入れ条件チェックボックス更新

充足した受け入れ条件のチェックボックスを `- [x]` に更新する。
一時ファイルに本文を書き出して `gh issue edit {番号} --body-file` を使う。

## 4. PR body の準備

以下の情報をレポートに書き出す（TAKT が PR 作成時に使用）:

### Design Decisions セクション

情報源:
1. Issue コメント（`<!-- issue-plan:approach:done -->`）の設計判断テーブル
2. `{report:interface-design-decisions.md}`（存在する場合）
3. `{report:implement-decisions.md}`（存在する場合）

## 判定

- 全受け入れ条件が OK → 「全受け入れ条件を充足」
- 未対応・要確認がある → 「未対応の受け入れ条件あり」（具体的にどの条件が未対応か明記）
