# テスト過剰生成

LLM は「テストは多いほど良い」と考え、ROI の低いテストを大量に生成する傾向がある。

## テスト要否の判断: 4象限モデル

コードを「複雑さ/ドメイン重要度」と「協力者（外部依存）の数」で4象限に分類し、象限ごとにテスト戦略を決める（Vladimir Khorikov）。

```
                  協力者が少ない              協力者が多い
               ┌─────────────────────┬─────────────────────┐
  複雑/        │ ドメインモデル &      │ 過度に複雑なコード    │
  ドメイン重要  │ アルゴリズム          │                      │
               │ → UT必須（最高ROI）   │ → まず分割してから    │
               ├─────────────────────┼─────────────────────┤
  単純/        │ トリビアルコード       │ コントローラ          │
  ドメイン非重要│ → テスト不要          │ → 統合テストで薄く    │
               └─────────────────────┴─────────────────────┘
```

| 象限 | 具体例 | テスト戦略 |
|------|--------|-----------|
| ドメインモデル & アルゴリズム | 価格計算、バリデーション、在庫ロジック | UT必須。エッジケースを網羅 |
| トリビアルコード | 単純な代入、1行の委譲、定数定義 | テスト不要。compile + lint で担保 |
| コントローラ | HTTP ハンドラ（委譲のみ）、オーケストレーション層 | 統合テストで 1 happy path。個別 UT は不要 |
| 過度に複雑なコード | ビジネスロジック + 外部依存が混在 | ドメインロジックを抽出して分割してからテスト |

**判断の問い**: 「このコードにバグが入ったとき、既存の compile + lint + 統合テストで検出できるか？」→ Yes ならUT不要。

## テストを書かない正当な理由

| 理由 | 根拠 | 代替手段 |
|------|------|----------|
| トリビアルコード | 回帰リスクがゼロに近い | compile + lint |
| モックコストが過大 | モック基盤 > アサーション（go-test.md ルール1） | 静的解析テスト、I/F抽出後にテスト |
| 変更量が微小 | 数行の変更に数百行のテスト基盤は不合理（go-test.md ルール3） | 既存テスト + build + lint |

## アンチパターン 1: 実装追従テスト

LLM が実装後にテストを書くと、既存の実装（バグ含む）を「正」として追認するテストになる。

```go
// Bad: 実装を見てからテストを書いた → 実装のバグをそのまま期待値にする
func TestCalcTotal(t *testing.T) {
    got := CalcTotal(100, 3)
    want := 330 // 実装が tax を二重計算するバグを持っていても、実装の出力をそのまま期待値にしてしまう
    assert(got, want)
}
```

**対策**: テスト先行（TDD）で書く。実装のコンテキストが入る前に、仕様から期待値を決める。

## アンチパターン 2: Assertion Roulette

LLM は1つのテスト関数に無関係な複数のアサーションを詰め込む傾向がある。失敗時にどの振る舞いが壊れたか特定できない。

```go
// Bad: 1テストに複数の独立した振る舞いを検証
func TestProductHandler(t *testing.T) {
    // 正常系
    resp := handler.List(ctx)
    assert(resp.Status, 200)
    assert(len(resp.Products), 5)
    // バリデーション（別の振る舞い）
    resp2 := handler.Create(ctx, invalidInput)
    assert(resp2.Status, 400)
    // 削除（また別の振る舞い）
    resp3 := handler.Delete(ctx, id)
    assert(resp3.Status, 204)
}

// Good: 振る舞い単位でテストを分ける
func TestProductHandler_List(t *testing.T) { ... }
func TestProductHandler_Create_InvalidInput(t *testing.T) { ... }
func TestProductHandler_Delete(t *testing.T) { ... }
```

**判断基準**: 同一の振る舞いに対する複数アサーション（ステータスコード + レスポンスボディの検証等）は1テストにまとめてよい。異なる振る舞い（正常系 vs 異常系、異なる操作）は分ける。
