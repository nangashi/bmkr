# 実装アンチパターン

LLM が生成するコードで繰り返し発生する実装上の問題と、その正しい対処法。

## LLM 傾向: 不要な追加

LLM は「良いコード = 抽象化・防御・外部化されたコード」と考え、不要なものを追加する傾向がある。

### 過剰抽象化

LLM は「良いコード = 抽象化されたコード」と考える傾向がある。

#### アンチパターン

- 1つの実装しかないのにインターフェースを定義する（テスタビリティ目的の DI 用インターフェースは除く）
- 1箇所でしか呼ばれないヘルパー関数を作る
- 将来の拡張性のためにデザインパターンを導入する

#### 正しいパターン

- 3回の類似コード > premature abstraction（3回繰り返されてから抽象化を検討）
- 標準ライブラリやプロジェクト既存の仕組みで済むなら新しい依存を追加しない
- 同じ要件を満たすもっとシンプルな方法がないか常に検討する

#### 判断基準

「この抽象化を外したら何が壊れるか？」を問う。答えが「何も壊れない」なら、その抽象化は不要。

### 設定の過剰外部化

LLM は値をハードコードすることを「悪い習慣」と判断し、何でも外部化しようとする傾向がある。

#### アンチパターン

- 固定で良い値を環境変数や設定ファイルに切り出す
- 変更される見込みのない値に定数定義を導入する
- 設定注入のためだけに構造体やコンストラクタを追加する

#### 正しいパターン

- 実行環境によって変わる値（DB接続先、APIキー等）だけを外部化する
- アプリケーション仕様として固定の値（ページサイズ20件、タイムアウト30秒等）はコード内にリテラルで書く
- 将来変わるかもしれない、ではなく、今変わる必要があるかで判断する

### 形式的エラーハンドリング

LLM は「エラーハンドリングは多いほど良い」と考え、意味のない処理を追加する傾向がある。

#### アンチパターン

**Go:**
- 意味のない `fmt.Errorf("failed to X: %w", err)` の連鎖（文脈情報を追加しない wrap）
- 呼び出し元で必ず処理されるエラーに対する冗長な nil チェック
- 到達しないエラーパスの処理

**TypeScript:**
- catch して re-throw するだけの try-catch
- 意味のない `.catch(err => { throw err })`
- フレームワークが処理するエラーの手動ハンドリング

#### 正しいパターン

- エラーを wrap するなら、呼び出し元が判断に使う文脈情報を追加する
- 既存コードのエラーハンドリングパターンを読んでそれに従う
- フレームワーク（Echo, Fastify）のエラーハンドリング機構を信頼する

#### エラー情報を握り潰さない

形式的ハンドリングの逆パターンとして、エラーを汎用メッセージに置き換えて元の情報を失うケースもある。

```go
// Bad: 元のエラーが失われ、ログやデバッグで原因追跡できない
return echo.NewHTTPError(http.StatusInternalServerError, "failed to fetch products")

// Good: 元のエラーを保持する（Echo がログに記録する）
return echo.NewHTTPError(http.StatusInternalServerError, err)
```

「意味のない wrap を減らす」と「エラー情報を保持する」は両立する。不要な中間層を減らしつつ、最終的なエラーレスポンスでは元の情報を渡す。

## ロジックバグ

### read → write path 変更時の入力検証漏れ

#### アンチパターン

read-only だったハンドラ（SELECT のみ）に書き込み処理（INSERT / UPDATE / DELETE）を追加する際、入力検証を追加しない。read-only 時代は不正な入力でもデータを汚染しなかったが、write path になると不正データが永続化される。

#### 正しいパターン

ハンドラに書き込み処理を追加する場合、同じリソースを操作する他のハンドラと同等の入力検証が存在するか確認する。なければ追加する。

#### 確認手順

1. 変更対象のハンドラが新たに INSERT / UPDATE / DELETE を行うか確認
2. 同じリソース（テーブル）を操作する他のハンドラの入力検証を確認
3. 同等の検証がなければ追加する

### 状態変更後のデータ再取得

#### アンチパターン

状態を変更（クランプ、正規化、更新等）した後、変更前のデータをそのまま使う。典型例: ページ番号をクランプしたが、クランプ前のページ番号で取得したデータを返す。

#### 正しいパターン

状態を変更したら、変更後の値でデータを再取得する。

#### 具体例

```go
// Bad: クランプ後に再フェッチしない
products, _ := h.q.ListProducts(ctx, db.ListProductsParams{Offset: offset(page)})
if page > totalPages {
    page = totalPages // クランプしたが products は古い page で取得済み
}

// Good: クランプ後に再フェッチ
if page > totalPages {
    page = totalPages
    products, _ = h.q.ListProducts(ctx, db.ListProductsParams{Offset: offset(page)})
}
```
