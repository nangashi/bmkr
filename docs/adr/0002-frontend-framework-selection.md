# ADR-0002: フロントエンド技術スタックの選定

## ステータス

Proposed

## 日付

2026-02-26

## コンテキスト

クロスデバイス対応リーディングリスト管理ツールの開発にあたり、フロントエンド技術スタックを選定する必要が生じた。本決定では、フルスタックフレームワーク（Next.js/Nuxt/SvelteKit等）、JavaScriptランタイム（Node.js/Deno/Bun）、パッケージマネージャ（npm/pnpm/yarn/bun）の組み合わせを対象とする。

スコープ対象はフルスタックフレームワーク・ランタイム・パッケージマネージャの選定のみであり、UIコンポーネントライブラリ・CSSフレームワーク・データベース選定・認証方式の詳細設計・CI/CDパイプラインは対象外とする。

フルスタックフレームワークの変更はアプリケーション全体の書き換えを伴い、コンポーネント構造・ルーティング・データフェッチ・ビルド設定のすべてに影響するため、変更コストが高い。この決定の可逆性は低い（decision-statement.md 表記では"high"＝変更コストが高い）。

## 関連する既存決定

- **ADR-0001: ホスティングプラットフォームの選定** — Netlify（Starterプラン）を採用。標準Node.jsランタイムを採用しており、プラットフォーム固有APIへの暗黙的な誘導が最も少ない。フロントエンドフレームワークのSSR対応・ランタイム互換性・デプロイ方式に直接影響し、主要フルスタックフレームワーク（Next.js / Nuxt / SvelteKit / Remix）の選定自由度を制限しない。認証ライブラリの互換性制約もほぼ生じさせない。

## 前提条件

### 確実（Certainty）

- C1: ホスティングプラットフォームは Netlify（Starter プラン、標準 Node.js ランタイム）に確定済みである — 根拠: ADR-0001 で決定済み。変更には ADR の正式改訂が必要
- C2: 採用するフレームワーク・ランタイムは Node.js 互換性を持つこと — 根拠: ADR-0001 の制約。Netlify Starter は標準 Node.js ランタイムを使用。Deno・Bun は Node.js 互換モードで動作することを前提とする
- C3: npm / pnpm / yarn / bun のいずれのパッケージマネージャも Netlify 上で利用可能 — 根拠: ADR-0001 の調査済み事項。Netlify の標準 Node.js ランタイム環境で全て動作確認済み
- C4: Yarn Classic（v1）は新規プロジェクトへの採用が非推奨である — 根拠: Yarn Classic はメンテナンスモードに移行しており、重大修正のみ受け付ける状態（2026年時点のリサーチ結果）
- C5: アプリケーションはクロスデバイス（PC・スマートフォン）対応が必要である。PWA対応可能であること（Android共有メニュー連携 IR-001） — 根拠: ユーザーが明示した制約。problem-statement.md の保存方法制約「スマホ: Androidの共有機能でURLを送信」に対応
- C6: TypeScript を採用する — 根拠: ユーザーが明示した制約「TypeScript必須」
- C7: SSR/SSG 対応が必要である — 根拠: ユーザーが明示した制約「SSR/SSG対応」
- C8: Web アプリケーション（ブラウザ動作）として実装する — 根拠: スコープがフルスタック Web フレームワーク（Next.js/Nuxt/SvelteKit等）に限定されており、ネイティブアプリアプローチは除外されている
- C9: 単一フレームワークを選定する（フレームワーク混在は対象外）— 根拠: 「どの組み合わせを採用するか」という議題の形式から、複数フレームワークの併用は選択肢に含まれない

### 仮定（Supposition）

- S1: 現在の開発者は JavaScript / TypeScript の基本的なエコシステム知識を持つ — 根拠: リーディングリスト管理ツールの自作開発を前提としており、最低限の JS/TS 知識は前提と考えられる。検証方法: 開発者のスキルセットを確認する
- S2: プロジェクトは個人開発（1名）を想定している — 根拠: problem-statement.md のステークホルダーが「自分自身（個人開発者・利用者）」のみ。Netlify Starter プランの選定もこれと整合。検証方法: 開発体制を確認する
- S3: チームメンバーの採用拡大は短中期的に計画されていない — 根拠: S2 から導出。個人ツールの性質から拡張の可能性は低い。検証方法: 開発ロードマップを確認する
- S4: アプリケーションのトラフィックは低程度であり、高負荷対応は不要 — 根拠: problem-statement.md でデータライフサイクルが「月20件程度の保存を想定」。シングルユーザー前提。検証方法: ユーザー数の見込みを確認する
- S5: Next.js / Nuxt / SvelteKit のいずれを採用しても Netlify 上で SSR が動作する — 根拠: ADR-0001 時点で各フレームワークの Netlify アダプターの存在を確認済み。ただし ADR-0001 時点のスナップショット情報であり、アダプターのメンテナンス継続は変化しうる。特に SvelteKit はコミュニティ規模が相対的に小さく、リスクがやや高い。検証方法: 選定直前に各フレームワークの Netlify アダプター最新状態を確認する（ADR-0001 時点のCからSへ再分類）

### 不確実（Doubt）

- D1: 開発者が React / Vue / Svelte のどのフレームワーク経験を持つか — 影響範囲: フレームワーク選定に直接影響。学習コストが低いフレームワークを選ぶべきか、技術的な優位性を優先すべきかの判断が変わる。リスク: 不慣れなフレームワークを選択した場合、開発速度の大幅な低下・品質リスクが生じうる
- D2: アプリケーションのオフライン動作（完全オフライン）が要件か、それとも「クロスデバイス同期」が主目的か — 影響範囲: PWA のサービスワーカー要件の複雑さに影響。完全オフライン対応ではキャッシュ戦略・同期ロジックが複雑になり、フレームワークのサービスワーカー統合の成熟度が重要になる。リスク: オフライン要件が後から明確化された場合、実装コストが跳ね上がる可能性がある
- D3: モノレポ構成（将来的な API サーバー分離等）を想定するか — 影響範囲: パッケージマネージャの選定に影響。モノレポを想定する場合、pnpm の優位性が高まる。単一リポジトリであれば影響は小さい。リスク: 後からモノレポに移行する場合、パッケージマネージャの変更コストが発生する
- D4: エコシステムの成熟度と開発速度のどちらを優先するか — 影響範囲: フレームワーク選定の重みづけが変わる。成熟度優先なら Next.js/Nuxt、開発速度・バンドルサイズ優先なら SvelteKit が有利。リスク: 優先度の不一致により選定後の満足度が低下する

## 判断基準

優先度は決定者（人間）が設定する。

- **OBJ-1: 開発継続性の確保** — 個人開発者が長期にわたって開発・保守を継続できる環境を確保すること。評価の視点: フレームワークのドキュメント品質と日本語リソースの充実度、コミュニティの活発さ（質問への回答が得やすいか）、メジャーバージョン更新頻度と破壊的変更の頻度、採用企業・プロジェクト数
- **OBJ-2: 要件実現の確実性** — 必須要件（TypeScript・SSR/SSG・PWA・クロスデバイス対応）を追加の複雑性なく実現できること。評価の視点: TypeScript サポートの成熟度、SSR/SSG の実装容易さと Netlify 上での動作確実性（S5 に依存）、PWA の公式サポートまたは成熟したプラグインの存在、Netlify アダプターのメンテナンス状況
- **OBJ-3: 開発効率の最大化** — 個人開発者がアプリケーション固有のロジックに集中できるよう、定型作業・ボイラープレートを最小化すること。評価の視点: プロジェクト初期セットアップの容易さ、ファイルベースルーティングなどの規約による定型コード削減、開発体験（DX）の質、バンドルサイズと実行時パフォーマンスの最適化の自動化度
- **OBJ-4: 依存リスクの最小化** — 選定した技術スタックが将来的に廃止・非互換化・プラットフォーム制約によって使用不能になるリスクを最小化すること。評価の視点: ランタイム・パッケージマネージャの Netlify 標準サポート状況、フレームワークのメンテナンス主体とその継続性、Node.js 互換性要件との整合性、フレームワーク変更時の移行コスト
- **OBJ-5: 運用負荷の最小化** — ビルド・デプロイ・依存パッケージ管理において、個人開発者が継続的にメンテナンスできる負担水準に収めること。評価の視点: パッケージマネージャのロックファイル管理・依存解決の信頼性、Netlify への CI/CD 連携の容易さ、セキュリティアップデートの追跡容易性、モノレポ移行の将来的な容易さ（D3 に対応）

## 検討した選択肢

### ALT-1: Next.js + Node.js + pnpm

Reactベースのフルスタックフレームワーク。Vercel社が主導し最大のエコシステムを持つ。App RouterによるSSR/SSG/ISR、TypeScriptファースト設計、PWA公式ドキュメント（2024年秋）を備える。pnpmによる依存管理とD3（モノレポ）対応が可能。OBJ-1（開発継続性）、OBJ-2（要件実現）、OBJ-4（依存リスク）の観点から最優位として提案。

### ALT-2: Nuxt + Node.js + pnpm

Vueベースのフルスタックフレームワーク。NuxtLabsが主導。自動インポート・規約ベース開発による定型作業削減が特徴。@vite-pwa/nuxtによるゼロコンフィグPWA対応。pnpmによる依存管理。OBJ-3（開発効率）、OBJ-5（運用負荷）のバランス型として提案。

### ALT-3: SvelteKit + Node.js + pnpm

Svelteベースのフルスタックフレームワーク。コンパイラ方式でバンドルサイズが40〜60%小さく、記述量が少ない。vite-pwa/sveltekitによるPWA対応。pnpmによる依存管理。OBJ-3（開発効率）、OBJ-5（運用負荷）で最優位として提案。

### 除外した選択肢

- **SvelteKit + Bun + pnpm**: Bunランタイムの本番安定性が不十分（C2: Node.js互換性要件に対してリスクが高い）
- **Bun（ランタイム全般）**: C2の制約。Netlify標準Node.jsランタイムとの整合性が不確実。本番採用実績がNode.jsに劣る
- **Yarn Classic（v1）**: C4により除外（メンテナンスモード）
- **Yarn Berry（v4+）**: 技術的には可能だが、pnpmに対してコミュニティ採用率が低下傾向。PnPモードの互換性問題リスク
- **npm**: 技術的には可能だが、全評価エージェントがpnpmを優先。インストール速度・D3対応力・幻影依存防止でpnpmが優位

### ALT-1 × OBJ-1（◎）
- 評価: ◎
- 利点: Reactは世界最大のフロントエンドコミュニティ。Vercel社の企業サポートにより廃止リスクが最も低い。日本語リソースが圧倒的に豊富
- 欠点: Vercelの方針変更リスク。App Routerの新パラダイムにより学習リソースが新旧混在
- 根拠: GitHub Stars・npmダウンロード数・採用企業数で首位。長期メンテナンスの持続可能性指標が最高
- CSD依存: D1（React経験の有無で学習コストが変動）

### ALT-1 × OBJ-2（◎）
- 評価: ◎
- 利点: TypeScriptファースト設計。PWAが2024年秋に公式ドキュメント化。Netlifyアダプターの実績が最長
- 欠点: App RouterとPages Routerの並存による初期判断コスト
- 根拠: 必須要件すべてを「公式の手順に従うだけで実現できる」状態。Web Share Target APIの採用事例も最多
- CSD依存: S5（Netlifyアダプター継続性、ただしリスク最低）

### ALT-1 × OBJ-3（○）
- 評価: ○
- 利点: create-next-appのCLIが最も成熟。AIアシスタントの学習データが豊富で回答精度が高い
- 欠点: App Router/RSCの概念複雑性。Reactのhooksボイラープレートが他FWより多い。バンドルサイズが大きい
- 根拠: エコシステム・AI支援は優秀だが、リーディングリスト管理ツール規模ではRSCの複雑性が生産性を押し下げる可能性
- CSD依存: D1（React経験の有無）、D4（成熟度vs開発速度の優先度）

### ALT-1 × OBJ-4（◎）
- 評価: ◎
- 利点: Vercel主導で廃止リスク最低。React/Next.jsエコシステムの広さにより代替移行パスが豊富。pnpmでD3対応
- 欠点: Vercel（私企業）への意思決定集中リスク。App Routerの安定性予測がやや困難
- 根拠: フレームワーク廃止リスクが現実的に最も低い。Netlifyアダプターも企業主導で安定
- CSD依存: S5（最もリスクが低い）

### ALT-1 × OBJ-5（○）
- 評価: ○
- 利点: Netlify×Next.jsの事例が最多で設定トラブル時の解決コストが低い。npm auditエコシステムが成熟
- 欠点: バンドルサイズがSvelteKitより大きくCIビルド時間が長い。npmの幻影依存リスク（pnpm採用で軽減）
- 根拠: 「予測可能性」は最高だが、ビルド時間・バンドルサイズでSvelteKitに劣る
- CSD依存: S5（リスク最低）

### ALT-2 × OBJ-1（○）
- 評価: ○
- 利点: Vueエコシステムは世界2位。NuxtLabsの企業サポートあり。破壊的変更の頻度がNext.jsより低い傾向
- 欠点: コミュニティ規模がNext.jsより小さい。採用企業数・深い日本語技術記事が相対的に少ない
- 根拠: 企業バックアップがあり廃止リスクは低いが、Vercel規模には及ばない
- CSD依存: D1（Vue経験の有無）

### ALT-2 × OBJ-2（○）
- 評価: ○
- 利点: TypeScript公式サポート。@vite-pwa/nuxtでゼロコンフィグPWA。SSR/SSGがnuxt.configで一元管理
- 欠点: PWAがサードパーティモジュール依存。Android共有メニュー連携の事例がNext.jsより少ない
- 根拠: 必須要件はモジュールエコシステムで網羅できるが、一部が公式度でNext.jsに劣る
- CSD依存: S5（Nuxt Netlifyアダプター継続性、中程度のリスク）

### ALT-2 × OBJ-3（○）
- 評価: ○
- 利点: 自動インポート機能が強力で定型コード削減。ファイルベースルーティング。ゼロ設定に近い初期セットアップ
- 欠点: Composition APIの学習コスト（D1依存）。モジュールごとの設定の癖。自動インポートの「魔法」的動作
- 根拠: 自動インポートは強力だが、フレームワーク固有概念の習得コストがSvelteKitより高い
- CSD依存: D1（Vue/Nuxt経験の有無）

### ALT-2 × OBJ-4（○）
- 評価: ○
- 利点: NuxtLabsの企業バックアップ。Vue+Viteへの移行パスが存在。pnpmでD3対応
- 欠点: NuxtLabsの規模がVercelより小さい。採用企業数・コミュニティでNext.jsに劣る
- 根拠: 企業バックアップがあるが、Vercel規模には及ばない
- CSD依存: S5（Nuxt Netlifyアダプター、中程度のリスク）

### ALT-2 × OBJ-5（○）
- 評価: ○
- 利点: pnpmの依存管理信頼性。@vite-pwa/nuxtのゼロコンフィグ。自動インポートで管理箇所削減
- 欠点: バンドルサイズがSvelteKitより大きい。Nuxt 4の追従コスト。自動インポートのデバッグ複雑性
- 根拠: pnpmの優位性はSvelteKit+pnpmと共通。フレームワーク側のビルドサイズでSvelteKitに劣る
- CSD依存: S5（中程度のリスク）

### ALT-3 × OBJ-1（△）
- 評価: △
- 利点: コミュニティ急成長中。フレームワーク自体のコンセプトがシンプルで保守しやすい
- 欠点: コミュニティ規模がNext.js・Nuxtより著しく小さい。日本語リソースが少ない。Svelte 5のRunes移行で新旧混在
- 根拠: 採用企業数・コミュニティ活発さで他FWに劣り、長期継続性の外部指標として最も弱い
- CSD依存: D1（Svelte経験があっても日本語リソースの制約は残る）

### ALT-3 × OBJ-2（△）
- 評価: △
- 利点: TypeScriptサポートあり。SSR/SSGは基本機能。vite-pwa/sveltekitでPWA対応可能
- 欠点: PWA公式ドキュメントの充実度不足。Android共有メニュー連携の事例が最も少ない。トラブル時の情報収集困難
- 根拠: 技術的には実現可能だが「追加の複雑性なく」の定義に対して条件付き達成
- CSD依存: S5（アダプター継続リスクが最も高い）、D2（オフライン要件確定時に×に変化の可能性）

### ALT-3 × OBJ-3（◎）
- 評価: ◎
- 利点: ボイラープレート最少（20〜30%削減）。ランタイムなしで自動最適化。ViteベースHMRが即時反映。ゼロコンフィグPWA
- 欠点: エコシステム成熟度が低く、特定ライブラリのアダプターが存在しない場合がある
- 根拠: コンパイラ方式でフレームワーク概念の学習・記述が不要。個人開発（S2）ではチーム採用懸念がなく純粋な生産性向上が得られる
- CSD依存: D1（Svelte未経験時に初期学習コスト。ただし学習曲線はReact/Vueより緩やか）

### ALT-3 × OBJ-4（△）
- 評価: △
- 利点: バニラJSへのコンパイルにより出力成果物の再利用性は高い。pnpmでD3対応
- 欠点: 明確な企業バックアップなし。コミュニティ規模が最小。Netlifyアダプター継続リスクが最高。固有構文（$store、rune）の全面書き換えが必要
- 根拠: 依存リスクの観点で最も不確実性が高い。S5崩壊時のダメージが最大
- CSD依存: S5（リスク最高）

### ALT-3 × OBJ-5（◎）
- 評価: ◎
- 利点: バンドルサイズ40〜60%小でCIビルド時間最短。pnpmの依存管理信頼性。Netlifyアダプターがほぼゼロコンフィグ
- 欠点: アダプター不具合時の解決情報が少ない（コミュニティ規模の制約）
- 根拠: ビルド出力の軽量さとpnpmの組み合わせが日常的な運用負荷を最小化
- CSD依存: S5（アダプター継続リスクがやや高い）

## リスク分析

プレモーテム分析を実施しました（有力選択肢: ALT-1）。

### ALT-1: Next.js + Node.js + pnpm

#### 失敗シナリオ 1: App Router パラダイムの習熟失敗による開発停滞
- 状況: 採用から1年後、Server Componentsの誤用（クライアント状態の誤った扱い、ハイドレーションエラーの頻発）によりデバッグに多大な時間を費やしている
- 原因: React経験のレベルがApp Router以前（Pages Router時代）であった場合、RSC（React Server Components）の概念が従来と大きく異なり、一人開発では詰まった際の助けが得にくい
- CSD関連: D1（開発者のReact経験の有無）— Reactの表面的な知識しかなかった、またはReact未経験だった場合に特に深刻化
- 深刻度: medium
- 発生確率: medium
- 軽減策: 採用前にApp Routerを用いた小規模プロトタイプを2週間以上実施し、RSCの概念習熟を確認する。Pages Routerから始める判断も検討する

#### 失敗シナリオ 2: Vercel（企業）の方針変更による Next.js の方向性不安定化
- 状況: 採用から1年後、VercelがNext.jsのビジネスモデルを変更し、特定の高度機能がVercel有料プラン限定に変更された。Netlifyアダプター経由では利用できない機能が増加し、Next.jsの強みを生かした開発が困難になっている
- 原因: Next.jsはVercelという1社が主導するフレームワークであり、Vercel社の経営状況・方針変更がNext.jsのロードマップに直接影響する
- CSD関連: S5（Netlify上でSSRが動作する）の部分崩壊 — アダプターで基本機能は動作し続けるが、Next.js固有の最適化機能がVercelプラットフォームに縛られる側面が顕在化
- 深刻度: medium
- 発生確率: low
- 軽減策: Next.jsの標準的なSSR/SSG機能のみを使用し、Vercel固有機能（OG Image生成のEdge Runtime等）への依存を避ける。採用前にNetlify上での実際の動作を検証しておく

#### 失敗シナリオ 3: バンドルサイズ肥大化によるPWA・モバイルUX劣化
- 状況: 採用から1年後、機能追加を重ねた結果、バンドルサイズが膨張しスマートフォンからの初回読み込みに3-5秒かかるようになった。Android共有メニュー経由でのURL保存操作が「遅い」と感じられ、ツールとしての価値が低下した
- 原因: evaluation-matrix でALT-1のバンドルサイズがSvelteKitより大きいと指摘されている。個人開発ではパフォーマンス計測・最適化が後回しになりやすい
- CSD関連: S4（低トラフィック、高負荷対応は不要）の誤解 — トラフィックではなく「1ユーザーの体験品質」の問題であり、S4の解釈がモバイル読み込み速度への注意を下げた
- 深刻度: medium
- 発生確率: medium
- 軽減策: 初期から `next/dynamic` による遅延読み込みを徹底し、Lighthouseスコアの定期計測をCI/CDに組み込む。Core Web Vitalsをベースラインとして設定する

#### 失敗シナリオ 4: PWA実装の落とし穴によるAndroid共有メニュー連携の不具合
- 状況: 採用から6ヶ月後、Android共有メニューからのURL受け取り（Web Share Target API）が一部のAndroidバージョンやブラウザで動作しないことが判明した。Next.jsの公式PWAドキュメントはWeb Share Target APIの実装詳細を十分にカバーしておらず、サービスワーカーの設定に多大な試行錯誤を要した
- 原因: C5（PWA対応・Android共有メニュー連携）は確実な制約だが、「Next.jsで追加の複雑性なく実現できる」という評価は楽観的だった可能性がある
- CSD関連: D2（オフライン動作要件の確定）の崩壊 — オフライン動作まで要件として確定した場合、サービスワーカーの複雑性がさらに増大する
- 深刻度: medium
- 発生確率: medium
- 軽減策: 採用前にWeb Share Target APIの最小動作プロトタイプをAndroid実機で検証する。Next.jsのPWAサポートライブラリ（next-pwa等）の最終更新日・メンテナンス状態を確認する

#### 失敗シナリオ 5: App Router の破壊的アップデートによるメンテナンス負荷
- 状況: 採用から1年後、Next.jsのメジャーアップデートでApp Routerに大幅な変更が入り、既存コードの大部分が非推奨化された。一人開発の個人プロジェクトでは、フレームワークのアップデート追従コストが目的と釣り合わなくなり、プロジェクトが放置される
- 原因: Next.jsは比較的頻繁にパラダイムを変える（Pages Router → App Router）歴史を持つ。S3（チーム拡張なし）により、一人でアップデート対応を担い続ける
- CSD関連: S2（個人開発1名）の制約下での持続可能性 — アップデート追従コストを割り引く人的リソースがない
- 深刻度: low
- 発生確率: low
- 軽減策: LTS（Long Term Support）ポリシーを確認し、メジャーバージョンアップを即座に追わない運用方針を確立する。Dependabotのメジャーバージョンアップ通知を「確認するが即対応しない」設定にする

## 将来の意思決定への影響

連鎖分析を実施しました。

### LD-1: UIコンポーネントライブラリ・CSSフレームワークの選定
- 想定時期: 開発開始後 1〜3ヶ月（UI実装フェーズ開始時）
- 影響:
  - ALT-1 (Next.js): 選択肢が広がる — React向けコンポーネントライブラリ（shadcn/ui, MUI, Chakra UI, Radix UI等）の選択肢が最も豊富。styled-components、Tailwind CSS、CSS Modulesのいずれとも相性が良い
  - ALT-2 (Nuxt): やや狭まる — Vue向けライブラリ（Vuetify、PrimeVue、Nuxt UI等）に絞られるが選択肢は十分。自動インポートとの相性を考慮する必要がある
  - ALT-3 (SvelteKit): 狭まる — Svelte専用コンポーネントライブラリ（Skeleton, shadcn-svelte等）は数が少なく成熟度が低い。Tailwind CSS + ヘッドレスUIが現実的な選択肢となる

### LD-2: PWA高度化（バックグラウンド同期・プッシュ通知・オフライン本格対応）の判断
- 想定時期: MVP完成後 6〜12ヶ月
- 影響:
  - ALT-1 (Next.js): 広がる — Service WorkerのカスタマイズやBackground Sync実装の事例・ドキュメントが最も充実。Web Share Target API連携の先行事例が多い
  - ALT-2 (Nuxt): 中立 — 機能的には十分だが、Advanced PWA機能のNuxt固有の事例はNext.jsより少なく情報収集コストが若干高い
  - ALT-3 (SvelteKit): 狭まる — 高度なService Workerカスタマイズの実績・情報が最も少ない。問題発生時のトラブルシューティングが困難

### LD-3: 認証・アクセス制御方式の詳細設計
- 想定時期: 開発開始後 1〜2ヶ月
- 影響:
  - ALT-1 (Next.js): 広がる — NextAuth.js（現Auth.js）が業界標準として確立。OAuth・JWT・セッション認証のすべてにフルサポート。App RouterとのServer Actions統合も整備されつつある
  - ALT-2 (Nuxt): 中立 — nuxt-auth-utils、sidebase/nuxt-authなど選択肢は存在するが、Next.jsほど成熟度・採用実績が高くない
  - ALT-3 (SvelteKit): 中立 — Lucia Auth（SvelteKit向け）が存在し基本機能は網羅。ただし事例・情報量はNext.jsより少なく、エッジケース対応で差が出る可能性がある

### LD-4: フロントエンドのスケールアウト・マイクロフロントエンド化の判断
- 想定時期: 2〜3年後
- 影響:
  - ALT-1 (Next.js): 広がる — Module Federationとの統合実績が最も豊富。企業規模での採用実績が多く、大規模化の設計パターンが確立されている
  - ALT-2 (Nuxt): 中立 — 技術的に可能だが、大規模事例が少なく設計パターンの確立度でNext.jsに劣る
  - ALT-3 (SvelteKit): 狭まる — Svelte固有構文（Runes、$store等）がフレームワーク間統合の障壁になる。企業規模の事例がほとんど存在しない

### LD-5: Netlify以外のホスティング・インフラへの移行判断
- 想定時期: 1〜3年後
- 影響:
  - ALT-1 (Next.js): 広がる — Cloudflare Workers、Fly.io、Railway、AWS Amplify等への移行実績が最も豊富。アダプター層が最も成熟しており移行コストが最小になる可能性が高い
  - ALT-2 (Nuxt): 中立 — アダプターシステムは複数のプラットフォームをサポートしており移行の柔軟性は十分
  - ALT-3 (SvelteKit): やや狭まる — エコシステム規模が最小のため移行時のトラブル情報が少ない。S5崩壊時のダメージが最大

### 可逆性評価

| 代替案 | 可逆性 | 理由 |
|--------|--------|------|
| ALT-1 (Next.js + Node.js + pnpm) | 不可逆 | Reactコンポーネント構造・App RouterのServer Components・データフェッチパターンはNext.js固有であり、他フレームワークへの移行はアプリケーション全体の書き換えを伴う。ただしエコシステムの広さにより移行先の選択肢は最も豊富 |
| ALT-2 (Nuxt + Node.js + pnpm) | 不可逆 | Vue Composition API・Nuxtの自動インポートシステム・ファイルベースルーティングへの依存が深く、他フレームワークへの移行には全面書き換えが必要。Vue→React間のパラダイム差が大きい |
| ALT-3 (SvelteKit + Node.js + pnpm) | 不可逆（かつ移行先制約あり） | Svelte固有構文（Runes、$store、.svelte単一ファイルコンポーネント）は他フレームワークと互換性がなく全面書き換えが必要。加えて移行先での事例・情報量が最も少なく、問題発生時の対応コストが最大になる |

## 決定

ALT-1: Next.js + Node.js + pnpm を採用する。
