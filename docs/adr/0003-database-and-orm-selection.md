# ADR-0003: データベースおよびORM/クエリビルダーの選定

## ステータス

Proposed

## 日付

2026-02-26

## コンテキスト

リーディングリスト管理ツールのバックエンドデータ層として、データベースサービスおよびORM/クエリビルダーを選定する必要がある。

ADR-0001でNetlify（Starterプラン）をホスティングプラットフォームとして採用済みであり、実行環境はNetlifyのサーバーレスNode.jsランタイムに限定される。ADR-0002でNext.js + TypeScript + pnpmを採用済みであり、DBアクセスはServer ActionsまたはRoute Handlersを通じて行われ、TypeScript型安全なアクセスが必須要件となる。

スコープ: データベース（DBサービス）とORM/クエリビルダーの選定を対象とする。ホスティングプラットフォーム（ADR-0001確定済み）、フロントエンドフレームワーク（ADR-0002確定済み）、認証方式の詳細設計、CI/CDパイプラインの構成、UIコンポーネントライブラリは対象外とする。

## 関連する既存決定

- **ADR-0001: ホスティングプラットフォームの選定** — Netlify（Starterプラン）を採用する。NetlifyのサーバーレスNode.jsランタイム上での動作が必須要件となり、コネクション管理においてサーバーレス向けの対応が必要な場合がある。Cloudflare Pages採用の場合に生じたD1/Workers KV等への誘引はなく、DBサービスは技術的評価のみで自由に選定できる。
- **ADR-0002: フロントエンド技術スタックの選定** — Next.js + Node.js + pnpmを採用する。TypeScript必須（C2）・Node.js互換性要件・pnpmによる依存管理が確定しており、ORM/クエリビルダーはこれらすべてに適合している必要がある。

## 前提条件

### 確実（Certainty）

- C1: DBおよびORM/クエリビルダーはNetlifyのサーバーレスNode.jsランタイム上で動作すること — 根拠: ADR-0001確定。Netlifyのホスティング採用が確定しており、実行環境はNetlifyのサーバーレス関数（Node.js）に限定される。
- C2: ORM/クエリビルダーはTypeScript型安全アクセスに対応していること — 根拠: ADR-0002確定。TypeScript必須（ADR-0002 C6）が既定。
- C3: ORM/クエリビルダーはNode.js互換性を持つこと — 根拠: ADR-0002確定（C2の継承）。主要なORM（Prisma、Drizzle、Kysely等）はすべてNode.js対応のため実質的な制約は小さい。
- C4: 導入はpnpmで行うこと — 根拠: ADR-0002確定。実質的な制約は極めて小さい。
- C5: DBアクセスはNext.jsのServer ActionsまたはRoute Handlersを通じて行われること — 根拠: ADR-0002確定。Next.js App Router採用が確定済み。
- C6: Netlifyが最終デプロイ先であること — 根拠: ADR-0001確定。

### 仮定（Supposition）

- S1: このプロジェクトはシングルテーブル・シンプルなCRUD操作が中心である — 根拠: 決定ステートメントの記載およびリーディングリスト管理の性質。検証方法: エンティティ設計・ユースケース一覧を確認。崩れる可能性: タグ管理、ユーザー間共有等の機能追加によりデータモデルが複雑化した場合。
- S2: 個人開発であり、TypeScript・Node.jsスキルは中程度以上 — 根拠: リーディングリスト管理ツールの規模感とADR-0002の技術選定経緯。検証方法: 開発者のスキルセットを確認。崩れる可能性: ORM未経験やDB知識不足の場合、DXの重要度が上がる。
- S3: トラフィックは小規模（個人利用）である — 根拠: リーディングリスト管理ツールの性質、Netlify Starterプラン。検証方法: 想定ユーザー数を確認。崩れる可能性: 商用展開等。
- S4: 月額0円で運用する — 根拠: ユーザー明示の制約。検証方法: 予算要件を確認。崩れる可能性: 商用利用・スタートアップ展開時。
- S5: ORM/データベースクライアントがNetlify関数のバンドルサイズ制限内に収まる — 根拠: Netlifyサーバーレス関数のデフォルト50MB上限（最大250MB）。検証方法: 候補ORMのバンドルサイズを確認。崩れる可能性: Prisma等の大型バイナリを含むORMで制限に抵触する場合。
- S6: マルチユーザー対応が不要（シングルユーザー前提） — 根拠: problem-statement.mdのステークホルダーが「自分自身」のみ。認証はアクセス制御として別レイヤーで解決。検証方法: 要件定義のアクセスモデルを確認。崩れる可能性: 共有機能追加時にRLS等が必要になる場合。

### 不確実（Doubt）

- D1: Netlify環境でのSQLiteの実際の動作保証 — 影響範囲: SQLite系（Turso等）を選択した場合の技術的実現可能性。リスク: NetlifyサーバーレスのエフェメラルファイルシステムでのSQLiteファイル動作は未確認。ただしTurso等のリモートSQLiteサービスはHTTPベースのためファイルシステム制約の影響を受けない可能性がある。
- D2: サーバーレス環境でのコネクションプーリング要件の厳しさ — 影響範囲: PostgreSQL系の選択において接続プーリング対応の要否に影響。リスク: Netlify関数の同時実行数・コールドスタート頻度によりコネクションプール枯渇の可能性。Neon等はHTTPベース接続を提供しており軽減策がある。
- D3: Drizzle ORM v1.0-beta段階での本番利用適合性 — 影響範囲: Drizzleを選択した場合の将来的な安定性リスク。リスク: ベータ版のため破壊的変更の可能性。ただし多くのプロダクションで利用実績があり、実質リスクは限定的とも言える。

## 判断基準

優先度は決定者（人間）が設定する。以下は判断基準の定義のみ示す。

- **OBJ-1: サーバーレス環境への適合性** — Netlifyのサーバーレス（エフェメラル）実行環境上で、安定・確実に動作すること。コネクション管理・バンドルサイズ・ファイルシステム依存の観点でサーバーレスの制約に適合していること。評価の視点: サーバーレス関数でのコネクション管理方式（コネクションプール枯渇リスクの有無）、バンドルサイズがNetlify制限（デフォルト50MB、最大250MB）の範囲内に収まるか、エフェメラルファイルシステムへの依存がないか、Netlify環境での動作実績・公式サポートの有無。
- **OBJ-2: 無料枠での継続運用可能性** — 月額0円の制約のもとで、個人利用規模のトラフィックを継続的に処理できること。評価の視点: 無料プランのストレージ上限、無料プランのリクエスト数・接続数制限、将来的なプランアップグレードが必要になる閾値の高さ、管理費用（セルフホストの場合のインフラ維持コスト）の有無。
- **OBJ-3: TypeScript型安全アクセス** — TypeScriptの型システムを活用して、コンパイル時にデータアクセスの誤りを検出できること。評価の視点: スキーマから型の自動生成・推論が可能か、クエリ結果の型が静的に決定されるか（any型・キャスト不要か）、スキーマ変更時にコンパイルエラーで影響箇所を検出できるか、IDEの補完・型チェックとの親和性。
- **OBJ-4: 開発・運用の簡易性** — 個人開発者がセットアップ・スキーマ定義・日常的なCRUD操作・マイグレーションを最小の認知負荷で実施できること。評価の視点: 初期セットアップの手順数・難易度、シンプルなCRUDクエリの記述のしやすさ、マイグレーション管理の手軽さ、エラー発生時のデバッグのしやすさ。
- **OBJ-5: 将来の機能拡張への対応余地** — シングルテーブル・シンプルCRUDを超えた機能追加や、データモデルの複雑化が生じた場合に、技術的な制約で行き詰まらないこと。評価の視点: リレーション・JOIN等の複雑クエリへの対応能力、スキーマ変更時のマイグレーション対応、ORM/ライブラリのエコシステム成熟度・コミュニティサポート、現時点の技術選択が将来のDB移行コストに与える影響。

## 検討した選択肢

### ALT-1: Neon (PostgreSQL) + Drizzle ORM

サーバーレス向けHTTPベース接続を持つPostgreSQLサービスと、TypeScriptネイティブな軽量ORM（~7.4kb）の組み合わせ。コネクション問題を構造的に回避し、スキーマ=TypeScript=型定義の三位一体で型安全性を実現する。

提案元: OBJ-1（サーバーレス適合性）、OBJ-2（無料枠運用）、OBJ-3（型安全性）、OBJ-5（将来拡張）

### ALT-2: Neon (PostgreSQL) + Prisma 7

Neonに対してPrisma 7（Rustエンジン廃止、バンドル1.6MB）を組み合わせた構成。`prisma migrate dev`による一コマンドマイグレーション、Prisma StudioによるGUIデータ閲覧、成熟したドキュメントを特徴とする。

提案元: OBJ-4（開発・運用簡易性）、OBJ-5（将来拡張）、OBJ-1（サーバーレス適合性）

### ALT-3: Turso (libSQL/SQLite) + Drizzle ORM

HTTPベースのリモートSQLiteサービスとDrizzle ORMの組み合わせ。無料枠が突出して大きく（5GB、月5億行読み取り）、ローカル開発ではSQLiteファイルをそのまま使える簡便さが特徴。

提案元: OBJ-2（無料枠運用）、OBJ-1（サーバーレス適合性）

### ALT-4: Supabase (PostgreSQL) + Drizzle ORM

PostgreSQL + 認証・ストレージ・Realtimeを含むフルスタックBaaSとDrizzle ORMの組み合わせ。Supabase StudioによるGUIデータ管理、将来的なRLS対応が可能。

提案元: OBJ-1（サーバーレス適合性）、OBJ-5（将来拡張）

### 除外した選択肢

- **Kysely + Neon**: OBJ-3で○評価だが、マイグレーション管理が外部化されOBJ-4（開発簡易性）とのトレードオフが大きい。エコシステム規模がDrizzle/Prismaに劣る。
- **postgres.js（生SQL + 手動型付け）**: OBJ-3で△評価。スキーマからの型自動推論がなく、型安全性が手動作業に依存する。C2（TypeScript型安全アクセス必須）の制約を実質的に満たさない。
- **セルフホスト構成（VPS + PostgreSQL）**: S4（月額0円）の制約に非適合。最低$5/月のインフラコストが継続発生する。

### 各セルの詳細

**ALT-1 × OBJ-1（◎）**
- 利点: NeonのHTTPベース接続でコネクションプール問題を構造的に回避。Drizzleバンドル~7.4kbで50MB制限に余裕大。ファイルシステム非依存。
- 欠点: TCP接続使用時はD2の懸念が残るが、HTTPドライバー利用で回避可能。
- 根拠: OBJ-1の全評価視点（コネクション管理・バンドルサイズ・ファイルシステム非依存・Netlify動作実績）を高水準で満たす。

**ALT-1 × OBJ-2（◎）**
- 利点: Neon無料枠: コンピュート100CU-時/月、ストレージ$0.35/GB-月。スケールゼロ対応。Drizzleはコスト中立。
- 欠点: 無料枠の将来的な変更リスクは残るが、2025年に拡大した実績あり。
- 根拠: 個人利用規模で数年にわたり無料枠内に収まる見込み。スケールゼロが不定期アクセスと好相性。

**ALT-1 × OBJ-3（◎）**
- 利点: スキーマ=TypeScriptファイル=型定義。コード生成不要で編集即時反映。5,000超の型インスタンスを即時推論。
- 欠点: D3（v1.0-beta）によるAPI変更リスク。ただしプロダクション利用実績多数。
- 根拠: 型安全サイクルの即時性がPrismaのgenerate方式より優位。CSD依存: D3（Drizzle v1.0-beta段階での本番利用適合性）。

**ALT-1 × OBJ-4（○）**
- 利点: SQLに近いAPIでSQL経験者には直感的。コード生成ステップ不要。設定がシンプル。
- 欠点: Prisma Studioのような公式GUIなし。v1.0-betaでドキュメント整備がPrismaに劣る可能性。
- 根拠: SQL経験ある開発者には直感的だが、GUIツール不在とbeta段階がDXでPrismaに劣る。CSD依存: D3（Drizzle v1.0-beta段階での本番利用適合性）。

**ALT-1 × OBJ-5（◎）**
- 利点: PostgreSQLの全SQL機能を透過的に利用可能。Relational API v2で複雑リレーションに対応。raw SQLへのフォールバックも同一ライブラリ内で可能。
- 欠点: SQL知識が必要。D3によるAPI安定性リスク。
- 根拠: PostgreSQLの表現力+DrizzleのSQL透過性で、将来のスキーマ複雑化への対応余地が最大。CSD依存: D3（Drizzle v1.0-beta段階での本番利用適合性）。

**ALT-2 × OBJ-1（○）**
- 利点: Prisma 7でバンドル1.6MB（gzip 600KB）に90%削減。Driver Adapters経由でNeon HTTPドライバー公式サポート。
- 欠点: Drizzle（~7.4kb）比でバンドル約80倍大。コールドスタート初期化が若干重い。
- 根拠: Netlify 50MB制限内に十分収まるが、バンドルサイズ・初期化でDrizzle比で劣る。

**ALT-2 × OBJ-2（◎）**
- 利点: DB側（Neon）の評価はALT-1と同等。Prisma ORM自体は無料。
- 欠点: Prisma Accelerate等の有料アドオンを使用しなければコスト影響なし。
- 根拠: ORM選択がコストに与える影響はゼロ。DB側のNeon無料枠評価に準じる。

**ALT-2 × OBJ-3（○）**
- 利点: prisma generate生成の型付きクライアントで全クエリに型。v7で生成型98%削減、Language Server改善。
- 欠点: schema.prisma（独自DSL）とTypeScriptの二重管理。prisma generate忘れで型と実態が乖離。
- 根拠: 型安全性は十分だが、generateという手動ステップでDrizzleの即時反映に劣る。

**ALT-2 × OBJ-4（◎）**
- 利点: prisma migrate dev一コマンドフロー。Prisma StudioでGUIデータ閲覧。平易なエラーメッセージ。prisma.config.tsで型安全設定。
- 欠点: prisma generate忘れリスク。生成SQLの予測困難性。Neonローカル開発にDocker必要（or 開発用ブランチで代替可能）。
- 根拠: DX投資が最も進んだORM。ドキュメント・ツーリングの完成度が最高。

**ALT-2 × OBJ-5（◎）**
- 利点: 宣言的スキーマ定義・@relationによるリレーション管理の成熟度が最高。Prisma Migrateの本番適用実績豊富。
- 欠点: 抽象度が高く、Prismaで表現できないクエリはraw SQLフォールバック必要。
- 根拠: スキーマ定義・マイグレーション・リレーション管理の成熟度がDrizzleより高い。

**ALT-3 × OBJ-1（△）**
- 利点: TursoはHTTPベースでファイルシステム非依存。Drizzleバンドルサイズ小。
- 欠点: Netlify環境でのTurso利用の公式サポート・ドキュメントがNeon比で少ない。D1（Netlify+SQLite動作保証）の不確実性が残る。
- 根拠: 技術的には動作するはずだが、Netlify固有の動作実績がNeon比で不十分。CSD依存: D1（Netlify環境でのSQLiteの実際の動作保証）。

**ALT-3 × OBJ-2（◎）**
- 利点: 無料枠が突出: ストレージ5GB、月5億行読み取り、100データベース。個人利用では実質上限に達しない。
- 欠点: 有料プランは$4.99/月〜。
- 根拠: リーディングリスト管理の読み取り重視パターンと読み取りベース無料枠の相性が良い。

**ALT-3 × OBJ-3（◎）**
- 利点: 型安全性はDrizzle ORMが提供するため、ALT-1と同等。DBバックエンドの違いは型安全性に影響しない。
- 欠点: D3（Drizzle beta）リスクはALT-1と共通。
- 根拠: Drizzle採用の限り、PostgreSQL/libSQLの差異はOBJ-3評価に影響しない。CSD依存: D3（Drizzle v1.0-beta段階での本番利用適合性）。

**ALT-3 × OBJ-4（○）**
- 利点: ローカル開発でSQLiteファイルをそのまま使え、サーバープロセス不要でセットアップ最簡素。
- 欠点: ローカル（SQLite）と本番（Turso/libSQL）の環境差異がデバッグを複雑化する可能性。Netlify+Turso+Drizzleの事例が少ない。
- 根拠: ローカル開発の簡便さは最高だが、環境差異と情報量の少なさが懸念。

**ALT-3 × OBJ-5（△）**
- 利点: シングルテーブルからの段階的拡張は低コスト。エッジレプリケーション対応。
- 欠点: SQLiteの機能上限（データ型・制約・同時書き込み制限）がPostgreSQLより低い。PostgreSQL移行コスト中。
- 根拠: データモデル複雑化時の技術的壁がPostgreSQLより早期に現れる。

**ALT-4 × OBJ-1（○）**
- 利点: Connection Pooler（Supavisor）でサーバーレス向け接続管理。ファイルシステム非依存。
- 欠点: Transaction Mode制限（Prepared Statements不可）。Session ModeはNetlify短命コネクションに不適な場合あり。
- 根拠: サーバーレス対応の仕組みはあるが、NeonのHTTP-first設計に比べ接続管理の複雑さが残る。

**ALT-4 × OBJ-2（○）**
- 利点: 無料枠あり: 2プロジェクト、DBストレージ500MB。フルスタック機能で追加サービスコスト不要。
- 欠点: 非アクティブ時の自動一時停止リスク。DBストレージ上限がNeon/Tursoより小さい。
- 根拠: 不定期利用の個人ツールでは一時停止リスクが実用上問題になりうる。

**ALT-4 × OBJ-3（◎）**
- 利点: 型安全性はDrizzle ORMが提供するため、ALT-1と同等。
- 欠点: D3（Drizzle beta）リスクはALT-1と共通。
- 根拠: Drizzle採用の限り、DBバックエンドの違いはOBJ-3評価に影響しない。CSD依存: D3（Drizzle v1.0-beta段階での本番利用適合性）。

**ALT-4 × OBJ-4（○）**
- 利点: Supabase StudioでGUIデータ管理可能。事例・チュートリアル豊富。
- 欠点: サーバーレスコネクション設定が最も複雑（Pooler設定の落とし穴）。非アクティブ一時停止の運用手間。BaaSの機能過多。
- 根拠: GUIの利便性はあるが、接続設定の複雑さが他候補より高い。

**ALT-4 × OBJ-5（○）**
- 利点: PostgreSQL全機能利用可能。RLSでマルチユーザー対応への拡張余地。認証・ストレージ等フルスタック機能。
- 欠点: 無料枠の制約（500MB、一時停止）が長期安定運用に影響。Supabaseプラットフォームへのロックイン。
- 根拠: PostgreSQLとしての拡張性は同等だが、無料枠制約と依存度で相対的に評価が下がる。

## リスク分析

### ALT-1: Neon (PostgreSQL) + Drizzle ORM

**失敗シナリオ 1: Drizzle v1.0 正式リリース時の破壊的変更によるリグレッション**
- 状況: 採用から8ヶ月後、Drizzle ORMがv1.0正式リリースを行い、beta期間中に蓄積された破壊的変更が一括適用された。既存コードのクエリAPIがコンパイルエラーを多数発生させ、マイグレーションAPIのシグネチャも変わっていた。個人開発のため変更対応に着手できるリソースが限られ、機能追加が数週間停止した。
- 原因: D3（Drizzle v1.0-beta段階での本番利用適合性）の不確実性が顕在化した。
- CSD関連: D3の崩壊
- 深刻度: medium
- 発生確率: medium
- 軽減策: Drizzleのchangelogとmigration guideを定期的に購読する。major versionアップデートはローカル + Neon dev branchで先行検証してから本番適用する。破壊的変更が大きい場合のフォールバックとしてPrisma 7への移行パスを把握しておく。

**失敗シナリオ 2: Drizzle公式GUIの不在による運用上の事故**
- 状況: 採用から6ヶ月後、本番データの一部を誤った更新クエリで上書きした。公式GUIがないため復旧作業が手間取り、バックアップも自動化されておらず一部レコードが失われた。
- 原因: OBJ-4の評価で「GUIツール不在」が欠点として指摘されていた問題が顕在化した。
- CSD関連: S2（個人開発・TypeScript/Node.jsスキル中程度以上）の崩れ方との組み合わせで深刻化
- 深刻度: medium
- 発生確率: low
- 軽減策: Neon ConsoleのクエリエディタをGUI代替として習熟する。TablePlusなどのサードパーティGUIクライアント（PostgreSQL接続対応）を導入する。Neonのポイントインタイムリストア機能を有効化し、誤操作からの回復手順を事前に確認する。

**失敗シナリオ 3: Neon無料枠の制限変更による突発的コスト発生**
- 状況: 採用から14ヶ月後、Neonが無料枠の条件を改定し、コンピュート利用時間の上限が大幅に削減された。S4（月額0円で運用）が崩れた。
- 原因: S4と「無料枠の将来的な変更リスク」の顕在化。
- CSD関連: S4（月額0円で運用）の崩壊
- 深刻度: low
- 発生確率: low
- 軽減策: Neon Consoleでのコンピュート使用量アラートを設定する。代替としてTurso（ALT-3）またはSupabase（ALT-4）への移行コストを事前に把握しておく。

**失敗シナリオ 4: データモデル複雑化によるDrizzle Relational APIの限界到達**
- 状況: 採用から10ヶ月後、タグ・コレクション・共有リンクといった機能追加によりスキーマが5テーブル以上に拡張された。Drizzle Relational API v2で対応できないJOINパターンが出現し、raw SQLフォールバックが増加した。型安全性が部分的にしか保たれなくなった。
- 原因: S1（シンプルCRUD中心）の崩壊。
- CSD関連: S1（シングルテーブル・シンプルなCRUD操作が中心）の崩壊
- 深刻度: medium
- 発生確率: medium
- 軽減策: 機能追加時にDrizzleで対応可能かを先行評価する。raw SQLフォールバックのタイミングでPrismaへの段階的移行を検討する。

**失敗シナリオ 5: Netlifyコールドスタート時のNeon接続レイテンシによるUX悪化**
- 状況: 採用から4ヶ月後、不定期利用のため毎回コールドスタートが発生するようになった。Neonのスケールゼロからの起動が平均1-2秒、さらにHTTPドライバーの初回接続で500ms程度が加わり、初回リクエストのレスポンスが計2-3秒になった。
- 原因: D2（サーバーレス環境でのコネクションプーリング要件の厳しさ）の一側面が顕在化。
- CSD関連: D2（サーバーレス環境でのコネクションプーリング要件の厳しさ）
- 深刻度: low
- 発生確率: medium
- 軽減策: Neon ConsoleでAuto-suspendの遅延時間を長めに設定（デフォルト5分 → 30分等）してコールドスタート頻度を下げる。Netlify側でwarm-up用の定期的なpingリクエストを設定する。

### ALT-2: Neon (PostgreSQL) + Prisma 7

**失敗シナリオ 1: prisma generate忘れによる型と実態の乖離が本番障害を引き起こす**
- 状況: 採用から5ヶ月後、スキーマ変更後にprisma generateを実行せずにデプロイした。型エラーはCI時点では出なかったが、本番でランタイムエラーが発生し、サービスが数時間停止した。
- 原因: OBJ-3の評価で「generateという手動ステップ」の欠点として指摘されていた問題が具体的な障害に発展した。
- CSD関連: なし（設計上の欠点の顕在化）
- 深刻度: high
- 発生確率: medium
- 軽減策: CI/CDパイプラインにprisma generateをビルドステップとして必ず含める。Netlify Build Settingsのbuildコマンドに`prisma generate && next build`を設定し、generateなしのデプロイを構造的に防ぐ。

**失敗シナリオ 2: schema.prismaとTypeScriptコードの二重管理による認知負荷の蓄積**
- 状況: 採用から12ヶ月後、スキーマが10テーブル超に拡大し、schema.prismaとTypeScriptコードの二重管理コストが顕在化した。
- 原因: OBJ-3の評価で「schema.prismaとTypeScriptの二重管理」の欠点が、スキーマ規模の拡大（S1の崩壊）によって深刻化した。
- CSD関連: S1（シングルテーブル・シンプルなCRUD操作が中心）の崩壊
- 深刻度: medium
- 発生確率: medium
- 軽減策: Prisma Language Server（VSCode拡張）を必ず導入しDSLとTypeScriptの往復コストを最小化する。スキーマ変更の頻度が高くなった段階でDrizzle ORMへの移行を評価する（同じNeonバックエンドを維持できるため移行コストは低い）。

**失敗シナリオ 3: Prismaが生成するSQLが非効率でNeonのコンピュート枠を消費する**
- 状況: 採用から8ヶ月後、Prismaの高抽象度APIが生成するSQLがN+1問題を引き起こした。Neonのコンピュート利用時間が想定より速く消費されるようになった。
- 原因: OBJ-4の評価で「生成SQLの予測困難性」が欠点として指摘されていた問題の顕在化。
- CSD関連: S4（月額0円で運用）への間接的圧力
- 深刻度: medium
- 発生確率: low
- 軽減策: Prismaのlogging設定でクエリログを有効化し、発行されるSQLを定期的に確認する。リスト取得処理にはinclude/selectを明示してN+1を防止する。

**失敗シナリオ 4: Neonローカル開発環境のDocker依存による開発着手障壁の増大**
- 状況: 採用から2ヶ月後、ローカル開発セットアップにDockerが必要だと判明した。Docker環境の構築・維持コストが予想外に高く、個人開発での開発再開のたびにDocker起動が必要になった。
- 原因: OBJ-4の評価で「Neonローカル開発にDocker必要」が欠点として指摘されていたが、開発用ブランチでの代替を選択しなかった場合に発生する。
- CSD関連: なし（設計の欠点の顕在化）
- 深刻度: low
- 発生確率: low
- 軽減策: Neon開発用ブランチ機能を利用してローカルDBを不要にする。prisma.config.tsの環境変数でDEV_DATABASE_URLをNeon開発ブランチのURLに向ける。

**失敗シナリオ 5: Prismaの大型バンドルによるNetlify関数のコールドスタート悪化**
- 状況: 採用から3ヶ月後、Prisma 7（1.6MB、gzip 600KB）を含むNetlify関数のコールドスタートが、Drizzle構成（~7.4kb）と比べて明らかに遅いことが判明した。コールドスタート時間が常時500ms以上長くなった。
- 原因: OBJ-1の評価で「バンドル約80倍大・コールドスタート初期化が若干重い」という欠点として指摘されていた問題が性能として顕在化した。
- CSD関連: S5（ORMがNetlify関数のバンドルサイズ制限内に収まる）は崩れないが、性能特性の問題
- 深刻度: low
- 発生確率: medium
- 軽減策: Netlify Functionsのバンドル最適化設定を確認する。コールドスタートが許容できない場合はDrizzle ORM（ALT-1）への移行を検討（Neonバックエンドは維持可能）。

## 将来の意思決定への影響

### LD-1: マルチユーザー対応・認証連携方式の選択（想定時期: 1〜2年後）

- ALT-1 (Neon + Drizzle): 選択肢が広がる — PostgreSQLのRLS（Row Level Security）を将来利用できる。Neonはブランチ機能を持ち、ステージング環境の分離が容易。DrizzleのschemaファイルにRLSポリシー定義を追加するパスが開かれている。
- ALT-2 (Neon + Prisma 7): 中立 — PostgreSQL側のRLS機能は利用可能だが、PrismaはRLSをネイティブサポートしていないためraw SQL経由が必要。
- ALT-3 (Turso + Drizzle): 選択肢が狭まる — libSQL/SQLiteにはRLSがない。マルチユーザーのアクセス制御はアプリケーション層に全量実装が必要。
- ALT-4 (Supabase + Drizzle): 選択肢が広がる — Supabase AuthとRLSの統合が公式サポートされており、マルチユーザー化の実装コストが最小。ただし、Supabaseプラットフォームへの依存度が高まる。

### LD-2: スケールアウト・読み取りパフォーマンス最適化の選択（想定時期: 2〜3年後）

- ALT-1 (Neon + Drizzle): 選択肢が広がる — Neonのリードレプリカ機能を将来追加可能。PostgreSQL全機能のパーティショニング・マテリアライズドビューが利用可能。Drizzleはraw SQLへのフォールバックが同一ライブラリ内で可能。
- ALT-2 (Neon + Prisma 7): 中立 — リードレプリカはNeonの機能として同様に利用可能。Prisma Accelerate（有料）でグローバルキャッシュを追加可能。
- ALT-3 (Turso + Drizzle): 選択肢が広がる（一部狭まる） — Tursoはエッジレプリケーション（グローバル分散）が強みだが、同時書き込み性能と複雑クエリはSQLiteの制約に当たる可能性がある。
- ALT-4 (Supabase + Drizzle): 中立 — PostgreSQL全機能が利用可能だが、無料枠のストレージ上限（500MB）が早期に制約となる可能性がある。

### LD-3: 全文検索・高度なクエリ機能の導入（想定時期: 1〜2年後）

- ALT-1 (Neon + Drizzle): 選択肢が広がる — PostgreSQLのFull Text Search、pg_trgm拡張、将来的なpgvector（ベクトル検索）が利用可能。DrizzleはPostgreSQL固有機能のサポートが充実。
- ALT-2 (Neon + Prisma 7): 中立 — PostgreSQL拡張機能はNeon側で利用可能だが、PrismaのORMレイヤーではサポートが限定的。pgvectorなどの拡張は主にraw SQLで操作することになる。
- ALT-3 (Turso + Drizzle): 選択肢が狭まる — SQLiteにはPostgreSQLレベルの全文検索拡張がない（FTS5はあるが機能が限定的）。pgvectorに相当するベクトル検索拡張はlibSQLにない。高度な検索機能導入時にPostgresへの移行が強く動機付けられる。
- ALT-4 (Supabase + Drizzle): 選択肢が広がる — SupabaseはpgvectorをGUIで管理できる機能を提供しており、AI検索機能の追加が最も容易。

### LD-4: CI/CDパイプライン・DBマイグレーション自動化の設計（想定時期: 0.5〜1年後）

- ALT-1 (Neon + Drizzle): 中立 — `drizzle-kit push`/`generate` + `migrate`によるCI統合が可能。Drizzle v1.0-betaのCLIはまだ変化する可能性があり、CIスクリプトの修正が発生しうる。Neonのブランチ機能でPRごとのDB環境分離が可能。
- ALT-2 (Neon + Prisma 7): 選択肢が広がる — `prisma migrate deploy`はCI/CD統合の実績が豊富で、ドキュメントが充実している。マイグレーション自動化の設計が最も容易。
- ALT-3 (Turso + Drizzle): 中立 — DrizzleのマイグレーションはALT-1と同等。ローカル（SQLiteファイル）と本番（Turso）の環境差異がCIでのテスト戦略を複雑化する可能性がある。
- ALT-4 (Supabase + Drizzle): 選択肢が狭まる — Supabaseのマイグレーション管理は独自のSupabase CLIとの統合が前提。Drizzle + Supabase + CIの組み合わせは事例が比較的少なく、パイプライン設計に試行錯誤が発生しうる。

### 可逆性評価

| 代替案 | 可逆性 | 理由 |
|--------|--------|------|
| ALT-1: Neon + Drizzle | 部分可逆 | DB層（Neon→別PostgreSQL）はDrizzleのschemaを再利用しながら接続先変更で対応可能。ORM層（Drizzle→Prisma等）はschema定義の書き直しが必要だが、シンプルなCRUDのため規模は限定的。Drizzle v1.0-betaのAPI変化が将来の移行を複雑化するリスクがある。 |
| ALT-2: Neon + Prisma 7 | 部分可逆 | DB層の可逆性はALT-1と同等。ORM層の可逆性は、schema.prisma（独自DSL）からの移行にDrizzle/Kyselyへの書き直しが必要。Prismaは成熟しており、API安定性リスクはALT-1より低い。 |
| ALT-3: Turso + Drizzle | 部分可逆（制約あり） | ORM層（Drizzle）の可逆性はALT-1と同等。DB層（Turso/libSQL→PostgreSQL）の移行は、SQLiteとPostgreSQLのSQL方言差異・データ型差異により移行スクリプトの作成が必要。 |
| ALT-4: Supabase + Drizzle | 部分可逆（プラットフォーム依存） | ORM層の可逆性はALT-1と同等。DB層はPostgreSQLのため、論理的には他のPostgreSQLへの移行は可能。ただしSupabase固有機能（Auth統合、RLS等）を利用し始めると依存度が高まり移行コストが増大する。 |

## 決定

ALT-1: Neon (PostgreSQL) + Drizzle ORM を採用する。

## 根拠

全体的な評価が高いため（5項目中4項目で◎評価）。OBJ-1（サーバーレス適合性）、OBJ-2（無料枠運用）、OBJ-3（TypeScript型安全）、OBJ-5（将来拡張余地）の4項目で最高評価（◎）を獲得しており、今回の要件において最もバランスの取れた選択肢と判断した。

## 受け入れたトレードオフ

- **D3（Drizzle v1.0-beta段階での本番利用適合性）のリスクを許容する**: Drizzle ORMが現時点でv1.0-beta段階にあり、正式リリース時の破壊的変更リスクが存在する。多くのプロダクションでの利用実績があることから実質リスクは限定的と判断し、このリスクを受け入れる。軽減策として、Drizzleのchangelogを定期的に購読し、major versionアップデートはステージング環境で先行検証する。
- **OBJ-4（開発・運用の簡易性）が○評価であることを受け入れる**: Prisma 7（ALT-2）の◎には劣り、公式GUIがなくv1.0-betaでドキュメント整備がPrismaに劣る可能性がある点を受け入れる。Neon ConsoleのクエリエディタやサードパーティのPostgreSQL GUIクライアントで補完する。
