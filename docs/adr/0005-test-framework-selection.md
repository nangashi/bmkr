# ADR-0005: テストフレームワークの選定

## ステータス

Proposed

## 日付

2026-02-27

## コンテキスト

Next.js App Router（TypeScript strict）プロジェクトにおいて、ユニット/結合テストフレームワーク（Vitest vs Jest）およびE2EテストFW（Playwright vs Cypress）のどの組み合わせを採用するかを決定する必要がある。

**スコープ**

- 対象: ユニットテスト・結合テストのフレームワーク選定、E2Eテストのフレームワーク選定
- 対象外: テスト戦略（何をどのレベルでテストするか）、CI/CDパイプラインでのテスト実行方法、テストカバレッジ目標の設定

**可逆性**: medium — テストコードの書き換えが必要になるが、アプリケーションコード自体への影響は限定的。ただしテスト資産が蓄積するほど移行コストは増大する。MVP段階での決定のため、テスト資産が少ない今なら変更コストは比較的低い。

## 関連する既存決定

- **ADR-0002: フロントエンド技術スタックの選定** — Next.js + Node.js + pnpm を採用。テストフレームワークはTypeScript strict対応・Node.js互換性・pnpm使用・App Router固有コードへの対応（モック/スタブまたはE2E委譲）を前提とする。
- **ADR-0004: TypeScript/React/Next.jsプロジェクトのLintおよびフォーマットツールの選定** — Biome単独採用（リント+フォーマット統一）。ESLintを使用しないため、eslint-plugin-jestやeslint-plugin-vitestによるテストコードのリントルール強化は行わない。テストコードのリントはBiomeの組み込みルールで代替するか、リントを行わない方針となる。

## 前提条件

### 確実（Certainty）

- C1: テストフレームワークはTypeScript（strictモード）に対応していること — 根拠: ADR-0002で確定。プロジェクト全体がTypeScript strictを採用しており、再検討の対象ではない。
- C2: テストフレームワークはNode.js互換性を持つこと — 根拠: ADR-0002で確定。テストランナー自体はNode.js上で動作するため、Node.js互換性は必須。Edge Runtime上のコードをテストする場合でも、テストFW自体はNode.jsで実行されるためこの制約は維持される。
- C3: パッケージのインストール・管理はpnpmを使用すること — 根拠: ADR-0002で確定。
- C4: Next.js App Router固有のコード（Server Components、Server Actions、Route Handlers等）のテストは、モック・スタブによるユニットテストまたはE2Eテストで対応すること — 根拠: ADR-0002で確定。ユニットテストFWではモック/スタブで対応し、完全な動作確認はE2Eに委ねる。
- C5: ESLintは使用しないため、eslint-plugin-jestやeslint-plugin-vitestによるテストコードのリントルール強化は行わない — 根拠: ADR-0004で確定。Biome単独採用のため。
- C7: 対象プロジェクトはNext.js App Routerを採用している — 根拠: 決定ステートメントに明示。

### 仮定（Supposition）

- S1: チームの主要開発者はVitest・Jestのいずれかについて基礎的な習熟を持っている — 根拠: TypeScript + React + Next.jsを採用しているチームとしての想定。検証方法: チームメンバーへのヒアリング。
- S2: Playwright・Cypressのいずれかについて基礎的な習熟を持っている — 根拠: S1と同様の想定。検証方法: ヒアリング。
- S3: プロジェクトMVP段階ではテスト資産が少なく、FW移行コストは現時点で比較的低い — 根拠: 可逆性の評価（medium）。MVP段階を超えた後は移行コストが増大する。検証方法: 現在のテストファイル数を確認。
- S4: CI環境でのテスト実行時間が開発速度に与える影響を許容範囲内に抑えることが求められる — 根拠: 一般的な開発効率の観点。明示的な要件定義は未確認。検証方法: CIパイプライン要件の確認。
- S5: Next.jsのバージョンが近い将来に大きく変化しない（App RouterのAPIが安定している） — 根拠: App Routerは安定機能として提供中。ただしNext.jsの開発速度は高く、マイナーバージョンでも挙動が変化する事例がある。検証方法: リリースノートの定期確認。
- S6: プロジェクトは主にChromium系ブラウザをターゲットとし、Safari / Firefoxの網羅的なクロスブラウザE2Eテストを必須とはしない — 根拠: 一般的なWebアプリのターゲット想定。ターゲットブラウザは明示されていない。検証方法: ターゲットユーザーのブラウザ環境要件を確認。
- S7: 非同期React Server Components（async RSC）はユニットテスト/結合テストFW（Vitest/Jest）では直接テストできない — 根拠: 現在の技術的制約として確認済み。ただしReact 19 + @testing-library/reactのRSCサポートが進展中であり、数ヶ月〜1年以内に変化する可能性がある。検証方法: React/Next.jsリリースノートと@testing-library/reactのRSCサポート状況を定期確認。

### 不確実（Doubt）

- D1: VitestとJestのいずれがチームの既存スキルセットに近いか — 影響範囲: FW選定の「学習コスト」評価に直接影響する。リスク: 習熟FWと異なるものを選択した場合の学習コスト過小評価。
- D2: PlaywrightとCypressのいずれがチームの既存スキルセットに近いか — 影響範囲: E2E FW選定の「学習コスト」評価に直接影響する。リスク: D1と同様の構造。
- D3: E2EテストでSafari / Firefoxの対応が実際に必要かどうか — 影響範囲: Playwright（クロスブラウザ対応強）vs Cypress（Chrome系中心）の評価に影響する。リスク: Safari対応が必要な場合はCypressの評価が下がる。
- D4: CI/CDパイプラインでの並列実行環境の利用可否と許容コスト — 影響範囲: Playwrightは無料でネイティブ並列実行が可能、Cypressは無料プランで並列実行に制限あり。リスク: 並列実行活用時にCypressは有償Dashboardが必要となる可能性。

## 判断基準

優先度は決定者（人間）が設定。

- **OBJ-1: App Router対応能力** — Next.js App Router固有のコード（Server Components、Server Actions、Route Handlers等）を実用的にテストできること。C4の制約（モック/スタブによるユニットテスト、またはE2Eでの完全な動作確認）の範囲内で、最大限のテストカバレッジを実現できること。評価の視点: Server Components・Server ActionsのモックAPIサポート品質、App Routerとの統合設定の複雑さ、E2E FWがNext.jsアプリのローカル起動・テストに対応しているか。
- **OBJ-2: TypeScript strict統合品質** — TypeScript strictモードとの統合品質が高く、テストコード自体の型安全性が確保され、型エラーを早期に検出できること。評価の視点: テストコード（モック・スタブ含む）の型推論精度、strictモードでの追加設定の必要性、vi.fn() / jest.fn()等のモックAPIの型付け品質。
- **OBJ-3: 開発フィードバック速度** — ユニット/結合テストの実行速度が十分に速く、TDDや変更後の即時フィードバックサイクルを妨げないこと。評価の視点: テスト実行速度（cold start速度・ウォッチモードの再実行速度）、ESMネイティブ対応とトランスパイル不要度、並列実行のデフォルト対応。
- **OBJ-4: E2Eブラウザ対応柔軟性** — E2Eテストで現在および将来必要となるブラウザ環境をカバーできること。現時点ではChromium系が主要ターゲット（S6）だが、Safari / Firefox対応の必要性は不確実（D3）であり、対応が必要になった際に拡張できる柔軟性を持つこと。評価の視点: ネイティブに対応するブラウザエンジンの種類、クロスブラウザテストの追加コスト、CI環境でのマルチブラウザ並列実行サポート（D4との関連）。
- **OBJ-5: 導入・維持コストの低さ** — チームの既存スキルセットとの親和性が高く、初期導入・設定・長期維持にかかるコスト（学習コスト・設定複雑さ・ライセンスコスト）が許容範囲内であること。評価の視点: React / Next.jsエコシステムでの採用率・コミュニティの活発さ、初期設定の複雑さ、CI環境での実行コスト（D4との関連）、チームの学習コスト（D1・D2との関連）。

## 検討した選択肢

### ALT-1: Vitest + Playwright

ユニット/結合テストにVitest、E2EテストにPlaywrightを採用する組み合わせ。Next.js公式サポート、ESMネイティブ、クロスブラウザ対応、無料並列実行を備える。全5つの判断基準で最優位として提案された。

### ALT-2: Vitest + Cypress

ユニット/結合テストにVitest、E2EテストにCypressを採用する組み合わせ。Vitestの速度・TypeScript統合の優位性を維持しつつ、CypressのTime-Travel Debuggingを組み合わせる。OBJ-3（Cypressのデバッグ体験）、OBJ-5（Cypressの初学者向け学習コストの低さ）の観点で提案された。

### ALT-3: Jest + Playwright

ユニット/結合テストにJest、E2EテストにPlaywrightを採用する組み合わせ。Jestの成熟したエコシステムとPlaywrightのクロスブラウザ対応・並列実行を組み合わせる。OBJ-1（Jestの豊富なサンプルコード）、OBJ-5（Jestの歴史的普及による学習リソースの豊富さ）の観点で提案された。

### ALT-4: Jest + Cypress

ユニット/結合テストにJest、E2EテストにCypressを採用する組み合わせ。歴史的に最も広く使われた組み合わせだが、現在のトレンドからは後退。OBJ-5（チームに両方の経験者がいる場合の学習コスト最小化）の観点でのみ提案された。

### 除外した選択肢

なし（決定ステートメントのスコープ内の全組み合わせを評価対象としている）

### 各セルの詳細

**ALT-1 × OBJ-1（◎）**
- 利点: Next.js公式がVitestセットアップガイドを提供。App Router環境での公式サポート確立。PlaywrightはServer Actions・Route HandlersのE2Eテストに標準対応。
- 欠点: 非同期RSCのユニットテストは不可（S7）。E2Eに委ねる必要あり。
- 根拠: Next.js公式ドキュメント

**ALT-1 × OBJ-2（◎）**
- 利点: ViteのtsconfigをVitestが共有するためstrict追加設定不要。vi.fn()の型推論精度が高い。ESMネイティブでトランスパイル起因の型齟齬なし。PlaywrightはデフォルトTypeScriptで型定義充実。
- 欠点: 複雑なモックシナリオでは型引数の手動指定が必要な場合あり（Jestも同様）。

**ALT-1 × OBJ-3（◎）**
- 利点: ウォッチモード380ms（Jest 3.4秒比）。実行速度30〜70%高速。Playwrightの無料並列実行。テスト安定性92%。
- 欠点: E2Eはブラウザ起動を伴うため本質的にユニットテストより遅い（E2E共通）。

**ALT-1 × OBJ-4（◎）**
- 利点: Chromium・Firefox・WebKitを単一FWでネイティブサポート。設定変更のみでブラウザ追加可能。CI無料並列実行。
- 欠点: 複数ブラウザエンジンのバイナリインストールが必要（初期セットアップ時のみ）。

**ALT-1 × OBJ-5（◎）**
- 利点: Next.js公式サポートで設定シンプル。Playwright npm週間DL約3,319万（Cypressの約5倍）で情報豊富。並列実行無料。
- 欠点: Vitest+Playwrightをセットで解説するチュートリアルはやや少ない。

**ALT-2 × OBJ-1（○）**
- 利点: ユニットテスト側はALT-1と同等のApp Router対応能力。Cypressも基本的なE2Eテストは実施可能。
- 欠点: Safari（WebKit）E2E対応が限定的。D3が「必要」と判明した場合、カバレッジに制約。CSD依存: D3

**ALT-2 × OBJ-2（○）**
- 利点: Vitest側のTypeScript統合品質はALT-1と同等。
- 欠点: Cypressのカスタムコマンド型定義に手動対応が必要（cypress.d.ts）。TypeScript後付け設計のため型定義の網羅性がPlaywrightより劣る。

**ALT-2 × OBJ-3（○）**
- 利点: Vitestの速度優位を享受。CypressのTime-Travelデバッグでデバッグ効率は良好。
- 欠点: E2E実行速度がPlaywrightより35〜45%遅い。並列実行は有償Dashboard必要（D4）。テスト安定性81%（再実行コスト増）。CSD依存: D4, S4

**ALT-2 × OBJ-4（△）**
- 利点: Chrome系・Firefoxには対応。S6が維持される間は実用上問題なし。
- 欠点: Safariネイティブサポートなし。D3が「必要」と判明した場合、別ツール追加または移行が必要。拡張柔軟性が低い。CSD依存: D3（「必要」なら×に変化）, D4

**ALT-2 × OBJ-5（○）**
- 利点: Vitestの設定シンプルさ維持。Cypressの初学者向けデバッグ体験。
- 欠点: CI並列実行の有償化リスク（D4）。コミュニティ規模がPlaywrightの約1/5。採用トレンド横ばい。CSD依存: D4, D2

**ALT-3 × OBJ-1（△）**
- 利点: next/jestアダプターで公式対応あり。Playwright側はALT-1と同等のE2E能力。
- 欠点: React 19との依存関係競合。ESM前提のモジュールでモック摩擦。設定がVitestより複雑。CSD依存: S5

**ALT-3 × OBJ-2（△）**
- 利点: Playwright側の型統合品質はALT-1と同等。@types/jestで基本的な型チェック機能。
- 欠点: CJS優先でESM+strictの追加設定が必要。@types/jestと@jest/globalsの二系統で型定義管理が複雑。CSD依存: S5

**ALT-3 × OBJ-3（△）**
- 利点: Playwrightの無料並列実行はALT-1と同等。
- 欠点: ウォッチモード3.4秒（Vitest 380msの約9倍）。CJSトランスパイルでcold start遅い。React 19互換性問題のデバッグコスト。CSD依存: S4, S5

**ALT-3 × OBJ-4（◎）**
- 利点: PlaywrightのCloudium・Firefox・WebKitサポートにより、ALT-1と同等のブラウザ対応柔軟性。
- 欠点: なし（OBJ-4はE2E FWのみに依存）。

**ALT-3 × OBJ-5（△）**
- 利点: Jestの歴史的普及で学習リソース豊富。Playwrightの並列実行無料。
- 欠点: App Router環境での設定複雑さ。古い情報の混在で調査コスト増。ESM摩擦。CSD依存: D1, S1

**ALT-4 × OBJ-1（△）**
- 利点: 歴史的に最も多くのドキュメント・サンプルが存在。基本的なApp RouterテストはJestでも実施可能。
- 欠点: JestのApp Router設定複雑さ + Cypressのクロスブラウザ制約が複合。最新情報が相対的に少なくなる傾向。CSD依存: D3, S5

**ALT-4 × OBJ-2（×）**
- 利点: 最低限のTypeScript strict動作は保証。
- 欠点: JestのCJS/ESM混在 + Cypressの型統合弱点が累積。4候補中最低のstrict統合品質。CSD依存: S5

**ALT-4 × OBJ-3（×）**
- 利点: 既存スキルによる学習コスト低減（ただしOBJ-5の観点）。
- 欠点: Jestの遅さ + Cypressの並列制限 + 安定性低下が全て重なる。全評価軸で最低。CSD依存: D4, S4

**ALT-4 × OBJ-4（△）**
- 利点: Chrome系・Firefoxには対応。
- 欠点: ALT-2と同様、Safari非対応。拡張柔軟性が低い。CSD依存: D3, D4

**ALT-4 × OBJ-5（×）**
- 利点: チームにJest+Cypress両方の経験者がいる場合のみ学習コスト最小。
- 欠点: 両FWの弱点が重なる。トレンドと逆行で将来のサポート縮小リスク最大。長期維持コスト不確実性が最も高い。CSD依存: D1+D2, D4

## リスク分析

### ALT-1: Vitest+Playwright の失敗シナリオ

**シナリオ1: Next.jsメジャーバージョンアップでVitest互換性が崩壊**
- 状況: Next.js 16（仮称）がApp Routerのサーバーコンポーネントモデルを大幅に刷新した際、next/vitestアダプターがアップデートに追従できず、モック設定が機能しなくなる。全ユニットテストが赤になり、CIが全フェイルした状態が続く。
- 原因: VitestはNext.jsコアチームの内製ツールではなく、公式サポートはドキュメントとアダプターの提供にとどまる。メジャーバージョンアップ時にVitest側のアダプター対応が遅延するリスク。
- CSD関連: S5（Next.jsのバージョンが近い将来に大きく変化しない）の崩壊
- 深刻度: high / 発生確率: low
- 軽減策: Vitestのバージョンを明示的にロックし、Next.jsのメジャーアップデートをステージング環境でテストしてからマージする。`next/vitest`アダプターのGitHubリポジトリを定期監視する。E2EをPlaywrightで担保しているため、ユニットテストが一時的に壊れてもE2Eは機能し続ける。

**シナリオ2: React 19以降で非同期RSCのユニットテストが一般化し、テスト戦略の見直しを強いられる**
- 状況: React 19以降で@testing-library/reactのRSCサポートが本格化した場合、「E2E委譲」戦略のままでは開発フィードバックループが遅い状態が継続し、技術的負債として認識されながらも改善着手できない状況になりうる。
- 原因: S7を「当面変わらない制約」として扱い、将来の変化に対するアップデート計画を立てていなかった場合。
- CSD関連: S7（非同期RSCはユニットテストFWで直接テストできない）の変化
- 深刻度: medium / 発生確率: medium
- 軽減策: @testing-library/reactのRSCサポート状況を四半期ごとにレビューし、S7の前提が変化した場合に迅速にテスト戦略を更新できるよう、テスト方針のレビューポイントをロードマップに組み込む。

**シナリオ3: PlaywrightのブラウザバイナリでCI環境のセットアップコストが膨張**
- 状況: CI環境のランナーイメージ更新などによりPlaywrightが要求するブラウザバイナリが毎回インストールされ、E2EテストのセットアップにCI実行時間が倍増する。コスト削減のためE2Eを削減する議論が生まれ、テスト戦略の後退が起きる。
- 原因: PlaywrightはChromium・Firefox・WebKitのバイナリをインストールする必要があり、CIキャッシュの設定が不適切だった場合にセットアップコストが顕在化する。
- CSD関連: S4（CI環境でのテスト実行時間が許容範囲内）の崩壊
- 深刻度: medium / 発生確率: medium
- 軽減策: GitHub Actionsの`actions/cache`で`~/.cache/ms-playwright`をキャッシュする設定を初期構築時から必ず導入する。必要なブラウザを`chromium`のみに絞ることで初回インストール時間を大幅短縮できる。Safari/Firefoxテストは週次・リリース前のみ実行するようスケジュール分離する。

**シナリオ4: チームがCypress経験者中心で、Playwrightの学習コストが想定外に高かった**
- 状況: 採用後に判明した実態として、チームメンバーの大半がCypress経験者だった（D2が「Cypressに近い」と判明）。Playwrightの非同期API・ロケーター概念・デバッグフローへの適応に3〜4ヶ月を要し、その間E2Eテストの品質が低下する。
- 原因: D2（PlaywrightとCypressのどちらがチームのスキルセットに近いか）の検証を行わず採用判断を下した場合。
- CSD関連: D2（PlaywrightとCypressのチームスキルセット）が「Cypress」と判明したケース
- 深刻度: medium / 発生確率: medium（D2は未確定のため）
- 軽減策: 採用決定前にチームメンバーへのヒアリング（S2・D2の検証）を実施する。Playwrightの学習を加速するため、公式ハンズオンガイドと社内サンプルリポジトリを早期に整備する。初期段階はChromium単体・単純なシナリオのみに絞り、段階的に拡張する。

**シナリオ5: Safari対応が必須になりクロスブラウザ不具合の調査工数が想定を超えた**
- 状況: D3（Safari対応の必要性）が「必須」と判明し、Safari（WebKit）でのE2Eテストを本格化させた際に、Chrome通過テストがWebKitで失敗するケースが続出し、原因調査に多くの工数が取られる。
- 原因: D3が「必須」と判明するまでSafariテストの準備を後回しにしていた。Playwright自体は正しく選択されているが、運用準備が追いついていない。
- CSD関連: D3（Safari対応の必要性）が「必須」と確定したシナリオ
- 深刻度: low / 発生確率: medium（D3は未確定のため）
- 軽減策: D3の判断を早期に確定させる（ターゲットユーザーのブラウザ環境調査）。Playwright採用初期からWebKitもCIに含める（失敗は許容）ことで、問題の早期発見と習熟を進める。

## 将来の意思決定への影響

### LD-1: CI/CDパイプラインでのテスト並列実行戦略の決定（想定時期: 6ヶ月〜1年以内）

- ALT-1 (Vitest+Playwright): **選択肢が広がる** — Playwrightはネイティブ並列実行が無料。CIコスト追加なしで並列化できるため、実行戦略の選択肢が広い。Vitestの並列実行も標準サポート。
- ALT-2 (Vitest+Cypress): **選択肢が狭まる** — Cypressの並列実行は無料プラン非対応（D4）。Cypress Cloud（有償）の契約が必要になるか、E2E実行時間を許容するかの二択に制限される。
- ALT-3 (Jest+Playwright): **中立** — Playwrightの並列実行はALT-1と同等。Jestのユニットテスト並列化はVitestより設定が複雑になる場合がある。
- ALT-4 (Jest+Cypress): **選択肢が最も狭まる** — Jestの遅さとCypressの並列制限が重なり、CI実行時間の最適化手段が限られる。有償化への依存度が最も高い。

### LD-2: Safari / Firefoxクロスブラウザ対応要否の決定（想定時期: 3〜6ヶ月以内）

- ALT-1 (Vitest+Playwright): **選択肢が広がる** — PlaywrightはChromium・Firefox・WebKitをネイティブサポート。Safari対応が「必要」と判明しても、設定変更のみで対応可能。移行コストゼロ。
- ALT-2 (Vitest+Cypress): **選択肢が狭まる** — CypressはSafari（WebKit）ネイティブ非対応。「必要」と判明した場合、PlaywrightへのE2E FW移行または補完ツール追加が必要となる（中規模の移行コスト発生）。
- ALT-3 (Jest+Playwright): **選択肢が広がる** — E2E FWがPlaywrightであるため、Safari対応の判断はALT-1と同じ選択肢が維持される。
- ALT-4 (Jest+Cypress): **選択肢が狭まる** — ALT-2と同等の制約。さらにJestの移行コストも重なるため、FW全体の見直しコストが最も大きい。

### LD-3: React / Next.jsバージョンアップへの追随方針の決定（想定時期: 6ヶ月〜1年サイクル）

- ALT-1 (Vitest+Playwright): **選択肢が広がる** — VitestはNext.js公式がセットアップガイドを提供。Next.js側のアップデート追随コストが低い。PlaywrightはNext.jsバージョンに依存しないE2E構造のため影響を受けにくい。
- ALT-2 (Vitest+Cypress): **中立** — Vitest側はALT-1と同等。Cypress側のNext.js対応は公式サポートが薄い可能性があるが、大きな制約になりにくい。
- ALT-3 (Jest+Playwright): **選択肢が狭まる** — JestはNext.jsとの相性上、React 19対応やESM移行でアップデート追随時に追加設定コストが発生しやすい（S5と連動）。
- ALT-4 (Jest+Cypress): **選択肢が狭まる** — ALT-3のJest側リスクに加え、Cypress側の対応遅延リスクも重なる。

### LD-4: 非同期RSC（React Server Components）のユニットテスト対応方針の決定（想定時期: 1年以内の可能性あり）

- ALT-1 (Vitest+Playwright): **選択肢が広がる** — VitestはESMネイティブかつReactエコシステムとの統合が良好。RSCサポートが公式化した際に追随しやすい構造。現在E2Eに委ねているテストをユニットテストに引き下げる移行が容易になる可能性がある。
- ALT-2 (Vitest+Cypress): **中立** — Vitest側はALT-1と同等。RSCユニットテスト対応の恩恵を受けられる。Cypress側への影響はない。
- ALT-3 (Jest+Playwright): **選択肢が狭まる** — JestのCJS優先設計とESM摩擦のため、async RSCサポートへの追随がVitestより困難になる可能性がある。
- ALT-4 (Jest+Cypress): **選択肢が狭まる** — ALT-3と同等の制約。RSCサポートの恩恵を最も受けにくい構成。

### LD-5: テストツールチェーンの統一・刷新方針の決定（想定時期: 1〜3年後）

- ALT-1 (Vitest+Playwright): **中立** — 現時点でトレンドの主流に位置するため、「別FWへの移行」を迫られる外圧が最も少ない。技術的負債が最も蓄積しにくい。
- ALT-2 (Vitest+Cypress): **選択肢が狭まる（部分的）** — Cypressの採用シェア減少傾向が続いた場合、E2EのみPlaywrightへ移行を検討する判断が生じうる。ユニットテスト側（Vitest）は維持できるため完全移行は不要。
- ALT-3 (Jest+Playwright): **選択肢が狭まる（部分的）** — JestのESM対応遅延やApp Routerとの摩擦が蓄積した場合、Vitestへの移行を検討する判断が生じうる。E2E側（Playwright）は維持できる。
- ALT-4 (Jest+Cypress): **選択肢が最も狭まる** — 両FWでトレンドから後退した場合、FW全体の移行を迫られる可能性がある。テスト資産が積み上がった状態での全体移行コストは最大。

**可逆性評価**

| 代替案 | 可逆性 | 理由 |
|--------|--------|------|
| ALT-1 (Vitest+Playwright) | 部分可逆 | ユニットテストとE2Eテストで個別に移行できるため完全不可逆ではないが、テスト資産が蓄積するほど移行コストが増大する。現在のMVP段階では移行コストは比較的低い（S3）。 |
| ALT-2 (Vitest+Cypress) | 部分可逆 | Vitest側はALT-1と同等。CypressからPlaywrightへのE2E移行は、カスタムコマンドや固有APIの書き換えが必要。D3・D4が顕在化した場合に移行圧力が高まる。 |
| ALT-3 (Jest+Playwright) | 部分可逆 | Playwright側はALT-1と同等。JestからVitestへの移行は設定ファイルと一部モック構文の書き換えが必要だが、テスト自体の構造は概ね互換性がある。 |
| ALT-4 (Jest+Cypress) | 部分可逆（移行コスト最大） | 両FWの移行コストが重なる。テスト資産が積み上がった後の移行は現実的に困難になりうる。MVP段階では移行可能だが、その後急速にコストが増大する。 |

## 決定

ALT-1: Vitest + Playwright を採用する。

## 根拠

全目的（App Router対応能力、TypeScript strict統合品質、開発フィードバック速度、E2Eブラウザ対応柔軟性、導入・維持コストの低さ）で最優位（◎）であり、トレードオフがないため。

## 受け入れたトレードオフ

明示的に受け入れるトレードオフはなし。全目的で最優位のため、他の選択肢に対して犠牲にする判断基準がない。

ただし、非同期RSCのユニットテスト不可（S7 / OBJ-1の欠点欄参照）はVitest/Jest共通の制約であり、選択肢によらずE2Eに委ねる方針となる。この制約はALT-1固有のトレードオフではなく、現在の技術環境における全選択肢共通の制限として受け入れる。
